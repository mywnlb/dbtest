InnoDB Buffer Pool 内存侧链路

承接上一份《存储结构与数据链路》的磁盘侧，这份讲内存侧：页从 .ibd 进入 Buffer Pool 之后， LRU / Flush / Change Buffer / Redo / Undo / 检查点如何协同，支撑增删改查并保证崩溃不丢数据。
适用版本：MySQL 5.7 / 8.0，InnoDB 引擎。
一句话总览：上层永远只跟 Buffer Pool 打交道，从不直接读写磁盘文件。 读 → 逻辑读命中/未命中物理读；写 → 改内存页 + 写 redo（WAL），数据页异步刷盘；崩溃 → redo 前滚 + undo 后滚。

01Buffer Pool 整体结构

Buffer Pool 是 InnoDB 在内存里缓存磁盘页的核心区域，默认占物理内存的 50%~75%（参数 innodb_buffer_pool_size）。它的目标是：让绝大多数页访问命中内存，把昂贵的磁盘随机 I/O 降到最低。

四个层次

  • 多实例（innodb_buffer_pool_instances）：把缓冲池切成多份，各自独立加锁，减少高并发下的锁竞争。
  • chunk：每个实例由若干 chunk 组成，是在线扩缩容的单位（innodb_buffer_pool_chunk_size）。
  • 控制块 + 缓存页：每个 16KB 缓存页（frame）配一个控制块，记页号、LRU/Flush 链表节点、修改 LSN、状态位等。
  • 三大链表：Free（空闲页）、LRU（管冷热淘汰）、Flush（管脏页刷盘）。一个页可同时挂在多条链表上。
页的生命周期：Free → 读入挂 LRU → 被改也挂 Flush → 刷盘后离开 Flush → 淘汰回 Free。

02LRU 链表：young / old 分代

朴素 LRU（最近最少使用）有个致命缺陷：一次全表扫描或预读会读入海量页，若都插到链表头，会把真正的热点页挤出缓存。InnoDB 用分代 LRU 解决这个问题。

  • young 区（热端，约 5/8）：反复访问的热点页，链表头是最近最常用。
  • old 区（冷端，约 3/8）：新读入的页先放在 old 区头部（midpoint insertion），链表尾最先淘汰。

为什么能防污染

  • 新页只进 old 区，不会一上来就占据 young。
  • 「innodb_old_blocks_time（默认 1000ms）内的访问不升级」：全表扫描往往在 1 秒内读完一页的多条记录就走，这些页停在 old 区，很快被淘汰。
  • 只有跨越时间窗口、被反复访问的页才升入 young → 热点得以保留。
关键参数：innodb_old_blocks_pct（old 区占比，默认 37%）、innodb_old_blocks_time（升级时间窗口，默认 1000ms）。

03访问一个页：逻辑读与物理读

无论查询还是更新，访问任何一个页都走同一套入口：先查哈希表看页在不在 Buffer Pool。

  • 命中 逻辑读：直接返回内存缓存页，调整 LRU 位置，纳秒~微秒级，无磁盘 I/O。
  • 未命中 物理读：从 Free 链表取空闲页（没有就淘汰 LRU 尾部冷页），从 .ibd 读 16KB 填入，挂到 LRU old 区。
若被淘汰的冷页是脏页，必须先刷盘再复用——这就是 Free 链表枯竭时，一次读可能伴随一次写的原因。

预读 read-ahead

InnoDB 会提前批量读入可能用到的页：线性预读（顺序访问一个区超过阈值 innodb_read_ahead_threshold → 预读下一个区的 64 页）和随机预读（默认关闭）。预读的页同样只进 old 区，避免污染。

04查询链路（内存侧）

读操作是最轻的路径：不产生 redo、不产生 undo（除非 MVCC 需要读 undo 构造旧版本），只可能产生页读入和 LRU 调整。

第 ⑤ 步的 MVCC 是重点：InnoDB 按事务的 ReadView 判断每个版本是否可见，若当前页上的版本太新（不该被本事务看到），就顺着记录的 DB_ROLL_PTR 去 undo 链里找旧版本构造出可见的那一版。这也是查询可能读 undo 的唯一情形。

05更新链路与 WAL

写操作（INSERT / UPDATE / DELETE）的核心是 WAL（Write-Ahead Logging，预写日志）：先写日志，数据页延迟落盘。这是 InnoDB 高性能且不丢数据的根基。

为什么 WAL 能既快又安全

  • 快：数据页改动是随机写（页散落在文件各处），而 redo 是顺序追加写。提交时只需把 redo 顺序刷盘，数据页留在内存慢慢异步刷——把随机 I/O 换成了顺序 I/O。
  • 安全：提交成功 = redo 已落盘。即便此刻数据页还在内存（脏页），宕机后也能用 redo 重做恢复。

redo 落盘策略 innodb_flush_log_at_trx_commit

值行为权衡
1（默认）每次提交都把 redo 刷到磁盘最安全，不丢已提交事务；性能略低
2每次提交写到 OS 缓存，每秒刷盘一次MySQL 崩溃不丢，OS 崩溃可能丢 1 秒
0每秒由后台线程刷一次最快，崩溃可能丢 1 秒数据
undo 也是数据。undo 写在 undo 页里（同样在 Buffer Pool 中），undo 页的修改本身也会产生 redo——所以崩溃后 undo 也能被 redo 恢复，回滚才有依据。

06Change Buffer

当一个非唯一二级索引页不在 Buffer Pool 时，如果为了一次写就把它从磁盘随机读进来，代价很高。Change Buffer 把这些改动暂存下来，等该页因别的原因被读入时再一次性合并。

  • 适用：非唯一二级索引的 INSERT / DELETE-mark / purge，且目标页不在内存。
  • 不适用：唯一索引（必须读页校验唯一性）、聚簇索引、页已在内存。

收益：把多次随机写合并为一次，大幅减少二级索引维护的随机 I/O，对写多读少的场景尤其有效。代价：占用一部分 Buffer Pool；崩溃恢复需重放；某些读多场景可能反而增加开销。参数 innodb_change_buffer_max_size 控制其占比。

07持久化闭环：Redo / 检查点 / 恢复

这一节把内存和磁盘接成闭环：脏页如何落盘、redo 如何顺序写、检查点如何回收日志、崩溃后如何恢复。

脏页何时刷盘（4 个触发点）

  • LRU 淘汰：冷端要复用一个脏页时，先把它刷盘。
  • Flush 链表：后台线程按 oldest LSN 持续地刷。
  • redo 空间不足：环形 redo 日志快写满时，强制刷脏页以推进检查点。
  • 正常关闭：shutdown 时把所有脏页刷净（innodb_fast_shutdown 影响程度）。

检查点 Checkpoint

检查点是一条水位线 checkpoint_lsn，表示「这之前的脏页都已落盘」。它之前的 redo 不再被需要，对应的日志空间可以被环形复用。InnoDB 用模糊检查点（fuzzy checkpoint）——不停机、持续小步推进，避免一次性刷大量脏页造成性能抖动。

崩溃恢复：先 redo 后 undo

  1. redo 前滚（redo / roll-forward）：从 checkpoint_lsn 开始重放 redo，把「已提交但数据页未落盘」的改动补回到页上。恢复后内存页与提交时一致。
  2. undo 后滚（undo / rollback）：对崩溃时未提交的事务，用 undo 撤销其已经写到页上的改动。
最终效果：已提交的不丢（持久性 D）、未提交的不留（原子性 A）。redo 保证 D，undo 保证 A，两者配合完成崩溃恢复。

08关键结论速查

问题结论
上层会直接读写磁盘吗？不会。只跟 Buffer Pool 交互，未命中才由后台做物理 I/O
LRU 为什么分 young/old？防止全表扫描/预读把热点页挤出缓存（midpoint + 时间窗口）
查询会写日志吗？不写 redo/undo；仅 MVCC 需要时读 undo 构造旧版本
WAL 是什么？先写 redo 日志再延迟刷数据页，把随机写换成顺序写
提交了数据就落盘了吗？不一定。提交=redo 落盘；数据页可能还是脏页，靠 redo 兜底
redo 和 undo 分工？redo 重做保证持久性(D)，undo 回滚保证原子性(A)
Change Buffer 干什么？暂存非唯一二级索引写改动，合并随机 I/O
脏页什么时候刷盘？LRU 淘汰 / Flush 后台 / redo 不足 / 正常关闭
检查点的作用？推进已落盘水位线，回收可复用的 redo 空间
崩溃恢复顺序？先 redo 前滚补齐已提交，再 undo 后滚撤销未提交
闭环一句话：读靠缓存命中，写靠 WAL 顺序日志，安全靠 redo+undo+检查点。磁盘侧的段/区/页是「数据躺在哪」，内存侧的 Buffer Pool 是「数据怎么被高效又安全地用起来」。
