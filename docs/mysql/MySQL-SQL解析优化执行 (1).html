02

词法分析：字符流 → token

SQL 是一串字符，机器无法直接理解。第一步词法分析（Lexical Analysis / 扫描 Scanning）把字符流切成一个个有意义的词法单元（token）。MySQL 由词法器（源码 sql/sql_lex.cc，配合 lex.h 的符号表）完成。

token 的分类

词法器读字符、按最长匹配原则切出 token，每个 token 带类型和值：

类型例子说明
关键字 keywordSELECT FROM WHERE语言保留的词，对应文法的终结符
标识符 identifier表名、列名、别名用户起的名字；反引号包裹可与关键字同名
字面量 literal18 'abc' 3.14数字、字符串、日期等常量
运算符/标点= > , ( )比较、算术、分隔符

词法器还顺手做的事

  • 跳过空白与注释：空格、换行、-- ...、/* ... */ 不产生 token（但 /*+ ... */ 优化器提示会被特殊保留，见第 24 章）。
  • 识别边界：正确切分字符串引号、转义、数字与小数点，处理字符集与大小写（标识符大小写敏感性取决于系统与平台）。
  • 区分保留字与非保留字：MySQL 关键字分保留（reserved）和非保留（non-reserved）两类——非保留字（如 STATUS、BEGIN）在不引起歧义时可直接当标识符用，保留字（如 SELECT）必须加反引号才能当列名。
词法 vs 语法的分工

词法器只认"词"，不关心词的排列是否合法——FROM SELECT id 在词法层完全合法（三个有效 token），要到语法层才会被判为错误。这种"先分词、再分析结构"的两级设计，是几乎所有编译器/解释器的标准做法：词法器处理字符级复杂度，让语法分析器只需面对干净的 token 流。下一章看语法分析如何把这串 token 组装成树。

03

语法分析①：文法与 LALR(1) 移进-归约

语法分析（Syntax Analysis / Parsing）把 token 流按 SQL 的文法（grammar）组装成一棵树。MySQL 的语法分析器由 Bison（GNU 版 yacc）从文法文件 sql/sql_yacc.yy 生成，是一个 LALR(1) 类型的自底向上分析器。本章讲清"文法是什么"和"LALR 怎么分析"。

文法：用产生式描述语言

文法由一组产生式规则（production）构成，每条形如"非终结符 → 符号串"，意思是"左边这个结构可以由右边这串组成"。两类符号：

  • 终结符（terminal）：词法器产出的 token，是文法的"叶子"，不能再展开（如 SELECT、标识符、,）。
  • 非终结符（non-terminal）：语法结构的抽象名字，可继续展开（如 query、select_list、expr）。
// 高度简化的 SELECT 文法（示意，非真实 yacc 写法）
query       → SELECT select_list FROM table_ref where_opt
select_list → '*'
            | select_list ',' column     // 递归：支持任意多列
            | column
where_opt   → /* 空 */
            | WHERE expr
expr        → column op_compare value
            | expr AND expr
            | expr OR expr
            | '(' expr ')'

LALR(1)：自底向上的移进-归约

LALR(1) 用一个栈加向前看 1 个 token（lookahead）来分析。它自底向上工作，每步只做两个动作之一：

  • 移进 shift：把当前 token 压栈，读下一个。（"还凑不成一个结构，先收着"）
  • 归约 reduce：栈顶若干符号正好匹配某条产生式的右侧，就把它们弹出、压入左侧非终结符。（"凑成一个结构了，打包"）

分析器内部其实是一个状态机，用两张表驱动：ACTION 表（在某状态遇到某 token，该 shift 还是 reduce）和 GOTO 表（归约出非终结符后跳到哪个状态）。Bison 在编译期就把文法编译成这两张表。

为什么是 LALR(1) 而非递归下降

解析器主要分两大流派：自顶向下（如手写递归下降、LL）和自底向上（LR/LALR）。MySQL 选 LALR(1)（经 Bison 生成），因为它能处理的文法类更广（尤其左递归，如上面 select_list → select_list ',' column 这种"列表"结构，自底向上天然支持，而 LL 需要改写），且由工具自动生成、维护成本低。代价是文法冲突（下一章）需要靠优先级声明来消解，且生成的表较大、报错信息不如手写器友好。SQL 语法庞大复杂，LALR(1) 的表达力是关键考量。

04

语法分析②：运算符优先级与冲突消解

上一章的文法有个隐患：expr → expr AND expr | expr OR expr 是有歧义的。面对 a OR b AND c，分析器在某一步会既能移进、又能归约，不知道该听谁的——这就是 移进-归约冲突（shift-reduce conflict）。本章讲它从哪来、怎么解。

歧义：a OR b AND c 该怎么结合

当栈里是 expr OR expr、向前看到 AND 时，有两条路：

  • 归约：先把 expr OR expr 打包成一个 expr → 结果是 (a OR b) AND c。
  • 移进：先把 AND 收进栈，等 c 来了先结合 b AND c → 结果是 a OR (b AND c)。

两种解析语义完全不同！SQL 规定 AND 优先级高于 OR（像乘法高于加法），所以正确答案是 a OR (b AND c)，即应当移进。

用优先级声明消解冲突

Bison 不靠改写文法，而是靠显式声明运算符的优先级与结合性来裁决冲突。在 yacc 文件里用 %left / %right 按从低到高列出：

// 优先级从低到高（越靠下越高）；%left=左结合
%left  OR
%left  AND
%right NOT
%left  '=' '<' '>'     // 比较运算符
%left  '+' '-'
%left  '*' '/'           // 乘除最高

规则：遇到 shift-reduce 冲突时，比较"栈顶规则的优先级"与"向前看 token 的优先级"——谁高听谁；相同则看结合性（左结合则归约、右结合则移进）。AND 优先级高于 OR，故 a OR b · AND 选择移进，得到正确的 a OR (b AND c)。

两类冲突 + 实践含义

LALR 有两类冲突：移进-归约冲突（如上，靠优先级解）和归约-归约冲突（栈顶能匹配多条产生式，更棘手，常需改写文法）。Bison 编译文法时若发现未消解的冲突会告警。对使用者的含义：正因为优先级是这样定义的，写 SQL 时 WHERE a=1 OR b=2 AND c=3 实际等价于 a=1 OR (b=2 AND c=3)——很多逻辑 bug 源于误以为是 (a OR b) AND c。不确定时加括号，既消除歧义也让意图清晰。这正是编译原理细节直接影响日常 SQL 正确性的地方。

05

AST 构造与语义预处理

语法分析在移进-归约的同时构造出一棵树，并在之后做语义检查。本章讲"树长什么样"和"语义关怎么过"。

从解析树到 AST

每次归约时，分析器就创建一个树节点，把刚归约的子结构挂上去——于是整个移进-归约过程自底向上地搭出一棵树。原始的、严格对应文法的树叫解析树（parse tree / 具体语法树），它含很多语法噪声（如括号、分隔符节点）。MySQL 8.0 会进一步整理成更干净的抽象语法树（AST）——节点是 Parse_tree_root、Query_block、Item（表达式）等类，去掉冗余、只保留语义结构，便于优化器遍历改写。

预处理：语法对 ≠ 能执行

有了 AST，还要过语义关（预处理 / preprocessor，源码 setup_* 系列）。语法分析只保证"结构合法"，预处理才检查"是否真的说得通"：

  • 名称解析：表、列是否真实存在？查数据字典确认。不存在 → Unknown column / Table doesn't exist。
  • 歧义消解：多表有同名列时，SELECT id 的 id 到底指哪张表的？无法确定 → Column 'id' is ambiguous。
  • 权限检查：当前用户有没有权限读这些表/列？
  • 展开：把 * 展开成具体列、把视图展开成其定义、补全默认库名等。
  • 类型与函数检查：函数是否存在、参数个数/类型是否匹配。
三关递进，缺一不可

到此解析全部完成，三关层层递进：词法（第 2 章，"是不是合法的词"）→ 语法分析（第 3–4 章，"词的排列是不是合法结构"，靠文法 + LALR 移进归约 + 优先级建 AST）→ 语义预处理（本章，"这些名字是否真实、是否有权、能否说通"）。三关都过，才产出一棵语义明确的 AST，交给优化器。提醒：MySQL 8.0 已移除查询缓存，故没有"解析前查缓存命中直接返回"那一步，每条查询都会完整走解析与优化。

06

SQL 执行：火山模型

优化器产出执行计划后，执行器按计划驱动引擎取数。组织方式是经典的火山模型（Volcano / 迭代器模型）。执行计划是一棵算子树，每个算子（扫描、过滤、连接、排序、聚合）实现统一的 next() 接口，每次返回一行。上层算子调用下层 next() 拉取数据：控制自顶向下，数据自底向上一行行涌出。

两个延伸

① 执行器/引擎边界。执行器调用存储引擎 handler 接口（"读下一行""按索引定位"），怎么从 B+ 树/Buffer Pool 取数是引擎（前七篇）的事。② 火山模型的代价。逐行 next() 有虚函数开销，分析型大数据量不划算，现代数据库引入批量处理与算子合并，但理解经典火山模型是看懂执行计划树的基础。

07

优化器总览：架构与 RBO / CBO

优化器是 Server 层的"大脑"。它接过预处理后的 AST，分两大步变成最优执行计划：

  • 逻辑优化（查询变换）—— 等价改写 AST，不看成本，只为变成更易优化的形状（化简条件、子查询转半连接、派生表合并）。第 11–13 章。
  • 物理优化（基于成本选路）—— 为可行方案估算成本，选最便宜的：用哪个索引、哪种访问方式、哪种连接算法、什么连接顺序。第 14–23 章。
RBO 基于规则CBO 基于成本
选路依据固定优先级规则估算成本，选最小
看数据分布吗不看看（靠统计信息）
典型问题索引命中 90% 也照用→慢统计不准时会估错
MySQL 现状少数固定场景保留主体，以 CBO 为主
为什么 CBO 会"选错"

CBO 的聪明完全建立在成本估算准确上，而成本估算又依赖统计信息。统计过期/不准 → 估算偏差 → 选次优计划（放着好索引不用、选错连接顺序）。这就是"突然走错索引"第一反应是 ANALYZE TABLE 的原因。第 8–10 章讲它如何估、估什么；第 25 章的 Optimizer Trace 能让你亲眼看到它的估算与选择。

08

成本模型：成本怎么算

CBO 口中的"成本"是个无量纲相对值，只用来比较计划优劣，主要两部分相加：

  • I/O 成本：读多少页 × io_block_read_cost（≈1.0）。读"已在 Buffer Pool 的页"比"磁盘页"便宜。
  • CPU 成本：评估多少行 × row_evaluate_cost（8.0 ≈0.2）。
// 表 1000 页、10 万行
全表扫描 ≈ 1000 × 1.0 + 100000 × 0.2 = 21000
索引命中 200 行(回表200) ≈ 200 × 1.0 + 200 × 0.2 ≈ 240   → 选索引
索引命中 8 万行(区分度低) ≈ 8万+                          → 反比全表贵, 放弃索引
成本常量外置且引擎相关

8.0 把成本常量放系统表：mysql.server_cost（Server 层操作）与 mysql.engine_cost（引擎层，如 memory_block_read_cost vs io_block_read_cost）。一般无需改，但能解释"同一 SQL 在内存命中率不同时选择不同"。成本的核心驱动是"预估碰多少行/页"，这个预估来自统计信息（第 9 章）与行数估算（第 10 章）。

09

统计信息：基数与直方图

成本模型要估"碰多少行"，靠两类统计：索引基数和列直方图。

① 索引基数 Cardinality

基数 = 索引列上不同值的个数（估算），衡量区分度。来源：随机采样若干页（innodb_stats_persistent_sample_pages 默认 20）估算，是近似值。存储：8.0 默认持久化（innodb_stats_persistent=ON）于 mysql.innodb_index_stats。更新：数据变化超约 10% 自动重算，或手动 ANALYZE TABLE。

② 列直方图 Histogram（8.0 新增）

基数只说"有几种值"，不说"各占多少"。直方图刻画分布，尤其服务无索引列。两种：singleton（取值少，每值一桶记占比）、equi-height 等高（取值多，每桶覆盖大致相等行数）。建法：ANALYZE TABLE t UPDATE HISTOGRAM ON col;，存 mysql.column_statistics。

统计不准的连锁后果

统计是 CBO 的燃料，燃料脏了决策就歪：基数采样偏差→误判区分度→选错索引；缺直方图→倾斜列按均匀分布估→严重高/低估命中行→选错计划甚至连接顺序。莫名慢查询/走错索引的标准动作：ANALYZE TABLE 重收基数 + 为关键无索引过滤列建直方图。

10

范围优化与行数估算（index dive）

对 WHERE col BETWEEN a AND b 这类范围条件，优化器要估命中多少行——这直接决定"走索引值不值"。负责此事的是范围优化器（Range Optimizer）。

index dive：钻进索引数一数

光靠基数（平均）估范围不够准。MySQL 用 index dive（索引下潜）：真的钻进 B+ 树，定位范围的起点和终点，估算两点间隔多少条记录，比平均基数准得多，尤其对分布不均的数据。

两个实战要点

① IN 列表过长会放弃 dive。每个范围都要 dive 一次，当等值范围数超过 eq_range_index_dive_limit（默认 200）时改用基数平均估算——超长 IN (...) 偶尔行数估偏选错计划，源于此。② 范围优化有内存上限。条件过于复杂时受 range_optimizer_max_mem_size 限制而放弃精细分析，退回粗估。

11

逻辑优化①：条件化简与等价变换

逻辑优化是不看成本的等价改写，把查询变成更易物理优化的形态。先看"条件层面"的变换：

  • 常量折叠：WHERE a = 5 + 3 → a = 8。
  • 等值传递：WHERE a = b AND a = 10 → 推出 b = 10，让 b 也能走索引。威力很大——把一张表的过滤"传染"给等值连接的另一张表。
  • 常量表检测：某表被主键/唯一键等值锁定到至多一行时，当作常量表 const，优化阶段就先读出来代入其它条件。EXPLAIN type=const/system。
  • 恒真/恒假消除：去掉 1=1；WHERE id=1 AND id=2 矛盾 → 直接返回空（EXPLAIN Impossible WHERE），根本不查表。

外连接转内连接（重要）

若 LEFT JOIN 的 WHERE 含对右表列的非空过滤（如 WHERE b.x=5），"右表为 NULL"的行本会被滤掉，外连接等价于内连接。好处：内连接对连接顺序无限制，优化器能自由选驱动表，常解锁更优计划。

为什么先做这些

这些变换都在物理优化之前做，为物理优化铺路：等值传递让更多表能用索引、外连接转内连接放开连接顺序、Impossible WHERE 直接短路。规整后的查询，物理优化才能算出更好的账。

12

逻辑优化②：子查询与半连接四策略

子查询是优化重灾区。朴素执行可能对外层每一行都跑一遍子查询。优化器把 IN 子查询改写成半连接（semi-join），用连接算法高效执行。半连接 = 连接，但每条外层行只要"找到至少一个匹配就保留，且不因多个匹配而重复"（去重语义）。优化器有四种实现，按成本择优：

策略做法适合
FirstMatch外层每行去内层找，找到第一个就停内层每个外层行匹配多时，避免重复
Materialization子查询结果去重物化成临时表（带哈希索引），外层查它子查询结果小且被反复探测
Duplicate Weedout当普通连接做，事后用临时表去重连接高效、去重代价低
LooseScan利用内层索引有序，每个不同值只取一个内层连接列有索引且重复值多

四策略的执行直觉（同一查询 o.uid IN (SELECT id FROM u WHERE vip=1)）

-- FirstMatch：对每个 o，扫 u 找到一个匹配 uid 就停，跳到下个 o
-- Materialization：先 SELECT DISTINCT id FROM u WHERE vip=1 存临时表 t_m，
--                  再 o JOIN t_m ON o.uid=t_m.id
-- DuplicateWeedout：o JOIN u 正常连(可能产生重复 o)，最后按 o 主键去重
-- LooseScan：u 在 id 上有序，扫 u 时每个不同 id 只取一行，天然不重复
实践

多数情况无需手动把 IN 改 JOIN，优化器会做；老版本或它没选好时手动改写仍有效。用 EXPLAIN 看是否出现 <subquery>、FirstMatch、materialized 判断走了哪条；子查询结果集大小影响策略选择，必要时给子查询过滤列建索引。

13

逻辑优化③：派生表 合并 vs 物化

FROM (SELECT ...) AS t 派生表与视图，优化器有两种处理，差别巨大：

① 合并 Merge（首选，更快）

把派生表定义摊平合并进外层，变成单层查询。这样派生表的 WHERE/连接能和外层一起优化，外层条件可下推、索引可用上。如 SELECT * FROM (SELECT * FROM t WHERE a>5) d WHERE d.b=3 → SELECT * FROM t WHERE a>5 AND b=3。

② 物化 Materialize（兜底，较慢）

派生表无法合并时（含 GROUP BY/DISTINCT/LIMIT/窗口函数/UNION），先执行、结果存进临时表，外层再查临时表。代价：中间结果落地，且临时表上条件下推、索引利用受限。

实践要点

① 尽量让派生表可合并，能在外层做的过滤别塞进派生表再加聚合。② 物化临时表默认在内存（TempTable），超限转磁盘变慢，关注 tmp_table_size/temptable_max_ram。③ 8.0 会给物化临时表自动加索引（auto_key）加速外层连接，EXPLAIN 见 <auto_key0>。④ 非递归 CTE（WITH）处理方式与派生表一致。

14

访问路径与索引合并

进入物理优化。对单表，优化器选访问路径，再用成本模型比较。候选：const/system（主键/唯一等值，至多一行，优化阶段就读出）、全表扫描 ALL（命中占比大时最优）、索引访问（ref等值/range范围，可能回表）、覆盖索引（免回表，第 15 章）、索引合并 index_merge。

索引合并 Index Merge

WHERE 用 OR/AND 连接多个不同索引各自能用的条件时，可分别用多个索引各扫一遍再合并，而非放弃索引走全表。三种：union（OR 并集）、intersection（AND 交集）、sort-union（带排序并集）。EXPLAIN type=index_merge。

选路 = 成本 + 统计 + 行数估算

优化器对每条候选路径，用第 9 章统计 + 第 10 章 dive 估命中行数，用第 8 章成本算成本，选最小。"为什么没走我的索引"几乎总是：估"走它成本高于别的路径"（区分度低/命中太多/回表太贵），或统计不准。下一章讲怎么把"回表太贵"降下来。

15

减少回表：覆盖索引 / ICP / MRR

走二级索引常要回表（拿主键再查聚簇索引），命中多时大量随机 I/O是性能杀手。优化器三件武器对付回表：

① 覆盖索引 —— 根本不回表

查询所需所有列都在二级索引叶子里（索引列 + 主键），无需回表。EXPLAIN Extra: Using index。把高频查询返回列纳入联合索引即可制造。

② 索引条件下推 ICP —— 回表前先过滤

WHERE 有能用索引列但不能定位的条件（后缀 LIKE、跳列）时，ICP 让引擎在回表前用索引叶子里现成的列值先过滤，只有满足的才回表。EXPLAIN Using index condition。把过滤从"昂贵回表后"提前到"便宜索引扫描时"。

③ 多范围读 MRR —— 把随机回表变顺序

二级索引按索引列有序，但它指向的主键是乱序的，逐行回表对聚簇索引是随机 I/O。MRR：先把要回表的一批主键收集、排序，再按主键顺序回表——随机 I/O 变顺序 I/O。

三件武器的关系

覆盖索引釜底抽薪——不回表，最优；ICP少回表——回表前滤掉无关行；MRR回得快——非回不可时随机变顺序。优先级：能覆盖就覆盖，否则 ICP 减少回表行数 + MRR 加速剩余回表。三者常协同出现在同一计划。ICP、MRR 默认开启（受 optimizer_switch 控制，第 24 章）。

16

索引跳跃扫描 Skip Scan

这是 MySQL 8.0.13 新增的优化，专门挽救"违反最左前缀"的联合索引查询，是对第六篇最左前缀规则的一个重要补充。

问题：缺了最左列，索引就废了吗

设联合索引 idx(a, b)，查询 WHERE b = 100（没有 a 的条件）。按第六篇规则，缺最左列 a，b 在整棵索引里是无序的，传统上只能全表扫描。但如果 a 的不同值很少（低基数，如 a 只有 male/female 两种），情况可以改善。

Skip Scan：把缺失的前缀"枚举"出来

跳跃扫描的思路：既然 a 只有少数几个值，那就把 a 的每个值都"补"上，分别当作 a=val1 AND b=100、a=val2 AND b=100… 各走一次索引区间扫描，再合并。相当于优化器替你枚举了缺失的最左列，把一个"用不了索引"的查询拆成若干"能用索引"的子扫描。

适用边界

Skip Scan 只在最左列基数很低时才划算。若 a 有上万种值，枚举上万次索引扫描，还不如直接全表——优化器会基于成本判断，基数高时不会选它。所以它是个"锦上添花"的补救，不能因为有 Skip Scan 就忽视最左前缀：设计索引时仍应让查询能用上最左列，Skip Scan 只兜底"最左列恰好低基数又没被查询用到"的特例。EXPLAIN Extra: Using index for skip scan 标识它生效，受 optimizer_switch 的 skip_scan 控制。

17

连接顺序与搜索算法

多表连接时，优化器要决定表以什么顺序连接，这是物理优化里组合空间最大的决策。本章深入它怎么搜。

为什么顺序关键：前缀成本累积

连接逐表"长出"中间结果，优化器按"连接前缀"逐步累积成本。核心原则：让过滤性强（结果行数少）的表靠前，尽早把中间结果压小，后续每步都在小集合上做，总成本就低。

搜索过程：贪心扩展 + 前缀剪枝

N 张表有 N! 种顺序，全穷举不现实。MySQL 用一种有限深度的贪心搜索（源码 greedy_search / best_extension_by_limited_search）：从空前缀出发，每步在"还没加入的表"里挑使当前前缀成本增量最小的表加进来，并对明显差的分支剪枝。看一个三表 A/B/C 的搜索片段：

搜索深度与干预

optimizer_search_depth 控制搜索深度：在"搜得够全"与"优化本身开销"间权衡，设大更可能找到最优但优化变慢，默认（自适应）通常够用。驱动表（outer）先遍历，被驱动表（inner）对其每行匹配；关键是驱动表小 + 被驱动表连接列有索引（INLJ 高效前提，第 18 章）。干预：极少数情况可用 STRAIGHT_JOIN 或 JOIN_ORDER 提示强制顺序，但优化器选错更常源于统计不准/缺索引——先 ANALYZE TABLE、补索引。强制顺序是双刃剑，数据变化后可能反成枷锁。

18

嵌套循环族 NLJ / INLJ / BNL / BKA

嵌套循环是最基础的连接思想，长期是 MySQL 唯一连接方式（8.0 才加 Hash Join）。四个变体：

  • NLJ 简单嵌套循环：对驱动表每行，把被驱动表从头扫一遍，O(R×S)。被驱动表大且无索引时极慢，几乎不用纯 NLJ。
  • INLJ 索引嵌套循环（主力）：被驱动表连接列有索引，对驱动表每行用索引直接定位，降到约 O(R×log S)。EXPLAIN 被驱动表 ref/eq_ref。
  • BNL 块嵌套循环：被驱动表无索引时，把驱动表一批行攒进 join_buffer，被驱动表只扫一遍与整批比对。8.0.20 起在等值连接被 Hash Join 取代。
  • BKA 批量键访问：INLJ 对被驱动表每次定位可能随机回表。BKA 把驱动表一批行的连接键攒起来排序，用 MRR 批量、有序地查被驱动表——本质 INLJ + join_buffer + MRR，把随机回表变顺序。EXPLAIN Using join buffer (Batched Key Access)，需开 batched_key_access。
现代版本的连接选择

有索引：默认 INLJ；大量回表时 BKA 可进一步顺序化。无索引等值连接：8.0.20 起走 Hash Join（取代 BNL）。给连接列建索引让其走 INLJ，仍是最常见也最有效的连接优化。

19

Hash Join 详解

MySQL 8.0.18 引入，专治"被驱动表连接列无索引的等值连接"。复杂度 O(R+S)，远胜嵌套循环 O(R×S)。两阶段：Build（选较小表，按连接列建内存哈希表）+ Probe（遍历较大表，每行算哈希去哈希表 O(1) 查找）。

何时用 + 怎么看

触发：等值连接 + 被驱动表连接列无可用索引。观察：普通 EXPLAIN Extra: Using join buffer (hash join)；EXPLAIN FORMAT=TREE / ANALYZE 直接显示 Hash Join 算子（第 25 章）。本质：把嵌套相乘换成哈希 O(1) 查找，对"大表×大表、无索引、等值"最显著。

20

Sort-Merge Join 详解

Sort-Merge Join（排序归并连接）是数据库理论三大经典连接算法之一。先讲原理，再诚实说明 MySQL 的实际情况。

原理：各自排序，再归并

Sort：两表都按连接列排序（来自索引已有序则跳过）。Merge：双指针同时从两个有序序列头部往后走，像拉链——哪边小推进哪边，相等则输出。两表各扫一遍。复杂度 O(R logR + S logS) + O(R+S)；输入已有序时只剩 O(R+S)，且能高效处理范围连接（如 R.k ≥ S.k）——Hash Join 做不到。

关于 MySQL 的实话（重要）

MySQL 并没有实现独立的 Sort-Merge Join 连接算子——常见误解，务必讲清：① MySQL 历史上只有嵌套循环连接，8.0 才补 Hash Join，从未提供专门的 sort-merge join 算子。② Sort-Merge 是 Oracle、PostgreSQL、SQL Server 的标配。③ MySQL 用别的手段达成类似目的：连接键有索引走 INLJ（索引本身有序），无索引等值连接走 Hash Join；MySQL 大量用排序（filesort，第 22 章）但那是为 ORDER BY/GROUP BY，不是连接算子。记住：讲算法 Sort-Merge 是三大经典之一；落到 MySQL 实现，连接武器只有 INLJ（含 BKA）+ Hash Join 两件。

21

三种 JOIN 对比与选择

算法复杂度适用MySQL 支持EXPLAIN 标志
INLJ（含 BKA）O(R×log S)被驱动表连接列有索引主力，全版本ref/eq_ref（BKA：join buffer）
BNLO(R×S/缓冲)无索引（旧版兜底）8.0.20 起被 Hash 取代(Block Nested Loop)
Hash JoinO(R+S)等值 + 无索引8.0.18+(hash join)
Sort-MergeO(R logR+S logS)大表等值/范围、已有序MySQL 无独立实现—（他库才有）
实践结论

给连接列建索引、让连接走 INLJ，是绝大多数场景首选。只有连接列实在无法建索引（派生结果、表达式连接）时，Hash Join 才作高效兜底。看到 Hash Join 且慢，往往提示"被驱动表连接列该加索引"。Sort-Merge——记得它是经典算法，但 MySQL 里不必去找，INLJ 的索引有序性 + Hash Join 覆盖了它的场景。

22

排序、分组与去重优化

ORDER BY/GROUP BY/DISTINCT 的核心优化目标：尽量利用索引有序性，避免昂贵的额外排序和临时表。

ORDER BY：索引有序 vs filesort

  • 用索引有序（最优）：ORDER BY 列是某索引有序前缀时，顺着索引读即有序，无需排序，EXPLAIN 无 filesort。
  • filesort：无法用索引序时取出数据在 sort_buffer 排。Using filesort 不一定是磁盘排序：放得下内存排，放不下才用临时文件归并。两种内存策略：全字段排序（排序列+返回列一起排，少回表）和 rowid 排序（只排序列+主键，排完回表，省内存多回表）。

GROUP BY：松散 vs 紧凑索引扫描

  • 松散索引扫描（最优）：利用索引有序跳着读，每分组只取少量记录（配合 MAX/MIN）。EXPLAIN Using index for group-by。
  • 紧凑索引扫描：扫索引连续范围但按索引序，无需额外排序/临时表。
  • 临时表 + 排序（最差）：无法用索引序时建临时表分组 + filesort。Using temporary; Using filesort 是性能警报。
实践

① 让 WHERE + ORDER BY/GROUP BY 列组成合适联合索引（注意第六篇最左前缀：等值列在前、排序列紧随、方向一致）。② 看到 Using temporary; Using filesort 同时出现多半要调索引。③ DISTINCT 本质同 GROUP BY，能走索引序则免临时表。④ 8.0 支持降序索引，混合方向 ORDER BY a ASC, b DESC 也能用索引序。

23

分页与 LIMIT 优化

分页是深分页性能塌方的重灾区。SELECT * FROM t ORDER BY id LIMIT 1000000,10 看似只要 10 行，实则必须先扫描并丢弃前 100 万行，偏移越大白扫越多；若走二级索引，前 100 万行还可能每行回表。

解法一：覆盖索引 + 延迟回表（deferred join）

先用覆盖索引定位那 10 行的主键（不回表，快），再用主键取完整行，把"100 万次回表"降为"10 次"：

-- 快：子查询只走覆盖索引拿主键(不回表)，外层只回表10次
SELECT t.* FROM t
JOIN (SELECT id FROM t ORDER BY create_time LIMIT 1000000,10) d ON t.id = d.id;

解法二：游标分页 / keyset pagination（最优）

不用 OFFSET，记住上一页最后一条的位置，下一页从那之后取，翻多深都恒定快：

SELECT * FROM t WHERE id > :last_id ORDER BY id LIMIT 10;  -- 索引直达, 零丢弃
取舍

游标分页最优但有约束：只能顺序翻、无法跳到第 N 页；排序键需唯一（或加主键做 tiebreaker：WHERE (sort_key,id) > (:k,:id)）。延迟回表保留跳页能力、改造小，是不能用游标分页时的折中。能用游标分页（信息流/无限滚动）就用它；需页码跳转的传统分页用延迟回表 + 限制最大翻页数。

24

优化器提示与开关

优化器偶尔失手（统计不准、估算偏差），MySQL 提供干预手段。

① 优化器开关 optimizer_switch

一组布尔开关控制各项优化：index_merge、mrr、batched_key_access、semijoin、materialization、derived_merge、hash_join、skip_scan 等。一般保持默认，排查"某优化是否捣乱"时临时关某项验证。

② 优化器提示 Optimizer Hints（8.0 推荐）

写在 SQL 里的 /*+ ... */，只影响当条语句，比改全局开关精准安全：

提示作用
/*+ JOIN_ORDER(a,b,c) */指定连接顺序
/*+ INDEX(t idx) */ / NO_INDEX建议用/不用某索引
/*+ HASH_JOIN(t1,t2) */ / NO_HASH_JOIN强制/禁止哈希连接
/*+ BKA(t) */ / NO_BKA开/关批量键访问
/*+ MAX_EXECUTION_TIME(1000) */限制语句最长执行 1 秒

③ 索引提示 Index Hints（老语法）

USE INDEX 建议、FORCE INDEX 强制、IGNORE INDEX 忽略。

提示是双刃剑

能不用就不用。提示把"动态决策"变成"写死"，今天对、明天数据变了可能反成枷锁。正确顺序：先 ANALYZE TABLE 修正统计、补索引、改写 SQL；都试过仍选错，才考虑提示，优先语句级 /*+ Hints */ 而非改全局 optimizer_switch。

25

EXPLAIN 与 Optimizer Trace

看懂优化器决策，靠 EXPLAIN（看结果）和 Optimizer Trace（看推导过程）。

EXPLAIN 的几种格式

形式给什么何时用
EXPLAINtype/key/rows/Extra 概览快速看走没走索引、有没有 filesort
FORMAT=JSON详细成本、各步行数、过滤比深挖为何这样选（含 query_cost）
FORMAT=TREE算子树形结构看 Hash Join / 连接嵌套结构
EXPLAIN ANALYZE真正执行给出实际行数与耗时对比"预估 vs 实际"，定位偏差

关键列：type（梯子 const>eq_ref>ref>range>index>ALL）、key/key_len（用了哪个索引/几字节）、rows × filtered（预估扫描行 × 过滤后保留比，连接时上一步 rows×filtered 是下一步驱动行数）、Extra（Using index/index condition/MRR/filesort/temporary/hash join）。

Optimizer Trace：看见优化器的"思考"

EXPLAIN 给的是结论，Optimizer Trace 给的是推导全过程——优化器考虑了哪些候选、各自估了多少成本、为什么选了这个、为什么淘汰那个，全以 JSON 记录：

SET optimizer_trace = 'enabled=on';
SELECT ... ;                                  -- 执行目标查询
SELECT * FROM information_schema.optimizer_trace\G  -- 取出 trace
SET optimizer_trace = 'enabled=off';

trace 的 JSON 分三段：join_preparation（解析准备）、join_optimization（核心：条件化简、范围分析、各索引的成本估算、连接顺序搜索）、join_execution。在 join_optimization 里能看到每个候选索引的 rows_estimation、每种连接顺序的 considered_execution_plans 及其 cost——这是回答"优化器到底在想什么、为什么不选我的索引"的终极工具。

EXPLAIN ANALYZE + Trace 的配合

两者互补：EXPLAIN ANALYZE 告诉你"预估 vs 实际差多少"（发现问题）——若某算子预估 10 行实际 100 万行，就抓到了估算失准的元凶；Optimizer Trace 告诉你"它当初为什么这么估、这么选"（定位根因）——往往指向统计信息不准（回第 9 章 ANALYZE TABLE/建直方图）或缺合适索引。先用 ANALYZE 发现偏差，再用 Trace 追溯根因，是优化器调优的黄金组合。注意 ANALYZE 会真实执行（写语句慎用），Trace 有大小上限 optimizer_trace_max_mem_size。

26

一次完整的查询优化实战

前面拆开讲了各项优化。本章用一个慢查询把它们串起来，演示真实的优化流程：看 EXPLAIN 找症结 → 逐步施治 → 验证。场景：订单表 orders（千万行）、用户表 users，查"某城市 VIP 用户最近的 10 笔订单"。

-- 初始慢查询（执行 8 秒）
SELECT o.* FROM orders o
WHERE o.user_id IN (SELECT id FROM users WHERE city='北京' AND vip=1)
  AND o.status = 'paid'

动态 DDL：ALGORITHM 与 LOCK

动态 DDL（在线 DDL，Online DDL）：在不阻塞业务读写的前提下改表结构，呼应第五篇 MDL。

为什么传统改表会锁表

早期改表用拷贝法：建新结构临时表 → 逐行拷贝旧数据 → 替换旧表。拷贝期间禁止写入（甚至禁读），大表可能锁几小时。Online DDL 为消除此阻塞而生。

两个子句：ALGORITHM 与 LOCK

ALTER TABLE ... ALGORITHM=? LOCK=? 控制改表方式与并发度：

ALGORITHM做法数据拷贝/重建并发 DML
INSTANT（8.0.12+）只改元数据，不碰数据行完全不拷贝几乎不阻塞，秒级
INPLACE原表原地重建/修改可能重建，但原地多允许并发读写
COPY建临时表逐行拷贝（旧做法）全量拷贝阻塞写（至多允许读）
LOCK并发允许
NONE读 + 写都允许
SHARED只允许读，写阻塞
EXCLUSIVE读写都阻塞
显式指定能"快速失败"

ALTER TABLE t ADD COLUMN x INT, ALGORITHM=INSTANT;——若不支持 INSTANT，MySQL 直接报错，而非悄悄退化成慢且锁表的 COPY。你第一时间知道"此变更没法秒级完成"，再决定换方案。指定 LOCK=NONE 同理。显式声明是生产改表的安全习惯。

28

INSTANT / INPLACE 原理

INSTANT：只改元数据

8.0 的杀手锏。以加列为例：不重写每一行，而是把"新列及其默认值"记在表的元数据（数据字典）里。旧行物理上没有新列，读取时若是"加列前的旧行"就用元数据里的默认值补上，对用户透明；新行插入时才物理带上。因完全不碰已有行，无论表多大都是毫秒级。范围逐版本扩大：8.0.12 末尾加列，8.0.29 起任意位置加列及更多操作。

INPLACE：原地重建 + Row Log 收集并发改动

加索引等必须扫描/重组数据的变更走 INPLACE：原地构建的同时，用一份"变更日志"吸收期间并发写：

  1. 1开始拿短暂 MDL 排他锁做准备，随即降级，放开并发 DML。
  2. 2后台扫描旧数据、构建新结构。期间用户增删改记入 row log，不打断业务。
  3. 3接近完成时再拿一次短暂排他锁，把 row log 积累的改动追平，切换、释放。
呼应第五篇：在线 ≠ 完全无锁

首尾那两个"短锁"——即便 Online DDL，开始和结束仍各需一个极短的 MDL 排他锁更新元数据。第五篇的"长事务挡住 DDL 的 MDL 写锁、排队的 DDL 又挡住所有后续查询 → 表假死"对 Online DDL 同样适用。生产改表：① 即便用 Online DDL，先确认无长事务占着这张表；② 给 DDL 设 lock_wait_timeout 快速失败；③ 超大表或需灰度时用 gh-ost / pt-online-schema-change，把锁窗口压到极致。

优化的统一心法

优化器的一切决策都建立在"预估要碰多少行/页"之上，而这个预估来自统计信息。所以做优化的三个根本动作：① 喂准统计（ANALYZE、直方图）让它估得准；② 建对索引（覆盖、最左前缀、连接列索引）给它更便宜的路；③ 写好 SQL（避免深分页 OFFSET、让子查询可转半连接、让排序走索引序）从源头减少工作量。把这三件做好，绝大多数查询都不需要强制提示。诊断时 EXPLAIN ANALYZE 发现预估与实际的偏差、Optimizer Trace 追溯根因。

与前七篇汇合

本篇"执行器"调用前七篇构筑的引擎能力：优化器选定索引（⑥）→ 执行器调引擎按 B+ 树定位 → 页在 Buffer Pool 找（②）/ 没有从表空间读（①）→ 当前读加锁（⑤）/ 快照读选版本（④）→ 改动写 redo/undo（③）→ 提交走两阶段（⑦）。八篇构成完整链路：Server 层把 SQL 翻译成最优计划，引擎层把计划落到磁盘字节。
