InnoDB 故障恢复原理

承接前两份《存储结构》（磁盘侧）与《Buffer Pool 内存侧链路》（内存侧），这份讲崩溃恢复： 机器突然断电、进程被 kill、操作系统 panic ——重启之后，InnoDB 凭什么能把数据库带回一致状态，且已提交事务一条不丢？
适用版本：MySQL 5.7 / 8.0，InnoDB 引擎。
核心论断：InnoDB 的崩溃恢复遵循 ARIES 算法（IBM 1992 年提出的工业标准），核心是 WAL（先写日志） + 三阶段恢复（Analysis → Redo → Undo） + 双写。 这套机制让数据库在任何崩溃时刻重启后都能回到"已提交全在、未提交全无"的一致状态——这就是 ACID 中 A 和 D 的兜底保障。

01总览：恢复要解决什么问题

崩溃发生在「内存改了、但还没全部刷到磁盘」的瞬间。内存全部丢失——Buffer Pool 里的脏页、活跃事务结构、未刷的 Log Buffer 全没了；而磁盘上幸存的是不完整的画面：数据文件 .ibd 里的页可能是旧版本、可能是新版本、甚至可能是撕裂的;redo log 里有崩溃前的修改记录(从 checkpoint_lsn 往后的部分)。恢复的任务就是用磁盘上幸存的这些信息,重建出崩溃瞬间的内存状态,再撤销那些不该留下的痕迹。

恢复要达成的三个目标(对应 ACID)

  • D 持久性 已提交事务的所有修改必须完整保留——靠 redo 前滚 把已落盘 redo 中的改动全部重放到页上。
  • A 原子性 未提交事务的所有痕迹必须彻底清除——靠 undo 后滚 对 Analysis 识别出的活跃事务逐一回滚。
  • C 一致性 数据库约束、B+ 树结构、链表完整性等不能被破坏——靠前两者 + 双写(保证页不撕裂) + MTR(保证物理修改原子)。

02LSN 体系：恢复的时间轴

整个恢复机制建立在 LSN(Log Sequence Number,日志序列号) 这个抽象之上。LSN 是字节级单调递增的——每写一字节 redo,LSN 就加一。所有页、日志、检查点都用 LSN 锚定彼此的相对位置,让"哪些改动已经做了、哪些还没做"在数字上变得可比较。

四个关键 LSN

LSN位置含义
redo_lsn内存Log Buffer 中下一个 redo 字节的位置（最新）
flushed_lsn内存↔磁盘已刷盘 redo 的最高位置；flushed_lsn ≤ redo_lsn
page_lsn每个页内该页最后一次被修改时对应的 redo LSN（页头里存着）
checkpoint_lsn系统级水位「此前所有脏页都已落盘」的水位线，恢复起点

四条不变式(理解恢复的关键)

  • flushed_lsn ≥ checkpoint_lsn ——redo 落盘进度始终领先于检查点水位。
  • page_lsn ≤ redo_lsn ——任何页的修改都对应一条已存在的 redo 记录(WAL 协议)。
  • 恢复起点 = checkpoint_lsn ——比这早的 redo 对应的修改必然都已在页上,不需要重放。
  • 重放幂等性:对一条 redo,若目标页的 page_lsn ≥ 这条 redo 的 lsn,说明这个修改已经应用过,跳过即可。这条性质让 redo 重放天然幂等,可以反复重启而不会重复执行修改。
LSN 体系的精髓:把"时间"变成"可比较的数字"。任何时刻只要知道两个 LSN,就能判断它们的先后,这是分布式日志、复制、备份恢复都依赖的基础。

03MTR 与 redo 记录

用户写的一个事务,在 InnoDB 内部并不是直接对页做修改,而是分解为多个 Mini Transaction(MTR)。MTR 是 InnoDB 对页操作的最小原子单位,介于"事务"和"单条 redo 记录"之间。理解 MTR 才能理解 redo 的正确粒度。

为什么需要 MTR 这一层

考虑 B+ 树插入一条记录:可能涉及叶子页的记录插入、链表调整,如果触发分裂还要修改父节点、新建页、调整兄弟页的双向链表。这些对多个页的修改必须"同生同灭"——不能只改了一半就把 redo 写出去,否则崩溃恢复重放 redo 时会得到一棵结构错乱的 B+ 树。MTR 把这组修改打包成一个不可分割的单元:要么全部写入 redo log buffer,要么一条不写。

redo 的"物理逻辑"性质

InnoDB 的 redo 既不是纯物理日志(完整页镜像,太大)也不是纯逻辑日志(SQL 语句,无法在低层重放)。它是物理逻辑混合的:

  • 物理部分:精确指向"哪个 space、哪个 page、哪个 offset"。
  • 逻辑部分:记录"对此处做了什么操作"(MLOG_REC_INSERT、MLOG_REC_UPDATE_IN_PLACE 等),而不是新值的字节镜像。
"物理逻辑"日志的优点是体积小、写入快;但代价是必须基于完整页才能重放。这是双写存在的根本原因——只有页本身完整,redo 才有用武之地。

04检查点:水位线推进

redo log 是环形复用的(8.0.30 前 ib_logfile0/1 循环写,8.0.30 起 #ib_redoN 文件组)。但 redo 不能无限增长也不能无脑覆盖——必须先保证它对应的脏页已经落盘,否则覆盖了就丢数据。检查点(Checkpoint)就是用来推进"哪些 redo 可以安全回收"的水位线。

checkpoint_lsn 的精确定义

checkpoint_lsn = min(所有未刷盘脏页的 oldest_modification_lsn)

含义:取所有脏页"最早一次修改"的 LSN 中的最小值。这个值之前的所有 redo 对应的修改,必然都已经在数据页上落盘了——因为如果还有某个脏页的修改 LSN 比它小,它就不会是 min 了。

模糊检查点(Fuzzy Checkpoint)

传统检查点("sharp checkpoint")的做法是"停下所有写操作 → 把所有脏页刷盘 → 设置 checkpoint_lsn",这会造成秒级以上停顿,生产不可接受。InnoDB 用模糊检查点:不停机,后台线程持续地按 LSN 顺序刷脏页,每刷掉一批就把 checkpoint_lsn 往前推一点——抖动被均摊掉了。

检查点推进越快,恢复时要扫描的 redo 越少,恢复越快;但代价是脏页刷盘压力大。innodb_io_capacity / innodb_max_dirty_pages_pct 等参数就是在这两端做权衡。

05三阶段恢复(核心)

重启时,InnoDB 不对外开放连接(直到恢复完成),依次执行三个阶段。这套流程来自 ARIES 算法,是事务型数据库的工业标准。

① Analysis 分析阶段

从 checkpoint_lsn 开始顺序扫描 redo 到末尾,弄清楚"崩溃瞬间到底有哪些事务在活跃"。InnoDB 在内存里维护两张表:

  • 活跃事务表(Trx Table):trx_id → 状态(active/preparing/committed)。每遇到一条 redo 记录,就把对应事务记到表里;遇到 COMMIT 记录就从表里移除。扫完后仍留在表里的,就是崩溃时未提交的事务,要在 Undo 阶段回滚。
  • 脏页表(Dirty Page Table):page_id → recovery_lsn(该页的最早修改 LSN)。用于 Redo 阶段判断从哪条 redo 开始重放,跳过已应用的修改。

② Redo 前滚阶段(对所有事务一视同仁)

从重做起点开始,逐条读 redo 记录、定位目标页(若不在内存就从磁盘读入)、比对 page_lsn:

  • page_lsn ≥ redo_lsn:此修改早已应用,跳过(这就是幂等性的体现)。
  • page_lsn < redo_lsn:重放这条 redo,把修改应用到页上,更新 page_lsn。
关键:Redo 阶段对已提交、未提交事务一视同仁,全部重做。直觉上你可能觉得未提交的不该重做——但这是必须的,因为只有先恢复到"崩溃瞬间的完整状态",Undo 阶段才能拿到完整的 undo 链去做撤销。

③ Undo 后滚阶段

对 Analysis 阶段识别出的每个未提交事务,沿其 undo 链反向遍历,对每条 undo 记录执行反向操作:

  • INSERT → DELETE
  • DELETE-mark → 恢复 mark 标志
  • UPDATE → 用 undo 里的旧值覆盖

这里有一个精妙细节:反向操作本身也会产生新的 redo,叫 CLR(Compensation Log Record,补偿日志记录)。这保证了 Undo 过程本身也是幂等可恢复的——如果 Undo 阶段进行到一半再次崩溃,下次重启时这部分 CLR 也会被重放,不会出现"回滚了一半"的状态。

MySQL 8.0 起,Undo 阶段可以在数据库对外开放连接之后异步进行,后台 purge 线程协助完成。这能让恢复后的开放服务时间更短。

06双写:撕裂页防线

上面整套 redo/undo 机制的前提是页本身完整。但磁盘原子写单位通常只有 4KB,而 InnoDB 页是 16KB——一次 16KB 写在 OS 看来是 4 次独立的 4KB 写,中间崩溃就会留下"撕裂页":前 8KB 是新数据、后 8KB 是旧数据,校验和失败、B+ 树结构错乱。redo 是物理逻辑日志,基于撕裂页无法重放,所以必须有双写(Doublewrite)这道前置防线。

为什么"写两次"不是两倍 I/O

双写缓冲区默认 2MB,正好对应 128 个 16KB 页。脏页刷盘时:

  1. 把这批最多 128 个脏页,作为一个 2MB 块顺序写入磁盘上的双写区(系统表空间内或 8.0.20+ 独立的 .dblwr 文件)并 fsync;
  2. 再把这些页写到 .ibd 里它们各自真实的、离散随机的位置。

真正昂贵的是离散随机写(128 次寻址);多出来的那次是一次顺序 2MB 写,被均摊到 128 个页中,实际开销通常只增加 5%~15%,远低于直觉上的"翻倍"。

恢复时怎么用双写区

启动时 InnoDB 扫描每个数据页的校验和(File Trailer),若发现某页损坏,就从双写区找这个页的完整副本覆盖回去,然后再让 redo 接力补齐后续修改。两阶段崩溃都有解:

  • 崩溃在第一阶段(双写区还在写):.ibd 没动,数据页完整。直接走 redo 即可。
  • 崩溃在第二阶段(写 .ibd 中途):某页可能撕裂,但双写区有完整副本,覆盖修复后走 redo。
如果你的存储已经保证 16KB 原子写(某些企业级 SSD、ZFS、特定 NVMe atomic write),双写就是纯浪费,可以 innodb_doublewrite=OFF 关掉。

07四种崩溃时机

恢复机制要对各种崩溃时机都正确。下面四个场景覆盖了主要可能性。

场景崩溃时机恢复路径
A事务执行中、未提交Redo 全部重做 → Undo 回滚未提交 → 痕迹清除
B已提交,redo 已落盘,数据页未落盘Redo 重放补齐 → 该事务无需 Undo
C脏页正在落盘途中(撕裂风险)双写区找完整副本覆盖 → Redo 接力 → 必要时 Undo
Dredo 写到一半、未落盘未刷的 redo 丢失 → 但这些事务必然未返回 commit → 对用户相当于没发生
场景 D 是 ACID 中 D(持久性)的精确边界:commit 返回成功的事务一定不丢,反过来未返回成功的事务不保证保留。这就是 innodb_flush_log_at_trx_commit=1(默认)的语义。

08ACID 完整闭环

把前面所有机制串起来,看每个 ACID 特性是由谁保障的:

  • A 原子性 由 undo log + Undo 阶段 保障:任何修改前都生成可逆的 undo 记录,事务要么走到 COMMIT 全部生效,要么被 ROLLBACK 或 Undo 阶段全部撤销。
  • D 持久性 由 redo log + Redo 阶段 保障:提交时 redo 已落盘,即使数据页还在内存,崩溃后也能用 redo 重做。
  • I 隔离性 由 锁(Next-Key Lock 等)+ MVCC + ReadView 保障;MVCC 读旧版本依赖 undo,所以隔离性也间接依赖 undo。
  • C 一致性 是结果:A、I、D 都成立、加上 SQL 层的约束检查(唯一、外键、check)、加上双写保证页结构不损坏 + MTR 保证物理操作原子,共同维持。
一句话:undo 管"撤销",redo 管"重做",锁/MVCC 管"并发",双写管"页完整",检查点管"日志回收"——五者合一,构成 InnoDB 的崩溃安全闭环。

09实战与调优

恢复速度的影响因素

恢复时间 ≈ 要扫描的 redo 量 + 要重放的 redo 数 + Undo 量。优化恢复速度的核心是缩短崩溃时 checkpoint_lsn 与 redo_lsn 之间的距离(即"未落盘脏页对应的 redo 总量")。

参数作用
innodb_redo_log_capacity(8.0.30+)redo 总容量。太小频繁强制检查点;太大恢复慢
innodb_io_capacity / innodb_io_capacity_max后台刷脏页的速度上限
innodb_max_dirty_pages_pct脏页占 Buffer Pool 的最高比例,触发刷盘
innodb_flush_log_at_trx_commit1(默认,最安全)/ 2(OS 崩溃可能丢 1s)/ 0(MySQL 崩溃可能丢 1s)
innodb_doublewrite是否启用双写,在保证原子写的存储上可关闭

innodb_force_recovery:紧急启动模式

当正常恢复也失败时(比如 redo 损坏、系统表空间损坏),可以用这个参数跳过部分恢复步骤强制启动:

值跳过的步骤
0(默认)正常恢复
1忽略损坏页,继续启动
2不启动后台 master 和 purge 线程
3不做事务回滚(跳过 Undo)
4不计算统计信息
5不扫描 undo log(可能丢未提交事务)
6不重做 redo log(最危险,可能丢已提交事务)
使用守则:从 1 开始递增尝试,启动成功后立即用 mysqldump 导出数据,然后重建实例。设为 ≥4 后数据库会进入只读模式,且数据可能已经不一致——它是抢救数据的最后手段,不是日常运行模式。

查看恢复过程

崩溃恢复的过程会输出到 MySQL 错误日志,关键字搜:InnoDB: Starting crash recovery、Apply batch completed、Doublewrite buffer has been recovered、Rolling back trx with id …。一次"健康"的恢复完整日志通常在几秒到几十秒之间;若你看到恢复跑了几分钟甚至几小时,要检查 redo 容量是否设得过大、或脏页刷盘速度跟不上写入。
