InnoDB Undo 实现原理

承接《存储结构》《Buffer Pool 内存侧链路》与《故障恢复原理》，这份专门讲 Undo：一条记录被 UPDATE 之后，InnoDB 为什么还能回滚？普通 SELECT 为什么能读到旧版本？DELETE 为什么不是立刻物理删除？崩溃恢复时，未提交事务又是怎么被撤销的？
适用版本：MySQL 5.7 / 8.0 / 8.4，InnoDB 引擎。
核心论断：Undo 不是“反向 SQL”，而是 InnoDB 为读写事务维护的旧版本链路。它同时服务 事务回滚、MVCC 一致性读 和 崩溃恢复 Undo 阶段。Redo 负责“把页修改重做出来”，Undo 负责“知道如何撤销修改、如何构造旧版本”。

01总览：Undo 要解决什么问题

数据库里的写操作并不是“改完就永远只剩新值”。只要事务还没有最终提交，InnoDB 就必须保留“能撤回”的能力；即使事务提交了，只要还有旧的 ReadView 可能读取历史快照，旧版本也不能立刻丢。

场景问题Undo 的作用
ROLLBACK事务做了一半失败，怎么撤销？保存旧值或反向动作，按相反顺序恢复。
MVCC别人正在修改，我的普通 SELECT 怎么读旧快照？沿版本链找到对当前 ReadView 可见的旧版本。
崩溃恢复崩溃前未提交事务的页可能已经落盘，重启后怎么办？Redo 先前滚到崩溃瞬间，再用 Undo 后滚未提交事务。
DELETE 清理删除的记录为什么不马上物理删除？先 delete-mark，等没有快照需要旧版本后由 purge 清理。
关键理解：Undo 不是为了“替代 redo”。Undo 自己也是页，也需要 redo 保护。否则崩溃时如果 undo 丢了，数据库就无法撤销未提交事务。

02隐藏列：版本链的入口

InnoDB 在聚簇索引记录中维护几个隐藏字段，其中最关键的是 DB_TRX_ID 和 DB_ROLL_PTR。

隐藏列长度含义
DB_TRX_ID6 字节最近一次插入或更新这条记录的事务 ID。DELETE 在内部也被看作一次带 delete-mark 的更新。
DB_ROLL_PTR7 字节回滚指针，指向 rollback segment 中的一条 undo log record。
DB_ROW_ID6 字节当表没有合适主键时，InnoDB 自动生成的行 ID。
聚簇索引记录当前版本
┌────────────────────────────────────┐
│ id = 10                            │
│ name = 'Tom'                       │
│ age = 21                           │
│ DB_TRX_ID = 200                    │
│ DB_ROLL_PTR = 指向 undo record     │
└────────────────────────────────────┘

DB_TRX_ID 解决“这个版本是谁改出来的”；DB_ROLL_PTR 解决“如果这个版本对我不可见，去哪里找上一个版本”。这两个字段把一条记录串成一条从新到旧的版本链。

03物理组织：Undo 存在哪里

Undo 不是“某个简单日志文件”，而是位于 rollback segment 中的一组 undo log record。rollback segment 又位于 undo tablespace、system tablespace 或 temporary tablespace 中。

Undo Tablespace
└── Rollback Segment
    └── Undo Log Segment
        └── Undo Page
            └── Undo Log Record
层级作用
Undo Tablespace存放 undo 的表空间。MySQL 8.0 初始化实例时默认创建两个默认 undo tablespace。
Rollback Segment回滚段，内部管理多个 undo slot，给事务分配 undo log。
Undo Log Segment一个 undo log segment 位于 rollback segment 中，可被事务使用并在释放后复用。
Undo Log Record某次聚簇索引记录修改的撤销信息，也是 MVCC 旧版本来源。

临时表的 undo 比较特殊：用于修改用户临时表的事务，只需要在 mysqld 运行期间支持回滚，不需要崩溃恢复，所以位于 temporary tablespace 的 undo 不需要 redo 保护。

04三类写入：INSERT / UPDATE / DELETE

4.1 INSERT undo：只服务回滚

INSERT 产生的是新记录。对其他事务来说，未提交的新记录本来就不可见；如果事务提交，它就是合法的新版本。因此 INSERT undo 主要用于本事务回滚。

BEGIN;
INSERT INTO user(id, name, age) VALUES(1, 'Tom', 20);
ROLLBACK;

回滚动作：删除这条由本事务插入的新记录
结论：INSERT undo 在事务提交后通常可以较快释放，因为它不需要长期支撑一致性读。

4.2 UPDATE undo：保存被覆盖前的旧值

UPDATE 会把聚簇索引记录改成新版本，同时把旧版本写入 undo。

原记录：age = 20, trx_id = 100

UPDATE user SET age = 21 WHERE id = 1;

当前记录：age = 21, trx_id = 200, roll_ptr → undo1
undo1 ：old age = 20, old trx_id = 100, prev_roll_ptr = ...

如果当前事务要回滚，就用 undo1 把 age 改回 20。若另一个事务的 ReadView 认为 trx_id=200 不可见，也可以通过 undo1 重建旧版本。

4.3 DELETE undo：先标记删除，后物理清理

DELETE 在 InnoDB 内部通常不是立刻物理删除，而是把记录打上 delete-mark，并写 undo 保留删除前的版本。

DELETE FROM user WHERE id = 1;

内部效果：
1. 当前记录打 delete-mark
2. DB_TRX_ID 改为删除事务 ID
3. DB_ROLL_PTR 指向删除前版本的 undo
4. 等 purge 判断安全后再物理删除

这样做是为了保护旧快照。某个事务在 DELETE 提交前已经创建 ReadView，那么即使 DELETE 后提交，这个事务也仍可能需要看到删除前的版本。

05MVCC：一致性读如何找旧版本

普通 SELECT 在 REPEATABLE READ 和 READ COMMITTED 下通常走一致性非锁定读。它不是直接读取“最新版本”，而是读取“对当前 ReadView 可见的版本”。

读取一行记录
  ↓
读取当前版本的 DB_TRX_ID
  ↓
判断这个 trx_id 对当前 ReadView 是否可见
  ├── 可见：直接返回当前记录
  └── 不可见：沿 DB_ROLL_PTR 找 undo
          ↓
       重建上一个版本，再次判断
          ↓
       直到找到可见版本，或者确认这行不存在
隔离级别一致性读快照规则
REPEATABLE READ同一个事务内，第一次一致性读建立快照，后续一致性读复用这个快照。
READ COMMITTED每一次一致性读都建立新的快照，所以同一事务内两次 SELECT 可能看到不同已提交版本。
重点：MVCC 的“多版本”不是把一行的所有版本都堆在聚簇索引里，而是当前版本在聚簇索引页中，历史版本在 undo 中，通过 DB_ROLL_PTR 串起来。

06Rollback：如何反向执行

事务回滚时，InnoDB 按该事务的 undo 链从后往前撤销。后做的修改要先撤销，先做的修改后撤销。

BEGIN;
INSERT INTO user(id, name, age) VALUES(1, 'Tom', 20);  -- undo1
UPDATE user SET age = 21 WHERE id = 1;                 -- undo2
DELETE FROM user WHERE id = 1;                         -- undo3
ROLLBACK;

回滚顺序：
undo3：撤销 DELETE，取消 delete-mark
undo2：撤销 UPDATE，把 age 改回 20
undo1：撤销 INSERT，删除新插入记录

回滚不是“把日志删掉”，而是根据 undo 记录执行反向页修改。既然页又被修改了，这些反向修改本身也需要 redo 保护。

07Purge：历史版本什么时候清理

事务提交后，undo 不一定能马上删除。INSERT undo 通常可以较快释放；UPDATE / DELETE undo 还可能被其他事务的一致性读使用，必须等所有可能需要它的 ReadView 都结束。

类型提交后是否能立刻释放原因
INSERT undo通常可以较快释放只用于本事务回滚。
UPDATE undo不能立刻释放一致性读可能需要它重建旧版本。
DELETE undo不能立刻释放旧事务可能仍要看到删除前版本，delete-mark 记录也要延后物理清理。
事务提交
  ↓
update/delete undo 进入 history list
  ↓
purge 判断是否还有 ReadView 需要旧版本
  ├── 需要：继续保留
  └── 不需要：清理 undo，物理删除 delete-mark 记录和相关二级索引旧条目
线上经验：长事务最大的危害之一，不是它一直占着 CPU，而是它可能长期持有 ReadView，让 purge 不敢清理旧版本，最终导致 history list 增长和 undo tablespace 膨胀。

08崩溃恢复：Redo 后为什么还要 Undo

崩溃恢复不是“只重放已提交事务”。InnoDB 的恢复逻辑更接近：

① Redo 前滚
   把所有已经写入 redo 的页修改恢复出来
   包括已提交事务，也包括未提交事务

② Undo 后滚
   找出崩溃时仍未提交的事务
   沿 undo 链撤销它们

为什么 Redo 阶段要先把未提交事务也恢复出来？因为崩溃前，未提交事务修改过的数据页可能已经被刷到磁盘；如果只恢复已提交事务，页的物理结构可能不完整。正确做法是先恢复到崩溃瞬间的物理状态，再从事务语义上撤销未提交部分。

崩溃前：
T1 已提交
T2 未提交，但部分脏页可能已经落盘
T3 未提交，undo 也已经被 redo 保护

恢复时：
Redo：恢复 T1/T2/T3 的页修改
Undo：回滚 T2/T3
最终：只留下 T1 的结果

09实战：长事务与 History list

排查 undo 相关问题，常看 SHOW ENGINE INNODB STATUS\G 中的 History list length。

SHOW ENGINE INNODB STATUS\G

...
History list length 123456
...

它可以理解为：已经提交、但 purge 还没清理完的历史 undo 链表长度。数值持续升高，通常说明 purge 落后。

现象可能原因处理方向
History list 持续升高有长事务或长查询持有 ReadView查 information_schema.innodb_trx，尽快结束异常事务。
大批量 UPDATE / DELETE 后 undo 暴涨单事务修改太多分批提交，降低单次事务规模。
备份期间 undo 增长mysqldump --single-transaction 长时间持有快照避开高峰，缩短备份时间，评估物理备份。
purge 长期跟不上写入太猛或 purge 资源不足观察 innodb_purge_threads、I/O 压力和写入节奏。
SELECT
  trx_id,
  trx_state,
  trx_started,
  trx_mysql_thread_id,
  trx_query
FROM information_schema.innodb_trx
ORDER BY trx_started;

重点看 trx_started 很早、trx_state 长时间 RUNNING、trx_query 为空或不动的事务。它们可能就是阻塞 purge 的源头。

10总结：Undo / Redo / Binlog 分工

日志所属层核心作用主要解决的问题
Undo LogInnoDB保存旧版本，支持回滚、MVCC、崩溃恢复后滚原子性、一致性读
Redo LogInnoDB记录页修改，崩溃后重放持久性、页级恢复
BinlogServer 层记录逻辑变更事件复制、时间点恢复
undo：我怎么撤回这次修改 / 怎么构造旧版本
redo：我怎么重做这次页修改
binlog：我对外声明执行过什么逻辑变更
最终理解：Undo 的本质是 InnoDB 的“版本管理基础设施”。它不是只为 ROLLBACK 服务，而是贯穿写入、读取、回滚、purge、崩溃恢复整条链路。没有 Undo，InnoDB 就无法同时实现高并发一致性读和事务原子性。
