01

总览：索引的本质

前五篇讲了数据怎么存、怎么用、怎么恢复、怎么并发控制。但有一个贯穿始终的问题没正面回答：给定一个查询条件，InnoDB 怎么在百万、千万行里"瞬间"找到目标行？答案就是索引。

索引的本质一句话：用额外的、按 key 有序组织的数据结构，把"全表扫描的 O(n)"变成"树查找的 O(log n)"。它是空间换时间——多存一份有序结构，换查找时少扫无数行。

本篇主线

理解索引，要回答四个递进的问题：① 用什么数据结构？为什么？（第 2–4 章，B+ 树原理）② 索引和数据怎么摆放？（第 5–6 章，聚簇/二级）③ 一次查询具体怎么走、怎么优化？（第 7–10 章，回表/覆盖/最左前缀/ICP）④ 怎么看它走得对不对、怎么避免走歪？（第 11–13 章，EXPLAIN 与失效）。原理优先，每个结论都尽量讲清"为什么"。

02

原理推导：为什么偏偏是 B+ 树

这是本篇最重要的原理。InnoDB 用 B+ 树做索引不是随便选的，而是被磁盘的物理特性一步步逼出来的。我们顺着候选方案淘汰，看 B+ 树是怎么胜出的。

前提：磁盘 I/O 是瓶颈，要"少读盘"

内存访问以纳秒计，磁盘（即便 SSD）以微秒到毫秒计，差几个数量级。数据库的数据量远大于内存，索引结构主要躺在磁盘上。所以衡量一个索引结构好不好，核心指标是一次查找要读几次磁盘（几个磁盘页），而不是比较了多少次。磁盘按"页"（InnoDB 16KB）为单位读取——读 1 字节和读 16KB 的代价几乎一样。这个事实决定了一切。

候选淘汰赛

方案致命问题结论
哈希表等值 O(1) 很快，但不支持范围查询、排序（哈希打乱了顺序）；WHERE age > 18 退化为全扫范围场景出局
二叉搜索树 / 红黑树每个节点只存 1 个 key、2 个分叉，树又高又瘦。千万数据树高约 24 层 → 每次查找最坏读 24 个磁盘页树太高，读盘太多
B 树（B-Tree）一个节点存多个 key + 多个分叉，矮胖、读盘少。但每个节点都存数据，导致非叶子节点能放的 key 变少、扇出降低；且范围查询要中序回溯，跨节点跳来跳去接近了，但还能更好
B+ 树—— 见下方为什么它最优胜出 ✓
B+ 树为什么最优（三个理由）

① 非叶子节点不存数据，只存 key + 指针 → 同样大小的页能塞下更多 key → 扇出更大、树更矮 → 读盘更少。这是 B+ 树相对 B 树最关键的改进。
② 所有数据都在叶子节点，且叶子用双向链表相连 → 范围查询（BETWEEN、>、ORDER BY）只需定位起点，然后顺着链表横扫，不必在树里反复回溯。
③ 每次查找路径长度都一样（都走到叶子） → 查询性能稳定可预测，没有 B 树那种"有的 key 在上层很快、有的在底层很慢"的抖动。

03

B+ 树的结构详解

上一章讲清了"为什么"，这一章讲清"长什么样"。InnoDB 的 B+ 树有几个必须记住的结构特征——它们直接决定后面回表、覆盖、范围查询的行为。

三类节点

  • 根节点（root）—— 树的入口，常驻内存。每次查找都从这里出发。
  • 非叶子节点（internal）—— 只存索引键 + 指向下层节点的指针，不存任何行数据。它们是"路标"，负责把查找逐层导航到正确的叶子。
  • 叶子节点（leaf）—— 存放真正的数据（聚簇索引叶子存整行；二级索引叶子存索引列 + 主键值，见第 5 章）。所有叶子在同一层，且通过双向链表按 key 顺序首尾相连。
四个必记特征

① 数据只在叶子；非叶子节点的 key 是"分界路标"，会在叶子里重复出现。
② 叶子有序且双向链表相连 → 天然支持范围扫描和 ORDER BY 免排序。
③ 一个节点 = 一个磁盘页（16KB） → 节点内有多个 key，靠页内二分查找定位。
④ 树高极矮（通常 2~4 层），所以任意 key 的查找代价几乎恒定，下一章算给你看为什么这么矮。

04

扇出与树高：3 层为何能存千万行

"千万行只需 3~4 次 I/O"听起来像魔法，其实是一道乘法题。算清楚它，你就彻底理解了 B+ 树为什么快，也能解释很多调优现象。

关键概念：扇出（fan-out）

扇出 = 一个节点能指向多少个子节点，约等于"一个 16KB 页里能放下多少个 (key + 指针) 条目"。扇出越大，每下降一层覆盖的数据就翻越多倍，树就越矮。

一道乘法题（估算）

// 非叶子节点：放 key + 指针
页大小 = 16KB = 16384 字节
一个条目 ≈ key(8B, 如 bigint) + 指针(6B) = 14 字节
单个非叶子节点扇出 ≈ 16384 / 14 ≈ 1170 个子节点

// 叶子节点：放整行（假设一行 1KB）
单个叶子能放行数 ≈ 16384 / 1024 ≈ 16 行

// 各层级承载的数据量
1 层(只有根，是叶子)            : 16 行
2 层(根 + 叶子)    : 1170 × 16        ≈ 1.8 万行
3 层(根 + 内部 + 叶子): 1170 × 1170 × 16 ≈ 2190 万行
4 层                : 1170³ × 16       ≈ 256 亿行
由此能解释的调优现象

① 为什么主键不宜过长（如别用长字符串/UUID 当主键）。主键越长，非叶子节点每个条目越大、扇出越小、树越高、I/O 越多；且二级索引叶子都存主键值（第 5 章），主键长还会让所有二级索引一起变大。
② 为什么根节点和上层节点几乎总在内存里。就那么几个页，反复访问，Buffer Pool（第二篇）会把它们牢牢缓存，实际 I/O 往往只发生在最底层叶子。
③ 为什么单行过大有害。行越大，单叶子放的行越少，叶子层页数越多、树可能变高。这也是为什么大字段（TEXT/BLOB）常溢出到独立页存储。

05

聚簇索引 vs 二级索引

InnoDB 的表数据本身就是按一棵 B+ 树组织的——表里所有的索引都是 B+ 树，但叶子里放什么不同。分清这两类索引，是理解回表、覆盖、最左前缀的前提。

聚簇索引 Clustered Index（主键索引）

每张 InnoDB 表有且仅有一棵聚簇索引，它的叶子节点直接存放整行数据。也就是说——聚簇索引就是表本身，数据按主键顺序物理存放在它的叶子里。主键选取规则：有主键用主键；没主键找第一个非空唯一索引；都没有则用隐藏的 DB_ROW_ID（第四篇提过）。

二级索引 Secondary Index（辅助索引 / 非聚簇索引）

除主键外的每个索引都是二级索引，它也是一棵 B+ 树，但叶子节点存的是索引列的值 + 该行的主键值，不存整行数据。

一句话区分

聚簇索引的叶子 = 数据本身；二级索引的叶子 = 一张"索引列 → 主键"的对照表。走聚簇索引找到叶子就拿到全部列；走二级索引找到叶子只拿到主键，缺的列必须拿主键再去聚簇索引里捞一次。这个"再捞一次"就是回表。

06

为什么二级索引存主键而非行指针

这是一个高频原理追问：二级索引叶子为什么存"主键值"，而不直接存一个指向数据行物理位置的指针（地址）？用指针不是更快、能一步到位吗？答案揭示了 InnoDB 的一个核心设计权衡。

假如存物理指针会怎样

聚簇索引的叶子里存的是整行数据，而 InnoDB 的行会移动：页分裂、页合并、行更新导致变长放不下而迁移……每当一行的物理位置变化，所有指向它的"物理指针"都会失效。如果二级索引存的是物理地址，那么每移动一行，就要去更新所有二级索引里指向它的指针——一张表有 N 个二级索引，代价高到无法接受。

设计权衡的本质

存主键值，相当于加了一层稳定的逻辑标识：物理位置怎么变，主键不变，二级索引指向的就永远有效。代价是查询时多一次回表（拿主键再查聚簇索引），收益是写入时二级索引维护成本极低。InnoDB 选择了"写得省 + 用主键解耦"，把查找的那点开销留给"回表"——而回表又可以用覆盖索引（第 8 章）来避免。这是一个非常典型的"用一点读代价换大量写代价 + 结构稳定性"的工程权衡。

07

回表 Table Lookup

有了第 5、6 章的铺垫，回表就一目了然了。回表 = 通过二级索引拿到主键后，再用主键去聚簇索引查出完整行。它本质是走了两棵 B+ 树。

回表的代价与放大

单次回表只是多一棵树的查找，看似不贵。但当二级索引命中很多行时，每一行都要回表一次——例如 WHERE name LIKE 'B%' 命中 1 万行，就是 1 万次随机回表，I/O 被放大上万倍。这就是为什么优化器有时宁可全表扫描也不走二级索引：当预估命中行数占比较大时，大量随机回表反而比顺序全扫更慢。理解这一点，就能理解很多"明明有索引却没走"的现象（详见第 11 章 EXPLAIN）。

承上启下

回表的代价能不能干脆消掉？能——如果二级索引的叶子里已经包含了查询需要的所有列，就根本不用回聚簇索引。这就是下一章的覆盖索引。

08

覆盖索引 Covering Index

覆盖索引（covering index）不是一种特殊的索引类型，而是一种"查询恰好被索引覆盖"的状态：当一个查询所需的全部列都能从某个二级索引的叶子里直接取到时，就无需回表——这个二级索引"覆盖"了该查询。

对比：回表 vs 覆盖

设 name 上有二级索引（叶子含 name + 主键 id）：

查询需要的列二级索引能否提供结果
SELECT * WHERE name='Bob'所有列（含 age、city）不能（叶子只有 name、id）回表
SELECT id WHERE name='Bob'id能（叶子就有 id）覆盖，免回表
SELECT id,name WHERE name='Bob'id、name能（叶子两者都有）覆盖，免回表
价值与识别

价值：对命中行多的查询，覆盖索引把"每行一次随机回表"全部省掉，性能提升常常是数量级的。
识别：EXPLAIN 的 Extra 列出现 Using index，就代表用上了覆盖索引（注意别和 Using index condition 即 ICP 混淆，见第 10 章）。
实践：把高频查询常用的返回列也放进联合索引（哪怕它们不参与 WHERE），就能制造覆盖索引——这是"用索引宽度换免回表"的经典手法，但要权衡索引变大带来的写入与存储成本。

09

联合索引与最左前缀

联合索引（复合索引）是把多个列组合成一个索引，如 INDEX idx(a, b, c)。它最重要、也最容易踩坑的规则就是最左前缀原则（leftmost prefix）。要真正理解它，必须先看清联合索引是怎么排序的——这才是原理。

原理：联合索引按"列顺序"逐级排序

idx(a, b, c) 的 B+ 树叶子，是先按 a 排序；a 相同的再按 b 排序；a、b 都相同的再按 c 排序。就像电话簿按"姓 → 名 → 中间名"排。关键推论：只有当左边的列是"确定值"时，右边的列才在局部有序，才能用于索引查找。

哪些查询能用 idx(a,b,c)

查询条件能用到的索引列说明
a=1a用最左列，有效
a=1 AND b=10a, b连续从左用，有效
a=1 AND b=10 AND c=100a, b, c全用上，最佳
b=10（跳过 a）用不上缺最左列 a，c 整体无序，索引失效
a=1 AND c=100（跳过 b）仅 aa 有效定位后，c 在该段无序 → c 用不上索引（但 ICP 可优化，见第 10 章）
a>1 AND b=10（a 用范围）仅 aa 是范围，其后 b 在跨多个 a 段里无序 → b 用不上
两条铁律

① 必须从最左列开始、且中间不能断。跳过 a 直接查 b，或跳过 b 用 c，被跳过列右边的部分都无法走索引——因为它们在全局是无序的。
② 范围查询（> < LIKE BETWEEN）会"中止"后续列的索引能力。一旦某列用了范围，它本身能用索引，但它右边的列就无法再用索引定位了（因为范围跨越了多个段，段内才有序）。
实践推论：设计联合索引时，把等值查询、区分度高的列放左边，把范围查询的列放右边，能最大化索引利用。

10

索引下推 ICP（Index Condition Pushdown）

ICP 是 MySQL 5.6 引入的优化，专门减少回表次数。它和最左前缀的"失效"场景密切相关，理解它能解释很多"看似用不上索引的列其实帮了忙"的现象。

没有 ICP 时的浪费

考虑 idx(a, b) 和查询 WHERE a='X' AND b LIKE '%abc'。b 是后缀模糊，按第 9 章规则不能用于索引定位。没有 ICP 时：存储引擎只用 a='X' 在索引里筛出一批记录，把它们全部回表取出整行，再交给 Server 层逐行判断 b LIKE '%abc'。大量不满足 b 条件的行被白白回表了。

有 ICP：把 b 的判断"下推"到索引层

ICP 让存储引擎在回表之前，就用索引里现成的 b 值先过滤一遍——b 虽然不能用于"定位"，但它的值就在索引叶子里，拿来"过滤"是免费的。只有 a、b 都满足的记录才回表。

要点

① 触发条件：查询用到二级索引，且 WHERE 里有能用该索引列、但不能用于定位的条件（如后缀 LIKE、跳列的等值）。
② 识别：EXPLAIN 的 Extra 显示 Using index condition（注意与覆盖索引的 Using index 区分——前者下推过滤但仍可能回表，后者根本不回表）。
③ 本质：ICP 不改变"哪些列能定位"，它改变的是"在哪一层过滤"——把过滤从昂贵的回表之后，提前到便宜的索引扫描之时。这是第 9 章"跳列的右侧条件用不上索引"的一个重要补救。

11

执行计划 EXPLAIN 详解

前面讲的都是"索引该怎么走"，EXPLAIN 则告诉你优化器实际打算怎么走。在 SQL 前加 EXPLAIN，不真正执行，只返回执行计划。读懂它是索引调优的基本功。

最该看的几列

列含义重点关注
type访问类型（性能等级）最重要。见下方梯子，越靠左越好
key实际使用的索引NULL = 没走索引，需警惕
key_len使用的索引长度（字节）判断联合索引用到了几列的关键
rows预估要扫描的行数越小越好；与实际差太大说明统计信息过期
Extra额外信息Using index=覆盖；Using index condition=ICP；Using filesort/Using temporary=要警惕
possible_keys可能可用的索引与 key 对比，看优化器为何没选某索引

type 性能梯子（从好到差）

用 key_len 反推用了几列

key_len 是个常被忽略但极有用的列。联合索引 idx(a,b,c)，若 a、b、c 都是 int NOT NULL（各 4 字节），那么 key_len=4 说明只用到 a，=8 说明用到 a、b，=12 说明三列全用。当你怀疑"最左前缀没生效"时，看 key_len 比看 key 更能确认到底吃进了几列。（可空列 +1 字节存 NULL 标记，变长列还要算长度前缀，计算时需留意。）

12

索引失效的常见场景

"明明建了索引，怎么没走？"几乎都能归到下面几类。它们的共同原理是：破坏了索引列的有序性，或让优化器无法利用这种有序性。

失效场景例子原理
对索引列做运算/函数WHERE YEAR(t)=2024
WHERE id+1=5		索引存的是原始值的有序结构；列被函数包裹后，有序性对函数结果不再成立 → 只能全扫
隐式类型转换phone 是 varchar，却 WHERE phone=138...（传数字）MySQL 把列转成数字再比，等价于对列做了函数 → 失效
前导通配 LIKEWHERE name LIKE '%bob'B+ 树按前缀有序，不知道开头是什么就无法定位 → 失效（'bob%' 则可用）
违反最左前缀idx(a,b)，WHERE b=1见第 9 章，缺最左列 → 失效
OR 连接非索引列WHERE a=1 OR d=2（d 无索引）OR 要求两边都能用索引，否则退化全扫（可用 UNION 改写）
区分度太低 / 命中过多WHERE gender='M'优化器估算走索引要回表的行数太多，主动放弃索引改全扫（这是优化器的正确选择，非 bug）
两个澄清

① "索引失效"有两种性质，别混。前四类是规则上根本用不了（有序性被破坏）；后两类是优化器算了一笔账后主动不用（用了反而更慢）。后者是优化器在帮你，不要硬逼它走索引。
② 函数失效在 8.0 有解。MySQL 8.0 支持函数索引，可对 YEAR(t) 这类表达式直接建索引；或改写查询为范围条件（如 t >= '2024-01-01' AND t < '2025-01-01'）让原始列索引重新生效。

13

索引设计原则

把前面的原理落到实践，索引设计有几条经得起推敲的准则：

  • 选区分度高的列建索引。区分度（cardinality / 总行数）越接近 1 越好。性别、状态这类只有几个取值的列单独建索引意义不大（命中太多、还要回表，优化器多半不走）。
  • 联合索引优于多个单列索引。一个 idx(a,b,c) 能服务 a、(a,b)、(a,b,c) 三种前缀查询，比建三个单列索引更省、更易命中覆盖与最左前缀。MySQL 对多个单列索引一般也只能用其一（或 index_merge，效率有限）。
  • 把等值列放左、范围列放右。承接第 9 章铁律：范围列会中止其右侧列的索引能力，所以让范围列尽量靠右。
  • 用覆盖索引消除高频查询的回表。把高频查询的返回列纳入联合索引，制造 Using index。
  • 前缀索引控制长字符串索引的体积。对很长的 varchar，可只索引前 N 个字符 INDEX(col(20))，在区分度和体积间权衡（但前缀索引无法用于覆盖和 ORDER BY）。
  • 主键要短、单调递增。短 → 扇出大、二级索引小（第 4、6 章）；单调递增（如自增 id）→ 插入总在 B+ 树右端追加，避免随机插入引发的页分裂。这也是不推荐用随机 UUID 当主键的核心原因。
  • 索引不是越多越好。每个索引都是一棵要维护的 B+ 树，写操作要同步更新所有相关索引，索引越多写越慢、占空间越大。只为真正高频的查询建索引。
页分裂为什么要避免

当向 B+ 树叶子的中间插入数据、而该页已满时，InnoDB 必须分裂页：新建一个页，把原页一半数据挪过去，再调整链表和父节点指针——这是一次昂贵的写放大，还会造成页空间利用率下降（碎片）。自增主键让插入永远发生在最右页的尾部，几乎不触发中间分裂；而随机主键（UUID）会到处插入，频繁分裂。这就是"主键单调递增"这条准则背后的原理，它和第 4 章"主键要短"共同决定了主键的选型。

14

自适应哈希索引与 Change Buffer

最后补两个与索引紧密相关、且呼应第二篇 Buffer Pool 的机制——它们是 InnoDB 在 B+ 树之上做的两层"加速"和"缓冲"。

自适应哈希索引 AHI（Adaptive Hash Index）

B+ 树等值查找要走 3~4 层。如果某些热点 key 被反复等值查询，每次都走树就有点亏。InnoDB 会自动监测访问模式，对频繁访问的索引页在内存里建一个哈希表，把"热点 key → 页内位置"直接映射，下次等值查找一步到位 O(1)，跳过逐层下降。

  • 全自动：由 InnoDB 根据访问频率自行决定建不建，无需也无法手动指定给哪个索引建。
  • 只服务等值查询，对范围查询无效（哈希无序）。
  • 是 B+ 树的"缓存层"，不替代 B+ 树；可通过 innodb_adaptive_hash_index 开关。某些写密集、热点分散的场景关掉它反而更好（维护哈希也有成本）。

Change Buffer（变更缓冲）

呼应第二篇：当修改一条数据时，聚簇索引的页通常已在 Buffer Pool（因为要先定位行），但二级索引的页不一定在内存——若每次改二级索引都先把对应页从磁盘读进来，会产生大量随机读 I/O。

Change Buffer 的优化：对不在内存中的二级索引页的修改（insert/delete/update），先记录在 Change Buffer 里，不立刻读盘；等将来那个页因别的原因被读进内存时，再把缓冲的修改"合并（merge）"进去。多次对同一页的修改可以攒着一次合并，把随机读摊薄。

为什么唯一索引用不了 Change Buffer

唯一索引插入时必须当场判断"这个值是不是已存在"，而要判断就必须把目标页读进内存检查——既然都要读页了，缓冲就没意义了。所以 Change Buffer 只对普通（非唯一）二级索引生效。这也是"唯一约束不是免费的"的一个体现：它换来了正确性，但放弃了这项写优化。写密集且二级索引多的表，可以适当调大 innodb_change_buffer_max_size。

系列总结：六篇成体系

本篇把 InnoDB 的"查找性能"讲透了，也为整个系列收官。回看全貌：

篇主题回答的核心问题
①存储结构与数据链路数据在磁盘上怎么组织（页/区/段/表空间）
②Buffer Pool 内存侧数据怎么在内存里被缓存和管理
③故障恢复原理崩溃后怎么靠 redo/undo/双写恢复（持久性 D）
④事务管理与 MVCC并发"读"怎么不加锁也一致（隔离性 I·读）
⑤锁系统并发"写"怎么保证正确（隔离性 I·写）
⑥索引与执行计划怎么在海量数据里快速定位（性能）
把六篇串成一条线

一次 SELECT * WHERE name='Bob' 的旅程，恰好穿过整个系列：优化器选中 name 二级索引（⑥），走 B+ 树定位（⑥），需要的页先在 Buffer Pool 找（②）、没有才从表空间的页里读（①）；回表拿整行（⑥）；如果是当前读还要按隔离级别加锁（⑤），普通快照读则按 ReadView 选版本（④）；整个过程产生的修改由 redo/undo 保驾（③）。六个子系统各司其职，又彼此咬合——这就是 InnoDB 作为一个成熟存储引擎的完整图景。
