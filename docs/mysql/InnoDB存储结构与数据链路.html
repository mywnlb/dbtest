01总览：四级存储层次

InnoDB 把磁盘空间组织成一个自顶向下的层次。理解全部内容的前提是先记住这条归属链：

表空间 Tablespace (.ibd 文件)
  └─ 段 Segment        逻辑概念，跨区，不连续
       └─ 区 Extent     物理连续，64 个页 = 1MB
            └─ 页 Page   物理实体，16KB，I/O 与缓存的最小单位
                 └─ 记录 Record  实际数据行
层次性质大小作用
表空间物理文件按需增长一张表（独立表空间）对应一个 .ibd
段 Segment逻辑不固定把同类页归为一组（叶子段 / 非叶段）
区 Extent物理连续1MB（64 页）整块分配，保证顺序 I/O
页 Page物理实体16KB读写与缓冲池的最小单位
记录 Record数据行变长实际存储的一行数据

02物理结构 vs 逻辑结构

这是最容易混淆的地方。两者描述的是同一批页的两个视角。

物理结构（磁盘上真实存在的）

.ibd 文件就是一长串 16KB 的页，按页号 0、1、2、3… 紧挨着线性排列：

  • 物理地址公式：page N 的文件偏移 = N × 16384 字节
  • 区起始页号：Extent E 的首页号 = E × 64
  • 磁盘上你能「摸到」的只有三样东西：文件 → 页 → 页内字节。

.ibd 文件首区（Extent 0）的固定管理页

文件最开头是 顺序写死的固定四页。注意 XDES（区描述符）不是独立的页类型——它是一组 40 字节的条目，头 256MB 范围的条目直接内嵌在 page 0（FSP_HDR）里，并不单独占页。

页号类型作用
page 0FSP_HDR
（含 XDES）		表空间总入口：大小、Space ID、FREE / FREE_FRAG / FULL 区链表头、INODE 页链表；并内嵌头 256MB 的 XDES 区描述符条目
page 1IBUF_BITMAPChange Buffer 位图，记录每页是否有缓存的二级索引修改
page 2INODE段信息数组：每个段一条 Entry（192 字节），记该段的区链表头与 32 个碎片页指针
page 3SDI序列化数据字典，存表结构定义（8.0 起；5.7 无此页）
page 4 起INDEX / BLOB业务数据：聚簇索引与二级索引的 B+ 树（根 / 内部节点 / 叶子），以及超长字段的 BLOB 溢出页
之后每隔 16384 页（256MB）会再插入一组独立的 XDES + IBUF_BITMAP 管理页，因为单个 FSP_HDR 内嵌的 XDES 只能描述 256MB 范围内的区。

XDES 与 INODE 的分工

两者都和「区」打交道，但视角相反，配合起来让空间分配/回收达到 O(1)：

  • XDES Entry（区描述符，40B）——站在区的视角回答：这个区是 FREE / FREE_FRAG / FULL 哪种状态？归属哪个段？它的 64 个页哪些被占用（bitmap）？位置在 page 0 内或独立 XDES 页。
  • INODE Entry（段描述符，192B）——站在段的视角回答：这个段拥有哪些区（FREE / NOT_FULL / FULL 三条区链表）？有哪些碎片页？位置在 page 2（INODE 页）。

分配新页时：先查 INODE 找到段的 NOT_FULL 区链表拿一个区，再查该区的 XDES 看 bitmap 找空闲页置位——全程不用扫描。

undo 页在 8.0 里默认放在独立的 undo 表空间（undo_001 等），不在 t_user.ibd 内；只有老版本或特殊配置才在系统表空间 ibdata1。而 BLOB / 溢出页 确实在本表空间文件内。

逻辑结构（InnoDB 脑中的归属关系）

段在物理层面根本不存在——用十六进制工具打开 .ibd，找不到任何一段连续区域叫「叶子段」。段的本质是 page 2（INODE 页）里的一份归属清单：

page 2 (INODE 页)
  ├─ 非叶段 Entry (FSEG_ID=1)
  │    碎片页: page 4
  │    NOT_FULL 区链表: Extent 2
  │
  └─ 叶子段 Entry (FSEG_ID=2)
       碎片页: page 5,6,7
       FULL 区链表: Extent 1
       NOT_FULL 区链表: Extent 4

叠到物理文件上看，同一个段的页在磁盘里是跳着分布的：

叶子段的页（p5-7、E1、E4）被非叶段和空闲区在物理上切成了好几段，但逻辑上属于同一个段。

03段、区、页的关系

为什么要分这三层？

三层各自解决一个独立目标，无法用一个机制兼顾：

  • 页：I/O 和缓冲池的最小单位，太大浪费内存，太小管理开销大，16KB 是折衷。
  • 区：解决「物理顺序 I/O」。整块 1MB 连续分配，范围扫描时一次磁头定位读 64 页，避免随机寻道。
  • 段：解决「逻辑分组」。把叶子节点和内部节点分到不同的段，全表扫描只读叶子段的区，I/O 完全顺序，不被内部节点页打断。

段的两种核心类型

段存什么PAGE_LEVEL
叶子段 Leaf实际数据行= 0
非叶段 Non-leaf索引键 + 子页指针≥ 1

区的三种状态（链表归属）

  • FREE：完全空闲，可整区分配给某段
  • FREE_FRAG：碎片区，页可被不同段零散申请（小段优化用）
  • FULL_FRAG：碎片区页全部用光
  • 段内部还有 FREE / NOT_FULL / FULL 三个链表，管理已归该段的区

小段优化

段刚创建时不独占区，先从 FREE_FRAG 区申请单个「碎片页」，散落在公共区里。直到该段占用碎片页达到 32 个，才开始整区整区分配。

这就是为什么几行数据的小表对应的 ibd 文件可能只有 96KB。

两套独立的「顺序」

物理顺序（页号）逻辑顺序（B+ 树链表）
由谁决定分配时机：先分页号小key 大小：叶子页 PREV/NEXT 按主键串联
示例p5 紧挨 p128p70 → p65 → p300
主键自增的意义：让物理写入和逻辑顺序对齐，叶子页接近 100% 填充率。主键随机（如 UUID）会频繁页分裂，叶子物理无序，范围扫描退化为随机 I/O。

04页（Page）内部结构

一个 16KB 的索引页（叶子页）从头到尾的字节布局：

组成大小内容
File Header38B页号、类型、前后页指针（FIL_PAGE_PREV/NEXT）、LSN、校验和
Page Header56B记录数、槽数、空闲指针、PAGE_LEVEL 等元信息
Infimum + Supremum—虚拟最小/最大边界记录，简化链表头尾处理
User Records变长实际数据行，向下增长
Free Space变长空闲区，新记录从此 append
Page Directory变长稀疏槽数组，从页尾向上增长
File Trailer8B校验和，检测页是否写完整

Page Header 关键字段

字段含义
PAGE_N_DIR_SLOTS槽数组的槽数
PAGE_HEAP_TOP空闲空间起始偏移，新记录 append 起点
PAGE_N_HEAP堆中记录总数（含 Infimum/Supremum）
PAGE_FREE已删除记录构成的「垃圾链表」头，插入时优先复用
PAGE_GARBAGE可回收字节数，达阈值触发页内重组
PAGE_LAST_INSERT / DIRECTION / N_DIRECTION判断顺序插入还是随机插入，影响分裂策略
PAGE_N_RECS用户记录数
PAGE_LEVELB+ 树中的层级，0 为叶子
PAGE_BTR_SEG_LEAF / TOP指向叶子段/非叶段的段头（仅 root 页用到）

记录头里的两个关键字段

  • heap_no：记录在堆中的物理序号。Infimum=0，Supremum=1，用户记录从 2 开始按插入顺序递增——这正是「物理无序」的体现。
  • n_owned：只在每个分组的最后一条（key 最大的那条）上记录该组成员数，组内其他记录为 0。

next_record 链表与 Page Directory

所有记录通过记录头里的 next_record 串成一条严格按 key 升序的单向链表，与物理位置无关；Page Directory 则按分组稀疏地指向每组末尾记录：

next_record 指向的是记录数据的起始位置（记录头之后的第一个字节），可直接读到列数据。

05Page Directory 与分组机制

为什么需要稀疏槽？

最朴素的做法是每条记录一个槽，但插入/删除时槽数组移动开销大，且占空间。InnoDB 的做法是分组：

  • 每 4~8 条记录为一组（常量 PAGE_DIR_SLOT_MIN_N_OWNED=4，MAX=8）
  • 只给每组的最后一条（最大的）建一个槽
  • 槽数组按 key 有序，存在页尾，向页头增长

分组规则

  • Infimum 组固定 1 条；Supremum 组固定 1 条
  • 中间组保持 4–8 条，超过 8 则分裂出新槽，少于 4 则与邻组合并

示例对应关系

slot₀ → Infimum  (n_owned=1)
slot₁ → rec₃     (n_owned=4，覆盖 Infimum+rec₁+rec₂+rec₃)
slot₂ → rec₇     (n_owned=4，覆盖 rec₄+rec₅+rec₆+rec₇)
slot₃ → Supremum (n_owned=1)

槽内前缀优化（通用设计，InnoDB 未采用）

某些页内记录管理方案会在槽里冗余存放 key 的前 4 字节，用少量空间换两件事：① 比较时不用跳转和解析记录（快一个数量级）；② 截断到固定长度后槽保持定长，让二分查找在变长 key 上也能用。

InnoDB 的 Page Directory 没有做前缀优化，它走「分组 + 链表扫描」路线。前缀优化更多见于 PostgreSQL B-tree、Mass Tree 等结构。

06查找记录的数据链路

以「查找主键 key=25 的行」为例，自顶向下完整链路：

阶段 1：定位 B+ 树根

由表定义找到聚簇索引 → 拿到 root 页号（如 page 4）
物理偏移 = 页号 × 16384，从 .ibd 读入该页（若不在 Buffer Pool 则磁盘读）

阶段 2：B+ 树自上而下（跨段）

root 页(非叶段, LEVEL≥1) → 页内定位子页指针 → 下降到内部节点
→ 再定位 → … → 直到 PAGE_LEVEL=0 的叶子页

阶段 3：页内查找（核心）

① 二分槽数组：
   slot₁=rec₃ key=18 < 25 → 目标在右
   slot₂=rec₇ key=56 ≥ 25 → 落入 slot₂ 覆盖的 group2
② 取上一槽 slot₁ 指向的 rec₃ 作为扫描起点
③ 从 rec₃.next_record 沿链表线性扫描（组内最多 8 步）
④ rec₄ key=25 命中
复杂度：O(log₂槽数 + 组内步数)，组内 ≤8 步可忽略。

07插入记录的数据链路

以「插入主键 key=20 的新行」为例：

1. 自顶向下定位到应插入的叶子页（同查找阶段 1-2）
2. 页内二分找到插入位置：在 rec₃(k18) 和 rec₄(k25) 之间
3. 申请记录空间：优先复用 PAGE_FREE 垃圾链表，否则 Free Space append
4. 写入记录数据，分配新的 heap_no（按插入顺序，不是 key 顺序）
5. 修改链表指针：rec₃.next → 新记录 → rec₄
6. 更新所属 group 的 n_owned +1；组 > 8 则分裂加槽
7. 更新 Page Header：PAGE_N_RECS、PAGE_LAST_INSERT、PAGE_DIRECTION

空间不足时的两级处理

  • 先：页内重组（reorganize）——清理 PAGE_GARBAGE 标记的已删除记录，整理碎片、重排槽数组，再次尝试插入。
  • 仍不足：页分裂（page split）——向所属段申请新页（碎片页或整区）→ 搬约一半记录 → 调整叶子层 PREV/NEXT 双向链表 → 向上层（非叶段）插入新索引项，可能级联分裂。
顺序插入（自增主键）走「右分裂」优化：新页几乎不搬数据、老页保持满、填充率高；随机插入需搬一半数据、碎片多。

08删除记录的数据链路

InnoDB 的删除是两阶段的：先标记，后真正清理。

同步 阶段 1：delete mark（事务执行时）

1. 自顶向下定位到记录所在叶子页 → 页内定位目标
2. 记录头 deleted_flag 置 1（标记删除，记录仍在原位）
   ↑ 此时记录对 MVCC 仍可见（其他事务快照可能需要旧版本）
3. 旧版本信息进入 undo log，供 MVCC 和回滚使用

异步 阶段 2：purge（后台线程）

4. purge 线程判断：没有活跃事务的快照再需要旧版本
5. 真正移除：调整 next_record 跳过该记录 → 空间挂入 PAGE_FREE 垃圾链表
   → 累加 PAGE_GARBAGE → n_owned -1 → 组 < 4 则与相邻组合并删槽
6. 清理对应 undo log
7. 页空 → 页挂回 FREE 链表；区空 → 区还给表空间 FREE 链表
标记而非立即物理删除，是因为 MVCC 下其他事务的一致性读可能还需要这条记录的旧版本。

09修改记录的数据链路

InnoDB 的更新分两种情况，路径差异很大。

情况 A：原地更新（in-place update）

满足条件：未修改主键 且 更新后记录长度不变或变小。

1. 定位记录所在叶子页 → 页内定位目标
2. 直接在原位修改列值（覆盖写）
3. 旧值写入 undo log（供 MVCC / 回滚）
4. 更新隐藏列：DB_TRX_ID、DB_ROLL_PTR

代价最小，不动链表、不动槽、不分裂。

情况 B：非原地更新（delete-mark + insert）

触发条件：修改了主键 或 更新后记录变长、原位放不下。

1. 对旧记录执行 delete mark（deleted_flag=1）
2. 构造新记录，执行一次完整 insert（同插入链路）
   - 改了主键 → 新记录可能落到完全不同的叶子页
   - 可能触发页分裂
3. 旧记录由 purge 线程后续异步清理
二级索引的更新总是 delete-mark + insert（不做原地更新），因为索引列值变化即位置变化。修改主键 = 删旧行 + 插新行，代价最大，应尽量避免。

10关键结论速查

问题结论
段是物理的还是逻辑的？逻辑。归属清单存在 page 2（INODE 页），磁盘上没有连续的「段」
区是什么？物理连续的 64 个页 = 1MB，整块分配的单位，保证顺序 I/O
页内记录有序吗？物理上无序（按插入顺序堆放），靠 next_record 链表逻辑有序
为什么分叶子段/非叶段？让全表扫描只读叶子段的区，I/O 顺序不被内部节点打断
小表为什么文件很小？小段优化：段先用碎片页，占满 32 个才整区分配
删除是立即物理删除吗？否。先 delete-mark，purge 线程异步真正清理
修改主键为什么慢？等价于「删除旧行 + 插入新行」，可能跨页 + 页分裂
为什么推荐自增主键？让物理写入与逻辑顺序对齐，避免随机插入导致的页分裂与碎片
查找一条记录的链路？表空间 → root → 内部节点 → 叶子页 → 页内二分槽 + 链表扫描
槽里存什么？InnoDB：记录偏移量；分组后只给每组末尾建槽（稀疏）
归属链一句话总结：表空间 ⊃ 段（逻辑）⊃ 区（物理连续）⊃ 页 ⊃ 记录。物理顺序（页号）由分配时机决定，逻辑顺序（B+ 树链表）由 key 大小决定，两者独立。
