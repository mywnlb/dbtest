01当前实现全景

当前工程的存储部分已经不是“文件里读写若干页”的单层示例。生产组合根 StorageEngine 会装配表空间、空间分配、Buffer Pool、MTR、B+Tree、事务、双类型 Undo、Redo ring、 Flush/Checkpoint、Doublewrite、Purge 与 Crash Recovery，并把上层 DML 收敛到稳定的存储 API。

WIRED 数据访问

表级 DML → 事务/锁 → B+Tree → Record → MTR → Buffer Pool → PageStore。

WIRED 持久化

MTR 生成 Redo；刷页前等待 WAL；页面经双文件 Doublewrite 后写回表空间。

WIRED 恢复

启动时依次执行 Doublewrite 修复、Redo replay、未提交事务回滚与 Purge 恢复。

本文的“已实现”只指当前生产代码确实装配并调用的能力。目标架构图中的长期方向、仅被测试覆盖的类型， 以及尚未接线的预留入口，不会被描述成当前事实。

02表空间、段、区与页

物理文件由 PageStore 负责，空间生命周期与分配则由 DiskSpaceManager 协调 FSP、表空间注册表和访问租约。上层不会直接拿到 FileChannel； 每次针对已注册表空间的生产访问都先取得共享 operation lease，再进入 Buffer Pool/MTR 或页存储层。

页大小会改变区内页数

当前实现支持 4、8、16、32、64 KiB 页面。对于不大于 16 KiB 的页面，一个区保持 1 MiB； 32 KiB 页面使用 2 MiB 区，64 KiB 页面使用 4 MiB 区，因此大页配置仍保持 64 页/区。

PageSizeExtent 物理大小每区页数设计含义
4 KiB1 MiB256XDES 固定 256 bit，全部有效
8 KiB1 MiB128只使用 bitmap 前 128 bit
16 KiB1 MiB64默认教学配置常见取值
32 KiB2 MiB64避免一个区只剩过少页面
64 KiB4 MiB64保持相同的区内页数量级

页面分配不是直接“文件尾追加”

  1. 先用 fragment page。Segment 尚小时，优先消费最多 32 个 fragment slot，避免过早独占整个区。
  2. 再用已有 segment extent。在属于该段且仍有空闲位的区内分配页面。
  3. 需要时申请新 extent。叶段可依据顺序插入方向批量预留 1～4 个区；非叶段不使用方向提示。
  4. 空间不足时自动扩展一次。DiskSpaceManager 扩展文件并更新 FSP 元数据，然后重新尝试分配。
  5. 同一 MTR 记录物理变化。分配至少产生 FSP_PAGE_ALLOC 与 PAGE_INIT Redo，避免元数据与新页初始化脱节。

03通用页信封与固定管理页

所有持久页都先服从 PageEnvelopeLayout： 前 38 字节是通用物理头，最后 8 字节是 trailer。INDEX、UNDO、FSP 等页面只在两者之间解释自己的类型区域。 这让磁盘层可以校验页身份、校验和与 page LSN，却不需要理解 record 或事务语义。

固定位置GENERAL 表空间UNDO 表空间当前用途
Page 0FSP_HDRFSP_HDR空间配置、生命周期与页 0 内嵌 XDES
Page 1IBUF_BITMAP（预留）IBUF_BITMAP（预留）UNWIRED 当前没有 Change Buffer
Page 2INODEINODE段头、fragment slot 与 extent 所有权
Page 3SDIRSEG_HEADER v4普通空间保存元数据映像；Undo 空间保存回滚段权威状态

表空间创建时先建立文件，再初始化 Page 0 并保留 Extent 0 的页 0～3。普通空间进入 GENERAL / NORMAL，Undo 空间进入 UNDO / ACTIVE。 后续读写必须通过生命周期校验；CORRUPTED、INACTIVE、 TRUNCATING 或 DISCARDED 空间不会继续接受普通业务操作。

04Buffer Pool 与 Mini Transaction

LruBufferPool 是对外门面，内部由 Router 把 PageId 映射到 BufferPoolInstance。 生产默认配置为 1 个 instance，但路由、容量拆分、刷新和统计已经按多实例实现。

Frame 状态含义关键边界
FREE未绑定页面，可从 free list 获取不能被业务线程直接读取
LOADING一个线程正在从 PageStore 装载同页请求在锁外等待同一个 future
CLEAN内存页与数据文件一致pin 为 0 时可参与淘汰
DIRTY_PENDING页已修改，MTR 尚未提交发布不能被 flush 提前写出
DIRTY已带有效 page LSN 的脏页进入 flush list，并约束 checkpoint
FLUSHING某个快照正在刷盘仍算 dirty boundary；并发修改用版本号区分
EVICTING / STALE淘汰中或映射已失效旧句柄不能重新发布到 page hash

命中、缺页与单飞装载

Instance 内部显式维护 page hash、frame mutex、free/LRU/flush list 锁与 page latch。 推荐锁序是 page hash → frame mutex → list lock → page latch。缺页时只有首个线程创建 LOADING frame 并发起磁盘读取；其他线程拿到相同的 load future，释放内部锁后再等待， 因而不会持有全局结构锁阻塞 I/O。

LRU 使用 midpoint old/new 分区；生产链已接入线性 read-ahead、热页 dump/load。 随机 read-ahead 类型存在，但默认关闭，不能把它写成默认运行能力。

因为未提交 MTR 的回滚不恢复 Buffer Pool 内容，修改者必须遵循“先完成所有可能失败的校验，再落字节”的写法； 或者自行保留可恢复快照。这个简化点是当前实现的重要边界。

05INDEX 页与记录组织

INDEX 页采用“记录堆从低地址向高地址增长、Page Directory 从 trailer 前向低地址增长”的布局。 IndexPageLayout 在 38 字节物理头后放置 28 字节 INDEX header，再放固定的 infimum、supremum，用户记录从 offset 98 开始。

记录头、隐藏列与目录搜索

  • 每条记录使用简化的 8 字节头：flags 1 B、heapNo 2 B、nOwned 1 B、next offset 2 B、 record length 2 B。它表达链表与目录分组所需语义，但不兼容 COMPACT/DYNAMIC 原始记录头。
  • 聚簇记录尾部保存 15 字节隐藏信息：8 字节 DB_TRX_ID 与 7 字节 DB_ROLL_PTR。MVCC 由此定位创建/修改事务和前一版本。
  • Page Directory slot 指向某个分组的最后一条记录。查找先对 slot 二分，再沿 next_record 扫描该小组；插入、删除会维护 nOwned 与必要的 slot 分裂/合并。
  • 打开任何已有 INDEX 页都会执行物理结构校验；拿到 schema 的入口还会验证字段解码、key 顺序、 hidden tail 与页面级不变量。损坏不是“尽量读”，而是 fail-closed。

字段层支持 28 种 TypeId、字符集/排序规则、日期时间等编解码。 超页内阈值的业务值可放入 BLOB 链，记录只保留外部引用。业务 LOB 页面与 Undo 大载荷使用的 UNDO_PAYLOAD 页面彼此独立，不能混为同一种外部页。

06B+Tree 访问与结构修改

生产实现使用 SplitCapableBTreeIndexService， 支持任意高度、叶/非叶分段、分裂、重分布、合并与根收缩。根页 PageId 保持稳定： 根分裂时旧根内容下沉并提高根 level，根收缩时把唯一子节点内容搬回稳定根页。

点查：S-crabbing

从根到叶逐层取得子页 S latch，确认子页已稳定后释放父页；不把整条路径长期锁住。

范围：兄弟页 hand-over-hand

沿叶链移动时先固定并锁住下一页，再释放当前页，避免页在交接窗口失去保护。

写入：乐观后悲观

常规插入只在叶页持 X；溢出后重启并保留安全祖先，必要时执行级联分裂。

为什么事务锁等待要重新定位记录

当前读使用两个短 MTR：第一次定位记录后释放 page latch、cursor 和 buffer fix，再进入可能阻塞的 LockManager；拿到行锁后开启第二个短 MTR，通过 B+Tree 重新定位并校验记录。 这样既避免“持页闩等待事务锁”的锁序反转，也承认等待期间页面可能发生分裂、合并或记录移动。

分裂前会先预留最坏情况下需要的空间，避免修改一半后才发现无法分配页面。高度至少为 2 时， 悲观写路径先在根上取得 SX；如果最终确实要改根，再以 X 模式重启，不做危险的 latch upgrade。 当前实现没有 B-link sibling high-key，也没有 optimistic latch coupling，因此并发策略仍属于 结构清晰、便于教学和验证的 crabbing 方案。

07一行数据如何完成 INSERT / UPDATE / DELETE

表级写入通过 TableDmlService 编排，而不是让 SQL 层分别操作聚簇与二级索引。数据库事务是跨多次 MTR 的 Unit of Work： 每个索引通常由独立短 MTR 修改，语句失败时依靠 Undo 和 statement guard 反向补偿。

INSERT：先冻结逻辑计划，再逐索引落盘

  1. 分配写事务身份并形成当前读请求。事务 ID 可延迟到第一次写时才分配； 聚簇 key 成为后续行锁、唯一性判断与重定位的稳定逻辑标识。
  2. 统一检查全部唯一约束。先对聚簇索引和所有唯一二级索引执行 current unique check、 获取必要锁，并冻结二级索引 key/covering payload 与对应 Undo 计划，避免写到一半才发现重复键。
  3. 取得 DML/Purge 行级 guard。防止前台写与 purge 对同一逻辑行并发做不相容的物理处理。
  4. 先写聚簇记录。第一个短 MTR 同时落聚簇页变化并建立逻辑 Undo 锚点； 记录携带 DB_TRX_ID 与 DB_ROLL_PTR。
  5. 逐个写二级索引。每个 secondary 使用独立短 MTR，索引项包含二级逻辑 key、 聚簇 key 以及覆盖列 payload。全部成功后语句 guard 才提交。

UPDATE 与 DELETE 的差异

操作聚簇记录二级索引当前限制/语义
非键 UPDATE FOR UPDATE 当前读；生成 UPDATE Undo；替换页内版本并更新隐藏列 逻辑 key 不变时保留；覆盖列变化则更新 payload 外部 LOB 引用可在 patch 中保留
二级键 UPDATE 先建立旧版本链，再写新聚簇版本 先发布新 key，再把旧 key delete-mark new-before-old 避免中途失败导致索引入口完全消失
主键 UPDATE UNSUPPORTED 当前不把主键更新改写成 delete + insert 调用方会收到明确的不支持异常
DELETE FOR UPDATE + UPDATE Undo + delete mark 所有 secondary 同步 delete-mark 普通事务只做逻辑删除；Purge 之后才物理移除

08Undo、MVCC 与 Purge

当前事务最多同时拥有两条 Undo log：INSERT 与 UPDATE 分离， 二者共享事务级单调递增的 undoNo。回滚段 Page 3 的 v4 header 是槽位、 history、free list 与两类 cache 的权威持久状态；Undo 页使用 v3 格式，首个页含 log header， 后续页只保存链与 record area。

Undo 类型记录什么提交后的去向主要消费者
INSERT Undo 撤销本事务新插入记录所需信息 提交后无需支持旧版本读取，可直接进入 cache/free/drop 流程 statement rollback / transaction rollback
UPDATE Undo 更新或删除前的旧值、旧隐藏列与索引补偿信息 提交后进入 history，直到没有 ReadView 可能再访问 MVCC、rollback、purge

一致性读如何沿版本链回溯

  1. 先读当前聚簇版本。记录中的 DB_TRX_ID 表明该版本由哪个事务产生。
  2. ReadView 判断可见性。REPEATABLE READ 在事务内复用首个 ReadView；READ COMMITTED 每个 statement 新建。
  3. 不可见则跟随 RollPointer。DB_ROLL_PTR 定位 Undo record，重建上一版本。
  4. 重复直到可见或链结束。链损坏、指针越界或 Undo 格式不匹配都会 fail-closed，而不是返回可能错误的数据。

当前 RollPointer 只编码 pageNo + offset，没有携带 space/rseg 身份。这与“单一系统 Undo 表空间、单一默认 Rollback Segment”的现状相匹配，但也是未来支持多 Undo 表空间时必须升级的格式边界。 大型 Undo payload 会写入专用 UNDO_PAYLOAD 链，避免占满普通 Undo 页。

Purge 为什么不能看到 delete-mark 就删

Purge 依据全局低水位选择 history 中已经对所有活跃 ReadView 不可见的版本，再通过版本安全检查、 DML row guard 和二级索引回查，物理移除 delete-mark 的 secondary/clustered record，最后回收 Undo。 当前 Purge 可由后台 worker 驱动，但“Purge 自动触发 Undo tablespace truncate”尚未生产接线； truncate 恢复服务已经存在，主动调度仍属于缺口。

09Redo、WAL、Doublewrite 与 Checkpoint

每个包含持久修改且成功提交的 MTR 都先把 Redo batch 追加到持久 Redo ring，取得连续 LSN 区间，再把 endLSN 盖到被修改页。生产默认使用 8 个 8 MiB 日志文件；日志按 512 B block 编码， block header 为 32 B、payload 为 472 B，末尾 8 B 保存 trailer。文件和容量均可配置，默认值不是格式上限。

Redo 同时包含通用物理记录与少量模块语义记录： PAGE_INIT、PAGE_BYTES、FSP allocate/free/metadata delta、 Undo metadata/payload、B+Tree page delta 和 transaction state delta。 因此它是“物理页修改为主、关键元数据带语义”的混合方案，不是纯逻辑 binlog，也不是对 MySQL MLOG 的复刻。

WAL：dataPage 可写入数据文件 ⇔ redo.flushedLSN ≥ page.pageLSN

批量刷页的真实顺序

  1. 选择候选页。Page Cleaner 从 flush list 或 LRU 形成 batch；FLUSHING 页仍保留在脏页边界中。
  2. 按 spaceId 排序取得共享 lease。阻止 drop/truncate 与当前批次交错，同时避免跨空间锁序不稳定。
  3. 复制页面快照并读取 pageLSN/dirtyVersion。慢 I/O 不持有 page latch；快照版本用于完成时复核。
  4. 等待 Redo durable。所有快照对应的 pageLSN 被 flushed LSN 覆盖后，才允许继续。
  5. 重算 checksum 并写 Doublewrite。页面按来源进入 FLUSH_LIST 或 LRU 的独立物理文件；默认每通道 1024 slots。
  6. 批量写表空间并 force。按 space 分组写入真实数据文件；全部成功后才释放 Doublewrite slot。
  7. 按版本完成 flush。只有 frame generation 与 dirtyVersion 未变化才清理脏状态；并发再修改的页继续留在 flush list。

Doublewrite 默认工作在 DETECT_AND_RECOVER 模式，也支持 OFF 与 DETECT_ONLY。Checkpoint 使用双 4 KiB slot 持久化标签；推进前先持久化事务恢复 sidecar， 再写 Redo checkpoint label，最后才允许回收旧 Redo，避免日志边界跑到事务恢复信息前面。

10崩溃后如何恢复

CrashRecoveryService 只恢复物理页与事务状态，不执行普通 SQL。恢复期间流量保持关闭；任何阶段发现不满足继续运行的不变量， 都抛出启动恢复异常并 fail-closed。

  1. TRAFFIC_CLOSED。保持存储入口关闭，防止前台访问观察到半恢复状态。
  2. DOUBLEWRITE_REPAIR。从两个通道扫描完整页副本，修复被 torn write 破坏且可判定目标身份的数据页。
  3. REDO_REPLAY。从 checkpoint 边界扫描完整 Redo batch；仅当页上 LSN 落后时幂等重放。
  4. REDO_BOUNDARY_INSTALL。把恢复扫描得到的 closed/written/flushed/reclaim 边界安装回运行时 Redo 状态。
  5. UNDO_TABLESPACE_RESUME。恢复被中断的 Undo tablespace truncate 状态机，使生命周期回到可解释状态。
  6. SPACE_FILE_RECONCILE。对照持久元数据与实际文件长度，处理已记录空间扩展但文件尾尚未完成的窗口。
  7. UNDO_ROLLBACK。从事务恢复状态识别 loser transaction，按 Undo 逆序撤销未提交修改。
  8. RESUME_PURGE。重建 history/purge 进度，继续清理由已提交旧版本留下的垃圾。
  9. 恢复结果 force，然后 OPEN_TRAFFIC。必要时持久化恢复产生的新 Redo 和脏页，再开放流量；force 是阶段间持久化动作，不是独立的 RecoveryStageName。
只读诊断模式走另一条链：它扫描并校验 Redo、Doublewrite 与表空间，但不 apply、force 或写入恢复结果， 最终只开放 READ_ONLY_DIAGNOSTIC。这不是“用只读方式执行一次普通恢复”。

在公共 DatabaseEngine 外层，存储恢复完成后还会进行数据字典发现、DDL recovery 和 SDI reconcile， 最后才开放完整数据库服务。本文聚焦存储部分，因此没有把 DDL 阶段画成 CrashRecoveryService 的内部步骤。

11当前实现边界与关键结论

下面的状态表是阅读本文的边界：它防止把“借鉴 InnoDB”误解为“已经复刻 MySQL”，也防止把仓库里的 预留类型误认作生产链路。更细的逐模块事实以 current-implementation-map.md 和源码为准。

状态当前能力必须保留的边界
WIRED 表空间/FSP/segment/page 分配、分片 Buffer Pool、MTR、任意高度 B+Tree、Record/LOB、 多索引 DML、锁/MVCC、双 Undo log、Purge、持久 Redo、WAL/Doublewrite/Checkpoint、Crash Recovery 这些能力由 StorageEngine 生产装配，并有对应协作/恢复测试
SIMPLIFIED 页/记录/Undo/Redo 都使用项目自有格式；页 0 内嵌 XDES；默认一个 Buffer Pool instance； 单系统 Undo 表空间与单默认 rseg；物理 Redo 为主 不与 MySQL 数据文件或日志二进制兼容；默认值不代表架构只能如此

UNSUPPORTED 主键 UPDATE、READ UNCOMMITTED / SERIALIZABLE、任意复合比较范围、命名 savepoint、 MySQL 原生表空间直接互读 调用方应得到明确错误，不能静默降级为可能错误的行为

关键结论速查

#结论为什么重要
1页大小可配置，Extent 内页数不是永远 64。分析页号、XDES 和容量时必须先看 PageSize。
2Disk/FSP 不理解 record，Record 也不直接碰文件。这是当前存储分层最核心的依赖边界。
3MTR 原子地保护物理修改，Transaction 原子地保护一条业务写。一次表级 INSERT 会跨多个短 MTR。
4事务锁等待前必须释放 page latch/fix，成功后重新 B+Tree 定位。避免锁序反转，也防止使用已经移动的物理位置。
5Page Directory 是二分入口，不是记录本体。实际查找是“slot 二分 + 分组链扫”。
6二级键 UPDATE 采用 new-before-old。语句中途失败时仍能由 Undo 找回可达入口。
7INSERT Undo 与 UPDATE Undo 的提交后命运不同。只有 UPDATE Undo 必须为历史一致性读长期保留。
8数据页写盘前必须满足 Redo WAL。否则崩溃后可能出现无法由日志解释的新数据页。
9Doublewrite、Redo、Undo 解决三种不同问题。分别处理 torn page、已提交/物理重放、未提交事务回滚。
10Checkpoint 是安全交集，不是“当前最大 LSN”。推进过快会回收仍被脏页或事务恢复状态需要的日志。
11恢复必须按阶段串行并保持流量关闭。Doublewrite 修复、Redo、Undo、Purge 的顺序具有恢复语义。
12当前实现借鉴 InnoDB 思想，但不承诺 MySQL 二进制兼容。阅读布局数字时应以本项目源码为准，不能套用 MySQL 常量。
