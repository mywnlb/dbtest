01

总览：一条 SQL 的一生

前七篇深入 InnoDB 引擎内部。本篇站在 MySQL Server 层，讲一条 SQL 文本如何变成可执行计划、再驱动引擎取数。本版相比初版大幅扩充了优化器部分——因为"SQL 写得好不好、优化器选得对不对"，往往是性能差异的最大来源。

02

SQL 解析：词法 · 语法分析 · 预处理

SQL 是字符串，要先变成结构化的树。分三步：词法分析切词、语法分析建树、预处理做语义校验。本版重点把语法分析讲细。

① 词法分析 Lexical Analysis

把字符流切成词法单元（token）：关键字、标识符、运算符、字面量等。例如 SELECT id FROM t → [SELECT][id][FROM][t]。词法器（lexer）只认"词"，不管语法。它还负责跳过空白和注释、识别字符串/数字边界。

② 语法分析 Syntax Analysis（重点展开）

语法分析（也叫 parsing）把 token 流按 SQL 语法规则（文法 / grammar）组装成一棵树。MySQL 的语法分析器是基于 Bison（yacc 家族）的 LALR(1) 分析器，文法定义在源码 sql/sql_yacc.yy 里，由成百上千条产生式规则（production rule）描述"什么样的 token 序列是合法的什么结构"。

文法与产生式

文法用类似 BNF 的产生式描述语言结构。简化地看，一条 SELECT 语句的规则大致是：

// 高度简化的产生式（示意，非真实 yacc 语法）
query        → SELECT select_list FROM table_ref where_opt
select_list  → '*' | column (',' column)*
where_opt    → ε | WHERE expr
expr         → expr AND expr | expr OR expr | column op value | '(' expr ')' | ...

分析器读入 token，不断尝试用规则去"规约（reduce）"：当一串 token 匹配某条规则右侧，就把它们归约成左侧的非终结符（如把 id , name 归约成 select_list）。

LALR(1) 的移进-归约（shift-reduce）

LALR(1) 是一种自底向上的分析法，用一个栈和"向前看 1 个 token"来决策，每一步只做两个动作之一：

  • 移进（shift）：把当前 token 压入栈，继续读下一个。
  • 归约（reduce）：栈顶的若干符号匹配了某条产生式右侧，就弹出它们、压入左侧非终结符。

不断 shift/reduce，最终整个输入被归约成起始符号 query，同时自底向上构造出语法树。若读到某个 token 时既无法移进也无法归约（不符合任何规则），就在此处报 You have an error in your SQL syntax near '...'——错误信息里的"near"正是分析卡住的位置。

语法分析的产物在 MySQL 8.0 中是一棵规整的抽象语法树 AST（如 Parse_tree_root → Query_block 等节点），结构清晰、便于后续遍历改写。

③ 预处理 Preprocessor

语法对≠能执行。预处理做语义检查与展开：表/列是否存在、列名是否有歧义（多表同名）、是否有访问权限、把 * 展开成具体列、把视图展开成基表定义等。通过后得到语义明确的 AST，交给优化器。

三步各管一层

词法管"是不是合法的词"，语法分析管"词的排列是不是合法的结构"（靠文法 + 移进归约建树），预处理管"这些名字是否真实、是否有权访问"（语义）。三关都过，才有一棵可优化的树。8.0 已移除查询缓存，故没有"解析前查缓存"这一步。

03

SQL 执行：火山模型

优化器产出执行计划后，执行器按计划驱动引擎取数。组织方式是经典的火山模型（Volcano / 迭代器模型）。

执行计划是一棵算子树，每个算子（扫描、过滤、连接、排序、聚合）都实现统一的 next() 接口，每次返回一行。上层算子调用下层算子的 next() 拉取数据：控制自顶向下，数据自底向上一行行涌出，故名"火山"。

两个延伸

① 执行器/引擎边界。执行器调用存储引擎 handler 接口（"读下一行""按索引定位"），怎么从 B+ 树/Buffer Pool 取数是引擎（前七篇）的事。② 火山模型的代价。逐行 next() 有虚函数开销，分析型大数据量不划算，现代数据库引入批量处理与算子合并优化，但理解经典火山模型是看懂执行计划树的基础。

04

优化器总览：架构与 RBO / CBO

优化器是 Server 层的"大脑"。它接过预处理后的 AST，分两大步把它变成最优执行计划：

  • 逻辑优化（查询变换）—— 等价改写 AST，不看成本，只为变成更易优化的形状（化简条件、子查询转半连接、派生表合并）。见第 8–10 章。
  • 物理优化（基于成本选路）—— 为可行方案估算成本，选最便宜的：用哪个索引、哪种访问方式、哪种连接算法、什么连接顺序。见第 11–19 章。

RBO 与 CBO

RBO 基于规则CBO 基于成本
选路依据固定优先级规则（"有索引就用"）估算成本，选最小
看数据分布吗不看看（靠统计信息）
典型问题索引命中 90% 也照用 → 慢统计不准时会估错
MySQL 现状少数固定场景保留主体，现代优化器以 CBO 为主
为什么 CBO 会"选错"

CBO 的聪明完全建立在成本估算准确上，而成本估算又依赖统计信息。统计过期/不准 → 估算偏差 → 选了次优计划（放着好索引不用、选错连接顺序）。这就是"突然走错索引"第一反应是 ANALYZE TABLE 的原因。理解"CBO 依赖统计估算"，就理解了优化器一大半脾气。第 5–7 章正是讲它如何估、估什么。

05

成本模型：成本怎么算

CBO 口中的"成本"是个无量纲相对值，只用来比较计划优劣，主要两部分相加：

  • I/O 成本：读多少个页 × io_block_read_cost（≈1.0）。读"已在 Buffer Pool 的页"比"磁盘页"便宜，优化器会参考缓存状态。
  • CPU 成本：评估多少行 × row_evaluate_cost（8.0 ≈0.2）。比较、算表达式都算 CPU 成本。
// 表 1000 页、10 万行
全表扫描 ≈ 1000 × 1.0 + 100000 × 0.2 = 21000
索引命中 200 行(回表200次) ≈ 200 × 1.0 + 200 × 0.2 ≈ 240   → 选索引
索引命中 8 万行(区分度低) ≈ 80000 回表 ≈ 8万+                → 反而比全表贵, 放弃索引
成本常量外置且引擎相关

8.0 把成本常量放在系统表：mysql.server_cost（Server 层操作，如建临时表 temptable_create_cost、排序）与 mysql.engine_cost（引擎层，如 memory_block_read_cost vs io_block_read_cost）。一般无需改，但理解它们能解释"同一 SQL 在内存命中率不同时选择不同"。成本的核心驱动是"预估要碰多少行/页"，这个预估来自统计信息（第 6 章）与行数估算（第 7 章）。

06

统计信息：基数与直方图

成本模型要估"碰多少行"，靠两类统计：索引基数和列直方图。

① 索引基数 Cardinality

基数 = 索引列上不同值的个数（估算），衡量区分度。优化器用它估"等值/范围查询命中多少行"。来源：InnoDB 不全表精确统计，而是随机采样若干页（innodb_stats_persistent_sample_pages 默认 20）估算，故是近似值。存储：8.0 默认持久化（innodb_stats_persistent=ON），存 mysql.innodb_index_stats，重启不丢。更新：数据变化超约 10% 自动重算，或手动 ANALYZE TABLE。

② 列直方图 Histogram（8.0 新增）

基数只说"有几种值"，不说"各占多少"（分布）。直方图刻画分布，尤其服务无索引的列。两种：singleton（取值少，每值一桶记占比）、equi-height 等高（取值多，每桶覆盖大致相等行数）。建法：ANALYZE TABLE t UPDATE HISTOGRAM ON col;，存 mysql.column_statistics。

统计不准的连锁后果

统计是 CBO 的燃料，燃料脏了决策就歪：基数采样偏差 → 误判区分度 → 选错索引；缺直方图 → 倾斜列按"均匀分布"估 → 严重高/低估命中行 → 选错计划甚至连接顺序。莫名慢查询/走错索引的标准动作：ANALYZE TABLE 重收基数 + 为关键无索引过滤列建直方图。

07

范围优化与行数估算（index dive）

对 WHERE col BETWEEN a AND b 这类范围条件，优化器必须回答一个关键问题：这个范围大概命中多少行？这个估算值直接决定"走索引值不值"（命中少→走索引，命中多→全表扫描）。负责此事的是范围优化器（Range Optimizer）。

index dive：钻进索引数一数

对范围的行数估算，光靠基数（只是平均）不够准。MySQL 用一种叫 index dive（索引下潜）的方法：真的钻进 B+ 树，定位范围的起点和终点两个位置，估算这两点之间隔了多少条记录。这比用平均基数估得准得多，尤其对分布不均的数据。

两个实战要点

① IN 列表过长会放弃 dive。每个范围（每个 IN 值、每段 OR）都要 dive 一次，代价随数量上升。当等值范围数量超过 eq_range_index_dive_limit（默认 200）时，优化器放弃逐个 dive，改用基数平均估算——可能不准。所以超长 IN (...) 列表偶尔会出现行数估偏、选错计划，是这个机制导致的。
② 范围优化也有成本上限。当 WHERE 里 OR/AND 组合出的范围过于复杂，构造与评估这些范围本身耗费过大时，优化器会受 range_optimizer_max_mem_size 限制而放弃精细的范围分析，退回更粗的估算。复杂条件下行数估不准，可往这两个方向排查。

08

逻辑优化①：条件化简与等价变换

逻辑优化是不看成本的等价改写，目标是把查询变成更规整、更易物理优化的形态。先看一组"条件层面"的变换：

① 常量传递与折叠

  • 常量折叠：WHERE a = 5 + 3 → a = 8，编译期算好。
  • 等值传递：WHERE a = b AND a = 10 → 推出 b = 10，让 b 也能走索引。这是个威力很大的变换——它能把一张表的过滤条件"传染"给等值连接的另一张表。
  • 常量表检测：若某表在 WHERE 里被主键/唯一键等值锁定到至多一行，优化器把它当作常量表（const），在优化阶段就先读出来，其值直接代入其它条件。EXPLAIN 的 type=const/system 就是它。

② 恒真/恒假与冗余消除

  • WHERE 1=1 AND x>5 → 去掉恒真项 1=1。
  • WHERE 1=0 或 WHERE id=1 AND id=2 → 矛盾，优化器直接返回空结果（Impossible WHERE），根本不查表。EXPLAIN Extra 会显示 Impossible WHERE。
  • 无用列/排序消除：如 COUNT(*) 时去掉无关的 ORDER BY；子查询里多余的 DISTINCT 等。

③ 外连接转内连接（重要）

若 LEFT JOIN 的 WHERE 含对右表列的非空过滤（如 WHERE b.x = 5），那些"右表为 NULL"的行本会被滤掉，外连接等价于内连接。转成内连接的好处是：内连接对连接顺序没有限制，优化器能自由选驱动表（外连接的驱动顺序是受限的）。这个变换常常能解锁更优的连接计划。

为什么先做这些

这些变换都在物理优化之前做，是为了给物理优化"铺路"：等值传递让更多表能用索引、外连接转内连接放开连接顺序、Impossible WHERE 直接短路。规整后的查询，物理优化才能算出更好的账。下一章看更复杂的变换——子查询。

09

逻辑优化②：子查询与半连接四策略

子查询是优化重灾区。朴素执行可能对外层每一行都跑一遍子查询（关联子查询），慢得离谱。优化器的核心手段是把 IN 子查询改写成半连接（semi-join），再用连接算法高效执行。

什么是半连接

SELECT * FROM o WHERE uid IN (SELECT id FROM u WHERE vip=1)：语义是"o 中 uid 能在 u 里找到匹配的行"。半连接 = 连接，但每条外层行只要"找到至少一个匹配就保留，且不因多个匹配而重复"。关键在去重语义——这正是它和普通 JOIN 的区别。优化器有四种策略来实现它，按成本择优：

策略做法适合场景
FirstMatch外层每行去内层找，一旦找到第一个匹配就停，不再找更多内层每个外层行有大量匹配时，避免重复
Materialization 物化先把子查询结果去重物化成临时表（带哈希索引），外层再去查它子查询结果集小且被反复探测
Duplicate Weedout 去重当普通连接做，把多余的重复行用临时表事后去重连接本身高效，去重代价低时
LooseScan 松散扫描利用内层索引有序，每个不同值只取一个（跳过重复），天然去重内层连接列有索引且重复值多
其它子查询变换 + 实践

反连接 antijoin：NOT IN (...) / NOT EXISTS 表达"找不到匹配的行"，8.0.17 起优化器能把它转成反连接高效执行（但要小心 NOT IN 遇到子查询含 NULL 时的三值逻辑陷阱）。
关联子查询解关联：能转成连接的尽量转，把"逐行 N 次"变成"一次连接"。
实践：① 多数情况无需手动把 IN 改写成 JOIN，优化器会做；但遇到老版本或它没选好策略时，手动改写仍有效。② 用 EXPLAIN 看是否出现 <subquery>、FirstMatch、materialized 等标记判断走了哪条路。③ 子查询结果集大小直接影响策略选择，必要时给子查询的过滤列建索引。

10

逻辑优化③：派生表 合并 vs 物化

FROM (SELECT ...) AS t 这种派生表（derived table），以及视图，优化器有两种处理方式，差别巨大：

① 合并 Merge（首选，更快）

把派生表的定义"摊平"合并进外层查询，变成一个没有子查询的单层查询。这样派生表的 WHERE、连接能和外层一起优化，外层条件可以下推进去、索引可以用上。例如 SELECT * FROM (SELECT * FROM t WHERE a>5) d WHERE d.b=3 合并成 SELECT * FROM t WHERE a>5 AND b=3，两个条件一起走索引。

② 物化 Materialize（兜底，较慢）

当派生表无法合并时（含聚合 GROUP BY、DISTINCT、LIMIT、窗口函数、UNION 等"不可摊平"的结构），优化器把它先执行、结果存进一张内存临时表，外层再查这张临时表。代价：要落地一份中间结果，且临时表上条件下推、索引利用都受限。

实践要点

① 尽量让派生表可被合并。能在外层做的过滤就别塞进派生表里再加聚合，避免不必要地触发物化。
② 物化的临时表默认在内存（TempTable 引擎），超限转磁盘。大派生表物化转磁盘会明显变慢，可关注 tmp_table_size / temptable_max_ram。
③ 8.0 派生表可"按需物化 + 加索引"。优化器有时会给物化的临时表自动加一个索引（auto-key）以加速外层连接它——EXPLAIN 里会看到 <auto_key0>。
④ CTE（WITH）同理：非递归 CTE 的处理方式与派生表一致（合并或物化）。

11

访问路径与索引合并

进入物理优化。对单表，优化器要选访问路径——用什么方式取数据，再用成本模型比较。候选：

  • const / system——主键或唯一键等值，至多一行，优化阶段就读出（第 8 章常量表）。最快。
  • 全表扫描 ALL——扫聚簇索引全部叶子。命中占比大时反而最优。
  • 索引访问——ref（等值）/ range（范围）走二级索引，可能回表。
  • 覆盖索引——查询列全在索引里，免回表（第 12 章）。
  • 索引合并 index_merge——见下。

索引合并 Index Merge

当 WHERE 用 OR/AND 连接了多个不同索引各自能用的条件，优化器可分别用多个索引各扫一遍再合并结果，而非放弃索引走全表。三种：union（OR 取并集）、intersection（AND 取交集）、sort-union（带排序的并集）。EXPLAIN type=index_merge。

选路 = 成本 + 统计 + 行数估算

优化器对每条候选路径，用第 6 章统计 + 第 7 章 index dive 估"命中行数"，再用第 5 章成本公式算成本，选最小。"为什么没走我的索引"几乎总是：优化器估"走它成本高于别的路径"（区分度低/命中太多/回表太贵），或统计不准估错。下一章讲怎么把"回表太贵"这个最大痛点降下来。

12

减少回表：覆盖索引 / ICP / MRR

第六篇讲过：走二级索引常要回表（拿主键再查聚簇索引），命中多时大量随机 I/O是性能杀手。优化器有三件武器专门对付回表，合起来理解最透彻。

① 覆盖索引 Covering Index —— 根本不回表

查询所需的所有列都在二级索引叶子里（索引列 + 主键），就无需回表，在索引这棵树里就返回。EXPLAIN Extra: Using index。把高频查询的返回列也纳入联合索引即可制造覆盖。

② 索引条件下推 ICP（Index Condition Pushdown）—— 回表前先过滤

当 WHERE 有能用索引列、但不能用于定位的条件（如后缀 LIKE、跳列条件），ICP 让存储引擎在回表之前，先用索引叶子里现成的列值过滤一遍，只有满足的行才回表。EXPLAIN Extra: Using index condition。它把过滤从"昂贵的回表后"提前到"便宜的索引扫描时"。

③ 多范围读 MRR（Multi-Range Read）—— 把随机回表变顺序

这是本版新增的重点。二级索引是按索引列有序的，但它指向的主键值是乱序的。所以逐行回表时，对聚簇索引的访问是随机 I/O（一会儿读这页一会儿读那页），磁盘最怕随机。MRR 的优化：先把要回表的一批主键收集起来、排好序，再按主键顺序去聚簇索引取——把随机 I/O 变成顺序 I/O。

三件武器的关系

覆盖索引是"釜底抽薪"——直接不回表，最优；ICP是"少回表"——回表前先滤掉无关行；MRR是"回得快"——非回不可时把随机 I/O 变顺序。优先级：能覆盖就覆盖，不能覆盖则靠 ICP 减少回表行数 + MRR 加速剩余回表。三者常协同出现在同一个执行计划里。
实践：① 为高频查询设计覆盖索引（第六篇）；② ICP、MRR 默认开启（受 optimizer_switch 控制，第 20 章），一般无需干预，了解它们能读懂 EXPLAIN 里的 Using index condition / Using MRR。

13

连接顺序与搜索算法

多表连接时，优化器要决定表以什么顺序连接。这是物理优化里组合空间最大、最烧脑的决策。本版深入讲它怎么搜。

为什么顺序关键：前缀成本累积

连接是逐表"长出"中间结果。优化器按"连接前缀"逐步累积成本：先定第一张表的成本，再在其基础上加第二张，依此类推。核心原则：让过滤性强（结果行数少）的表靠前，尽早把中间结果压小，后续每一步都在小集合上做，总成本就低。

搜索算法：穷举太贵，用贪心 + 剪枝

N 张表有 N! 种顺序，10 张表就是 360 万种，全穷举不现实。MySQL 的做法：

  • 贪心 + 深度限制：用 optimizer_search_depth 控制搜索深度。它在"搜得够不够全"和"搜索本身的开销"间权衡——设大一点更可能找到最优顺序，但优化耗时上升；设小则快但可能漏掉最优。默认值（62/自适应）通常够用。
  • 前缀剪枝：搜索中若某个"部分连接顺序"的累积成本已超过当前已知最优全序的成本，立即剪掉该分支，不再往下试——大幅缩小搜索空间。
  • 启发式：优先尝试"小表/强过滤表"作前缀，更快逼近最优。
驱动表术语 + 干预手段

驱动表（outer）先遍历，被驱动表（inner）对驱动表每行去匹配。优化关键：驱动表小（过滤后小）+ 被驱动表连接列有索引，正是 INLJ 高效（第 14 章）的前提。
干预：① 极少数情况你比优化器更懂数据，可用 STRAIGHT_JOIN 强制按书写顺序连接，或用连接顺序提示 JOIN_ORDER（第 20 章）；② 但更常见的是优化器选错源于统计不准或缺索引——先 ANALYZE TABLE、补索引，而非急着强制顺序。强制顺序是双刃剑，数据分布变化后可能反成枷锁。

14

嵌套循环族 NLJ / INLJ / BNL / BKA

嵌套循环是最基础的连接思想，长期是 MySQL 唯一连接方式（8.0 才加 Hash Join）。本版讲全四个变体（含 BKA）。

① 简单嵌套循环 NLJ

对驱动表每一行，把被驱动表从头扫一遍找匹配，O(R×S)。被驱动表大且无索引时极慢，MySQL 几乎不用纯 NLJ。

② 索引嵌套循环 INLJ（主力）

被驱动表连接列有索引：对驱动表每行，用索引直接定位匹配行（一次 B+ 树查找），从 O(R×S) 降到约 O(R×log S)。EXPLAIN 被驱动表 ref/eq_ref。

③ 块嵌套循环 BNL

被驱动表无索引时，把驱动表一批行攒进 join_buffer，被驱动表只扫一遍、与整批比对，把扫描次数从 R 次降到 R/缓冲行数次。EXPLAIN Using join buffer (Block Nested Loop)。8.0.20 起在等值连接场景被 Hash Join 取代。

④ 批量键访问 BKA（Batched Key Access）—— INLJ + MRR

本版新增。INLJ 虽快，但它对被驱动表的每次索引定位都可能随机回表（第 12 章 MRR 讲的痛点）。BKA 的优化：把驱动表的一批行先攒进 join_buffer，提取它们的连接键排好序，再用 MRR 批量、有序地去被驱动表查找——把 INLJ 的随机回表变成顺序回表。本质是"INLJ + join_buffer + MRR"的组合。EXPLAIN Using join buffer (Batched Key Access)，需开启对应 optimizer_switch。

现代版本的连接选择

有索引：默认 INLJ；大量回表时 BKA（需开 batched_key_access）可进一步把回表顺序化。无索引等值连接：8.0.20 起走 Hash Join（取代 BNL）。EXPLAIN 据此显示 (Block Nested Loop) / (hash join) / (Batched Key Access)。给连接列建索引让其走 INLJ，仍是最常见也最有效的连接优化。

15

Hash Join 详解

MySQL 8.0.18 引入，专治"被驱动表连接列无索引的等值连接"（旧版只能慢吞吞的 BNL）。复杂度 O(R+S)，远胜嵌套循环的 O(R×S)。

两阶段：build 与 probe

  • Build：选较小的表，按连接列算哈希，建一张内存哈希表（key=连接列，value=行）。
  • Probe：遍历较大的表，每行算同样的哈希，去哈希表 O(1) 查找匹配，命中即输出。
何时用 + 怎么看

触发：等值连接 + 被驱动表连接列无可用索引（有索引则 INLJ 通常更优）。观察：普通 EXPLAIN Extra: Using join buffer (hash join)；EXPLAIN FORMAT=TREE / EXPLAIN ANALYZE 直接显示 Hash Join 算子（第 21 章）。本质：把嵌套相乘换成哈希 O(1) 查找，对"大表×大表、无索引、等值"效果最显著。

16

Sort-Merge Join 详解

Sort-Merge Join（排序归并连接）是数据库理论的三大经典连接算法之一。先讲原理，再诚实说明 MySQL 的实际情况（易被误解，务必讲清）。

原理：各自排序，再归并

  • Sort：两表都按连接列排序（若来自索引已有序则跳过）。
  • Merge：双指针同时从两个有序序列头部往后走，像拉链：哪边小推进哪边，相等则输出。两表各只扫一遍。

复杂度 O(R logR + S logS)（排序）+ O(R+S)（归并）；输入已有序时只剩 O(R+S)，很高效，且能高效处理范围连接（如 R.k ≥ S.k）——这是 Hash Join 做不到的。

关于 MySQL 的实话（重要）

MySQL 并没有实现独立的 Sort-Merge Join 连接算子——这是常见误解，必须讲清：
① MySQL 历史上只有嵌套循环连接，8.0 才补 Hash Join，从未提供专门的 sort-merge join 算子。
② Sort-Merge 是 Oracle、PostgreSQL、SQL Server 等的标配，常用于大表等值/范围连接或输入已按连接键有序时。
③ MySQL 用别的手段达成类似目的：连接键有索引走 INLJ（索引本身有序，类似"有序访问"）；无索引等值连接走 Hash Join。MySQL 大量用排序（filesort，第 18 章），但那是为 ORDER BY/GROUP BY，不是连接算子。
记住：讲算法 Sort-Merge 是三大经典之一；落到 MySQL 实现，它的连接武器只有 INLJ（含 BKA）+ Hash Join 两件，没有第三件。

实践结论

排序、分组与去重优化

ORDER BY / GROUP BY / DISTINCT 的代价常被低估。优化器的目标是尽量利用索引的有序性，避免昂贵的额外排序和临时表。

ORDER BY：索引有序 vs filesort

  • 用索引有序（最优）：若 ORDER BY 列是某索引的有序前缀，顺着索引读即有序，无需排序，EXPLAIN 无 filesort。
  • filesort（额外排序）：无法用索引序时，取出数据在 sort_buffer 排。EXPLAIN Using filesort——不一定是磁盘排序：放得下就内存排，放不下才用临时文件归并。两种内存策略：全字段排序（排序列+返回列一起排，少回表）和 rowid 排序（只排序列+主键，排完回表，省内存多回表），由 max_length_for_sort_data 等阈值决定。

GROUP BY：松散 vs 紧凑索引扫描

  • 松散索引扫描 Loose Index Scan（最优）：能利用索引有序跳着读，每个分组只取需要的少量记录（如配合 MAX/MIN），不必扫全部。EXPLAIN Using index for group-by。极快。
  • 紧凑索引扫描 Tight Index Scan：扫描索引的一个连续范围，但仍按索引序，无需额外排序/临时表。
  • 临时表 + 排序（最差）：无法用索引序时，建临时表分组 + filesort。EXPLAIN Using temporary; Using filesort——性能警报。

DISTINCT

DISTINCT 本质是分组去重，优化路径与 GROUP BY 类似：能走索引有序则免临时表，否则建临时表去重。

实践

① 让 WHERE + ORDER BY/GROUP BY 的列组成合适联合索引（注意第六篇最左前缀：WHERE 等值列在前、ORDER BY 列紧随，且排序方向一致），使过滤与排序都走索引序。② 看到 Using temporary; Using filesort 同时出现，多半要调索引。③ 适度调大 sort_buffer_size 让排序留内存，但每连接独占该缓冲，勿设过大。④ 8.0 支持降序索引，ORDER BY a ASC, b DESC 这种混合方向也能用索引序了。

分页与 LIMIT 优化

分页是高频场景，也是深分页（deep pagination）性能塌方的重灾区。本版专门讲它的原理与解法。

深分页问题：LIMIT 大偏移为什么慢

SELECT * FROM t ORDER BY id LIMIT 1000000, 10 看似只要 10 行，实则必须先扫描并丢弃前 100 万行才能取到第 1000001 起的 10 行。偏移越大，白扫的行越多——这就是深分页慢的根因。若还走二级索引，前 100 万行可能每行都回表，雪上加霜。

解法一：覆盖索引 + 延迟回表（deferred join）

先只用覆盖索引定位到那 10 行的主键（这一步不回表，很快），再用这 10 个主键去取完整行。把"100 万次回表"降为"10 次回表"：

-- 慢：前100万行都可能回表
SELECT * FROM t ORDER BY create_time LIMIT 1000000,10;

-- 快：子查询只走覆盖索引拿主键(不回表)，外层只回表10次
SELECT t.* FROM t
JOIN (SELECT id FROM t ORDER BY create_time LIMIT 1000000,10) d
ON t.id = d.id;

解法二：游标分页 / keyset pagination（最优）

不用 OFFSET，而是记住上一页最后一条的位置（如 id 或排序键），下一页直接从那之后取。无论翻到多深，每次都只读需要的行，恒定快：

-- 记住上页最后 id = 上次的 last_id
SELECT * FROM t WHERE id > :last_id ORDER BY id LIMIT 10;
-- 直接用索引定位到 last_id 之后, 零丢弃
取舍

游标分页是性能最优解，但有约束：① 只能"下一页/上一页"顺序翻，无法直接跳到第 N 页；② 排序键需唯一（或加主键做 tiebreaker，如 WHERE (sort_key,id) > (:k,:id)）保证不漏不重。延迟回表则保留了任意页跳转能力、改造小，是不能用游标分页时的折中。能用游标分页（如信息流、无限滚动）就用它；需要页码跳转的传统分页，用延迟回表 + 限制最大可翻页数。

优化器提示与开关

优化器大多数时候选得对，但偶尔会失手（统计不准、估算偏差）。MySQL 提供几种干预手段，让你在确有把握时引导或纠正它。

① 优化器开关 optimizer_switch

一组全局/会话级的布尔开关，控制各项优化是否启用：index_merge、mrr、batched_key_access、semijoin、materialization、derived_merge、hash_join 等。一般保持默认，排查"某优化是否在捣乱"时可临时关某项验证：

SET optimizer_switch = 'mrr=on,mrr_cost_based=off';
SET optimizer_switch = 'batched_key_access=on';  -- BKA 默认关，需手动开

② 优化器提示 Optimizer Hints（8.0 推荐）

写在 SQL 里的 /*+ ... */ 注释，只影响当条语句，比改全局开关更精准、更安全。常用：

提示作用
/*+ JOIN_ORDER(a,b,c) */指定连接顺序
/*+ INDEX(t idx_a) */ / NO_INDEX建议用/不用某索引
/*+ HASH_JOIN(t1,t2) */ / NO_HASH_JOIN强制/禁止哈希连接
/*+ BKA(t) */ / NO_BKA开/关该表的批量键访问
/*+ MAX_EXECUTION_TIME(1000) */限制语句最长执行 1 秒

③ 索引提示 Index Hints（老语法）

USE INDEX (idx) 建议、FORCE INDEX (idx) 强制、IGNORE INDEX (idx) 忽略某索引。FORCE INDEX 最强硬。

提示是双刃剑

能不用就不用。提示把"由优化器根据当前统计动态决策"变成"写死"。今天数据分布下 FORCE INDEX 是对的，明天数据变了它可能反成枷锁，且无法随统计更新而自适应。正确顺序：先 ANALYZE TABLE 修正统计、补合适索引、改写 SQL；这些都试过仍选错，才考虑用提示，并优先用语句级的 /*+ Optimizer Hints */（影响面小）而非改全局 optimizer_switch。提示更适合作为"临时救火"或"已充分论证的特例"，而非常规手段。

EXPLAIN 深入

看懂优化器的决策，全靠 EXPLAIN。本版讲它的多种格式与关键读法——这是把前面所有优化知识"落地验证"的工具。

三种格式，各有所长

形式给什么何时用
EXPLAIN（传统表格）type/key/rows/Extra 等概览快速看走没走索引、有没有 filesort
EXPLAIN FORMAT=JSON详细成本、各步预估行数、过滤比深挖优化器为何这样选（含 query_cost）
EXPLAIN FORMAT=TREE算子树形结构，直观显示连接算法看 Hash Join / 连接嵌套结构
EXPLAIN ANALYZE（8.0.18+）真正执行并给出实际行数与耗时对比"预估 vs 实际"，定位估算偏差

关键列再强调

  • type：访问类型梯子 const > eq_ref > ref > range > index > ALL，越左越好，看到 ALL/大 rows 的 index 要警惕。
  • key / key_len：实际用的索引 / 用了它的几个字节（反推联合索引吃进几列，第六篇）。
  • rows × filtered：预估扫描行数 × 过滤后保留比例 = 预估真正参与后续的行数。filtered 是常被忽略的列：它反映 WHERE 的选择率估算，连接时上一步的 rows×filtered 就是下一步的驱动行数，估不准会连锁选错连接顺序。
  • Extra：Using index(覆盖)、Using index condition(ICP)、Using MRR、Using filesort、Using temporary、Using join buffer (hash join) 等——前面各章的优化是否生效，全看这里。
EXPLAIN ANALYZE：最强诊断

EXPLAIN ANALYZE 是 8.0 的利器：它真的执行查询，在树形输出里同时给出每个算子的"预估行数 (cost=...) vs 实际行数 (actual=...)"和实际耗时、循环次数。当"预估"和"实际"差很多时，就抓到了优化器选错的元凶——通常是统计信息不准（回到第 6 章 ANALYZE TABLE / 建直方图）。这是把"优化器为什么慢/选错"从猜测变成实证的关键一步。注意它会真实执行（含写操作的语句慎用）。

动态 DDL：ALGORITHM 与 LOCK

动态 DDL（在线 DDL，Online DDL）：在不阻塞业务读写的前提下改表结构，是生产运维命脉，呼应第五篇 MDL。

为什么传统改表会锁表

早期改表用拷贝法：建新结构临时表 → 逐行拷贝旧数据 → 替换旧表。拷贝期间禁止写入（甚至禁读），大表可能锁几小时，业务瘫痪。Online DDL 为消除此阻塞而生。

两个子句：ALGORITHM 与 LOCK

ALTER TABLE ... ALGORITHM=? LOCK=? 控制改表方式与并发度：

ALGORITHM做法数据拷贝/重建并发 DML
INSTANT（8.0.12+）只改元数据，不碰数据行完全不拷贝几乎不阻塞，秒级
INPLACE原表原地重建/修改可能重建，但原地多允许并发读写
COPY建临时表逐行拷贝（旧做法）全量拷贝阻塞写（至多允许读）
LOCK并发允许
NONE读 + 写都允许
SHARED只允许读，写阻塞
EXCLUSIVE读写都阻塞
显式指定能"快速失败"

ALTER TABLE t ADD COLUMN x INT, ALGORITHM=INSTANT;——若不支持 INSTANT，MySQL 直接报错，而非悄悄退化成慢且锁表的 COPY。这样你第一时间知道"此变更没法秒级完成"，再决定换方案。指定 LOCK=NONE 同理（"必须不阻塞 DML，做不到就报错"）。显式声明是生产改表的安全习惯。

INSTANT / INPLACE 原理

INSTANT：只改元数据

8.0 的杀手锏。以加列为例：不重写每一行，而是把"新列及其默认值"记在表的元数据（数据字典）里。

  • 旧行：物理上没有新列，读取时若发现是"加列前的旧行"，就用元数据里的默认值补上，对用户透明。
  • 新行：插入时才物理带上新列值。

因完全不碰已有行，无论表多大都是毫秒级。支持范围逐版本扩大：8.0.12 支持末尾加列，8.0.29 起支持任意位置加列及更多操作。

INPLACE：原地重建 + Row Log 收集并发改动

加索引等必须扫描/重组数据的变更走 INPLACE：原地构建的同时，用一份"变更日志"吸收期间并发写：

  1. 1开始拿短暂 MDL 排他锁做准备，随即降级，放开并发 DML。
  2. 2后台扫描旧数据、构建新结构。期间用户增删改记入 row log，不打断业务。
  3. 3接近完成时再拿一次短暂排他锁，把 row log 积累的改动追平到新结构，切换、释放。
呼应第五篇：在线 ≠ 完全无锁

图里首尾那两个"短锁"——即便 Online DDL，开始和结束仍各需一个极短的 MDL 排他锁更新元数据。第五篇的"长事务挡住 DDL 的 MDL 写锁、排队的 DDL 又挡住所有后续查询 → 表假死"对 Online DDL 同样适用：中间不锁，但开头那个 MDL 拿不到，整个改表照样卡住、连累全表。生产改表：① 即便用 Online DDL，先确认无长事务占着这张表；② 给 DDL 设 lock_wait_timeout 快速失败；③ 超大表或需灰度时用 gh-ost / pt-online-schema-change（触发器或 binlog + 影子表 + 分批拷贝 + 瞬时切换），把锁窗口压到极致。

优化的统一心法

读完这 18 章优化内容，可以收束成一句心法：优化器的一切决策都建立在"预估要碰多少行/页"之上，而这个预估来自统计信息。所以做优化的三个根本动作是——① 喂准统计（ANALYZE、直方图），让优化器估得准；② 建对索引（覆盖、最左前缀、连接列索引），给优化器更便宜的路可选；③ 写好 SQL（避免深分页 OFFSET、让子查询可转半连接、让排序走索引序），从源头减少工作量。把这三件做好，绝大多数查询都不需要去强制提示——优化器自然会做出好选择。诊断时用 EXPLAIN ANALYZE 对比预估与实际，差距就是病灶。

与前七篇汇合

本篇的"执行器"调用前七篇构筑的引擎能力：优化器选定索引（⑥）→ 执行器调引擎按 B+ 树定位 → 页在 Buffer Pool 找（②）/ 没有从表空间读（①）→ 当前读加锁（⑤）/ 快照读选版本（④）→ 改动写 redo/undo（③）→ 提交走两阶段（⑦）。八篇构成完整链路：Server 层把 SQL 翻译成最优计划，引擎层把计划落到磁盘字节。
