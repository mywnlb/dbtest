01当前实现全景

当前工程的存储部分已经不是“文件里读写若干页”的单层示例。生产组合根 StorageEngine 会装配表空间、空间分配、Buffer Pool、MTR、B+Tree、事务、双类型 Undo、Redo ring、 Flush/Checkpoint、Doublewrite、Purge 与 Crash Recovery，并把上层 DML 收敛到稳定的存储 API。

WIRED 数据访问

TableDmlService 编排事务/锁与 B+Tree；B+Tree 用 Record 算子解释页内结构，并经 IndexPageAccess / MTR 访问 Buffer Pool。

WIRED 持久化

MTR 生成 Redo；刷页前等待 WAL；页面经双文件 Doublewrite 后写回表空间。

WIRED 恢复

启动时按阶段完成 Doublewrite 修复、Redo replay、边界安装、Undo/文件续作、事务回滚与 Purge 恢复。

本文的“已实现”只指当前生产代码确实装配并调用的能力。目标架构图中的长期方向、仅被测试覆盖的类型， 以及尚未接线的预留入口，不会被描述成当前事实。

02磁盘物理结构与逻辑组织

物理结构和逻辑结构描述的是同一批数据的两个视角。物理视角回答“哪个文件、哪个页号、哪个字节偏移”； 逻辑视角回答“这张表的哪个索引、哪个 Segment 拥有哪些 Extent 和 Page”。当前实现的基本关系是： 

物理定位：实例目录 → 持久文件 → Page（页号）→ 页内字节
逻辑归属：DD Table → GENERAL Tablespace → Index / LOB → Segment → Extent / Fragment Page → Record
事务历史：Transaction → INSERT / UPDATE Undo Log → UNDO Segment → UNDO / UNDO_PAYLOAD Page

PageStore 只负责按页读写表空间文件； DiskSpaceManager 才负责 FSP 元数据、生命周期、空间分配和访问租约。上层不会直接取得 FileChannel， 而 Segment、Extent、Record 也都不是独立操作系统文件。

物理结构一：实例目录中真实存在的文件

公共组合根以 EngineConfig.baseDir 为实例目录。文件名相似不等于格式相同：例如 mysql.ibd 是页对齐的追加式 DD catalog，当前并不是由 PageStore 管理的 FSP 表空间； 真正采用“Page 0 + Extent + Segment”格式的是用户表空间和系统 Undo 表空间。

系统 Undo 表空间undo_<spaceId>.ibu当前实例一个系统 Undo 空间、一个默认 Rollback Segment；保存 INSERT/UPDATE Undo 及外部 Undo payload。

辅助文件buffer-pool.dump、recovery-progress.jsonl、mysql.instance.lock分别用于热页预取提示、恢复诊断和跨进程互斥；前两者不是恢复正确性的权威输入。

物理结构二：单个 PageStore 表空间是一串定长页

用户 .ibd 和系统 .ibu 都是单文件表空间：页号从 0 开始线性排列，文件长度必须按实例 PageSize 对齐。Extent 是连续页号形成的物理分配单位；Extent 0 整区由系统保留，只有 page 0～3 具有固定管理职责，余下页不会交给普通 Segment 分配器，因此业务页从 Extent 1 开始。

pageOffset = pageNo × PageSize.bytes　｜　pagesPerExtent = extentSizeBytes ÷ PageSize.bytes　｜　extentFirstPageNo = extentNo × pagesPerExtent

页大小同时决定 Extent 页数和当前 FSP 容量

当前实现支持 4、8、16、32、64 KiB 页面。页不大于 16 KiB 时 Extent 为 1 MiB；32 KiB 页面使用 2 MiB Extent，64 KiB 页面使用 4 MiB Extent。每个 XDES entry 固定 68 B，并含 256 bit 页位图； 只有前 pagesPerExtent 位有效。

当前实现边界：设计文档保留了未来独立 XDES 管理页的方向，但生产 ExtentDescriptorRepository 目前只支持 page 0 内嵌区域；超过上表容量会 fail-closed， 不能照搬 InnoDB“每隔固定范围再出现一组 XDES 页”的物理图。

逻辑结构：Segment 是归属清单，不是连续文件区域

当前 CREATE TABLE 为表生成一个 TableStorageBinding。每个聚簇或二级索引都持有稳定 root PageId、一个 INDEX_LEAF Segment 和一个 INDEX_NON_LEAF Segment； 表中存在可外置字段时还会建立一个表级 LOB Segment。Segment 的权威证据是 page 2 中的 SegmentInode entry，而不是文件中某段连续字节。

每个索引 root 初建时从 leaf Segment 分配并保持物理页号稳定；root split 会在原页重建更高层级， 新叶页继续从 leaf Segment 分配，新增内部 child/sibling 页从 non-leaf Segment 分配。因此不能仅凭 PAGE_LEVEL 反推 root 页最初的 FSP Segment 归属。

权威结构粒度当前落盘内容回答的问题
DD / SDI binding表、索引space/path、root PageId、leaf/non-leaf SegmentRef、可选 LOB SegmentRef“这张表和这棵索引从哪里打开？”
SegmentInodeSegment384 B：segment id/purpose、使用与预留页数、三条 extent list、32 个 fragment slot“这个 Segment 拥有哪些页和区？”
XDES entryExtent68 B：state、owner segment、双向 list node、256 bit 分配位图“这个 Extent 属于谁、哪些页已占用？”
FilePageHeaderPagespaceId、pageNo、prev/next、pageLSN、pageType、checksum/trailer“这张物理页是谁、如何恢复和校验？”

页面分配不是直接“文件尾追加”

  1. 先用 fragment page。Segment 尚小时，从公共 FREE_FRAG Extent 取单页，并把页号写进该 INODE entry 的最多 32 个 fragment slot。
  2. 再用已有 Segment Extent。分配器沿该 Segment 的 NOT_FULL 或 FREE extent list 找候选，并同时更新 XDES bitmap 和 INODE 计数。
  3. 需要时取得新 Extent。叶段可根据顺序插入方向一次取得并挂入 1～4 个区；非叶、LOB、UNDO 和无方向请求保持单区取得。
  4. 物理空间不足时先扩文件。DiskSpaceManager 通过 PageStore.ensureCapacity/extend 扩展零填充文件，更新 page 0 的 currentSizeInPages 后重试。
  5. 同一 MTR 封闭元数据与新页。分配至少产生 FSP_PAGE_ALLOC 和 PAGE_INIT，并为 XDES/INODE/FLST after-image 收集 Redo，避免“页已出现但归属账本未恢复”。

03通用页信封与固定管理页

所有已经格式化的表空间页都先服从 PageEnvelopeLayout： 前 38 字节是通用物理头，最后 8 字节是 trailer。INDEX、UNDO、FSP 等页面只在两者之间解释自己的类型区域。 这使 page0 loader、恢复扫描和各 typed entrance 能在不理解 record 业务语义的前提下校验各自负责的 页身份、格式、校验和或 page LSN；普通 Buffer Pool 装载本身不承担统一的全页校验。

Page 1IBUF_BITMAP 信封IBUF_BITMAP 信封创建表空间时已格式化物理信封；UNWIRED Change Buffer bitmap body 与生产语义尚未接线

DiskSpaceManager.createTablespace 先建立文件，再在同一 MTR 中初始化 Page 0、Page 1、Page 2 的信封，并把 Extent 0 的 XDES 写成 FSEG_FRAG / owner=0；其中位图只把固定页 0～3 标为已占用， 该 Extent 也不会进入普通 Segment 的候选链。随后 GENERAL 物理建表格式化 Page 3 SDI，Undo bootstrap 则把同一页格式化为 RSEG_HEADER。普通空间进入 GENERAL / NORMAL，Undo 空间进入 UNDO / ACTIVE。 后续读写必须通过生命周期校验；CORRUPTED、INACTIVE、 TRUNCATING 或 DISCARDED 空间不会继续接受普通业务操作。

04Buffer Pool 与 Mini Transaction

LruBufferPool 是对外门面，内部由 Router 把 PageId 映射到 BufferPoolInstance。 生产默认配置为 1 个 instance，但路由、容量拆分、刷新和统计已经按多实例实现。

Frame 状态含义关键边界
FREE未绑定页面，可从 free list 获取不能被业务线程直接读取
LOADING一个线程正在从 PageStore 装载同页请求在锁外等待同一个 future
CLEAN内存页与数据文件一致fixCount 为 0 时可参与淘汰
DIRTY_PENDING页已修改，MTR 尚未提交发布不能被 flush 提前写出
DIRTY已带有效 page LSN 的脏页进入 flush list，并约束 checkpoint
FLUSHING某个快照正在刷盘仍算 dirty boundary；并发修改用版本号区分
EVICTING已选为 victim，正在撤销映射或刷出不能再授予新的业务 fix
STALE原页面映射已失效旧句柄不能重新发布到 page hash

命中、缺页与单飞装载

Instance 内部显式维护 page hash、frame mutex、free/LRU/flush list 锁与 page latch。 源码声明的强制锁序是 page hash → frame mutex → list lock → page latch。缺页时只有首个线程创建 LOADING frame 并发起磁盘读取；其他线程拿到相同的 load future，释放内部锁后再等待， 因而不会持有全局结构锁阻塞 I/O。

LRU 使用 midpoint old/new 分区；生产链已接入线性 read-ahead、热页 dump/load。 随机 read-ahead 类型存在，但默认关闭，不能把它写成默认运行能力。

因为未提交 MTR 的回滚不恢复 Buffer Pool 内容，修改者必须遵循“先完成所有可能失败的校验，再落字节”的写法； 或者自行保留可恢复快照。这个简化点是当前实现的重要边界。

05INDEX 页与记录组织

INDEX 页采用“记录堆从低地址向高地址增长、Page Directory 从 trailer 前向低地址增长”的布局。 IndexPageLayout 在 38 字节物理头后放置 28 字节 INDEX header，再放固定的 infimum、supremum，用户记录从 offset 98 开始。

页内插入只消费连续空闲区；已被 purge 摘链的字节进入 GarbageList。若“连续空闲不足、空闲加 garbage 足够”，B+Tree 会先重组页面再决定是否分裂，不能把“heap 与目录相遇”简单等同于立即分裂。

记录头、隐藏列与目录搜索

  • 每条记录使用简化的 8 字节头：flags 1 B、heapNo 2 B、nOwned 1 B、next offset 2 B、 record length 2 B。它表达链表与目录分组所需语义，但不兼容 COMPACT/DYNAMIC 原始记录头。
  • 聚簇叶页的普通 CONVENTIONAL 记录尾部保存 15 字节隐藏信息：8 字节 DB_TRX_ID 与 7 字节 DB_ROLL_PTR。MVCC 由此定位创建/修改事务和前一版本。
  • Page Directory slot 指向某个分组的最后一条记录。查找先对 slot 二分，再沿 next_record 扫描该小组；插入、删除会维护 nOwned 与必要的 slot 分裂/合并。
  • 打开任何已有 INDEX 页都会执行物理结构校验；拿到 schema 的入口还会验证字段解码、key 顺序、 hidden tail 与页面级不变量。损坏不是“尽量读”，而是 fail-closed。

字段层支持 28 种 TypeId、字符集/排序规则、日期时间等编解码。 超过页内阈值的 TEXT/BLOB/JSON 等 overflow-capable 值可放入 BLOB 链，记录只保留外部引用。业务 LOB 页面与 Undo 大载荷使用的 UNDO_PAYLOAD 页面彼此独立，不能混为同一种外部页。

06B+Tree 访问与结构修改

生产实现使用 SplitCapableBTreeIndexService， 支持任意高度、叶/非叶分段、分裂、重分布、合并与根收缩。根页 PageId 保持稳定： 根分裂时旧根内容下沉并提高根 level，根收缩时把唯一子节点内容搬回稳定根页。

点查：S-crabbing

从根到叶逐层取得子页 S latch，确认子页已稳定后释放父页；不把整条路径长期锁住。

范围：兄弟页 hand-over-hand

沿叶链移动时先固定并锁住下一页，再释放当前页，避免页在交接窗口失去保护。

写入：乐观后悲观

多层树的无分裂乐观插入只在叶页持 X；溢出后重启并保留安全祖先，必要时执行级联分裂。

为什么事务锁等待要重新定位记录

当前读使用两个短 MTR：第一次定位记录后释放 page latch、cursor 和 buffer fix，再进入可能阻塞的 LockManager；拿到行锁后开启第二个短 MTR，通过 B+Tree 重新定位并校验记录。 这样既避免“持页闩等待事务锁”的锁序反转，也承认等待期间页面可能发生分裂、合并或记录移动。

分裂前会先预留最坏情况下需要的空间，避免修改一半后才发现无法分配页面。高度至少为 2 时， 悲观写路径先在根上取得 SX；如果最终确实要改根，再以 X 模式重启，不做危险的 latch upgrade。 当前实现已经使用 S/SX/X page latch 的 crabbing / latch coupling，但没有 B-link sibling high-key， 也没有基于页版本校验的 OLC 重定位协议。

07一行数据如何完成 INSERT / UPDATE / DELETE

表级写入通过 TableDmlService 编排，而不是让 SQL 层分别操作聚簇与二级索引。数据库事务是跨多次 MTR 的 Unit of Work： 每个索引通常由独立短 MTR 修改；外层 SQL storage gateway 在调用前建立 statement guard， 失败时显式按 Undo 边界反向补偿，成功时才关闭该 guard。

INSERT：先冻结逻辑计划，再逐索引落盘

  1. 分配写事务身份并形成当前读请求。事务 ID 可延迟到第一次写时才分配； 聚簇 key 成为后续行锁、唯一性判断与重定位的稳定逻辑标识。
  2. 先完成全部发布检查。聚簇索引检查唯一性；每个不含 NULL 的二级逻辑键先取得事务锁， 再检查发布前态，其中不含 NULL 的唯一二级索引扫描 logical prefix，其余路径检查完整 physical identity。随后冻结 “二级逻辑键 + 完整聚簇键后缀”与对应 Undo mutation。
  3. 取得 DML/Purge 行级 guard。防止前台写与 purge 对同一逻辑行并发做不相容的物理处理。
  4. 先写聚簇记录。第一个短 MTR 同时落聚簇页变化并建立逻辑 Undo 锚点； 记录携带 DB_TRX_ID 与 DB_ROLL_PTR。
  5. 逐个写二级索引。每个 secondary 使用独立短 MTR，索引项包含二级逻辑 key、 完整聚簇 key 后缀。全部成功后，外层调用方才把 statement guard 作为成功边界关闭。

UPDATE 与 DELETE 的差异

操作聚簇记录二级索引当前限制/语义

08Undo、MVCC 与 Purge

当前事务最多同时拥有两条 Undo log：INSERT 与 UPDATE 分离， 二者共享事务级单调递增的 undoNo。回滚段 Page 3 的 v4 header 是槽位、 history、free list 与两类 cache 的权威持久状态；Undo 页使用 v3 格式，首个页含 log header， 后继页保留并清零 first-page 专属 header 区，同时复制 Undo kind；所有页的 record area 都从 offset 136 开始。

Undo 类型记录什么提交后的去向主要消费者
INSERT Undo 撤销本事务新插入记录所需信息 提交后无需支持旧版本读取，可直接进入 cache/free/drop 流程 statement rollback / transaction rollback
UPDATE Undo 更新或删除前的旧值、旧隐藏列与索引补偿信息 提交后进入 history，直到没有 ReadView 可能再访问 MVCC、rollback、purge

一致性读如何沿版本链回溯

  1. 先读当前聚簇版本。记录中的 DB_TRX_ID 表明该版本由哪个事务产生。
  2. ReadView 判断可见性。REPEATABLE READ 在事务内复用首个 ReadView；READ COMMITTED 每次一致性读新建，并由语句边界负责关闭。
  3. 不可见则跟随 RollPointer。DB_ROLL_PTR 定位 Undo record，重建上一版本。
  4. 重复直到可见或链结束。链损坏、指针越界或 Undo 格式不匹配都会 fail-closed，而不是返回可能错误的数据。

当前 RollPointer 只编码 pageNo + offset，没有携带 space/rseg 身份。这与“单一系统 Undo 表空间、单一默认 Rollback Segment”的现状相匹配，但也是未来支持多 Undo 表空间时必须升级的格式边界。 大型 Undo payload 会写入专用 UNDO_PAYLOAD 链，避免占满普通 Undo 页。

Purge 为什么不能看到 delete-mark 就删

Purge 只从持久 history 队首按 FIFO 推进；队首提交号尚未越过全局低水位，或 creator 仍活跃时， 不能跳过它处理后继。满足边界后，再通过版本安全检查、零等待 DML row guard 和二级索引回查， 物理移除 delete-mark 的 secondary/clustered record，持久化 logical-head 进度，最后回收 Undo owner。 当前 Purge 可由后台 worker 驱动，但“Purge 自动触发 Undo tablespace truncate”尚未生产接线； truncate 恢复服务已经存在，主动调度仍属于缺口。

09Redo、WAL、Doublewrite 与 Checkpoint

每个包含持久修改且成功提交的 MTR 都先把 Redo batch 追加到 RedoLogManager 的 pending 队列， 取得连续 LSN 区间，再把 endLSN 盖到被修改页；后台 writer/flusher 随后把 batch 写入并 force 持久 Redo ring，依次推进 written/flushed 边界。生产默认使用 8 个 8 MiB 日志文件；日志按 512 B block 编码， block header 为 32 B、payload 为 472 B，末尾 8 B 保存 trailer。文件和容量均可配置，默认值不是格式上限。

Redo 同时包含通用物理记录与少量模块语义记录： PAGE_INIT、PAGE_BYTES、FSP allocate/free/metadata delta、 Undo metadata/payload、B+Tree page delta 和 transaction state delta。 因此它是“物理页修改为主、关键元数据带语义”的混合方案，不是纯逻辑 binlog，也不是对 MySQL MLOG 的复刻。

WAL 必要条件：写 dataPage 之前，必须满足 redo.flushedLSN ≥ page.pageLSN

批量刷页的真实顺序

  1. 选择候选页。Page Cleaner 从 flush list 或 LRU 形成 batch；FLUSHING 页仍保留在脏页边界中。
  2. 按 spaceId 排序取得共享 lease。阻止 drop/truncate 与当前批次交错，同时避免跨空间锁序不稳定。
  3. 复制页面快照并读取 pageLSN/dirtyVersion。慢 I/O 不持有 page latch；快照版本用于完成时复核。
  4. 等待 Redo durable。所有快照对应的 pageLSN 被 flushed LSN 覆盖后，才允许继续。
  5. 重算 checksum 并写 Doublewrite。页面按来源进入 FLUSH_LIST 或 LRU 的独立物理文件；默认每通道 1024 slots。
  6. 批量写表空间并 force。按 space 分组写入真实数据文件；全部成功后才释放 Doublewrite slot。
  7. 按版本完成 flush。只有 frame generation 与 dirtyVersion 未变化才清理脏状态；并发再修改的页继续留在 flush list。

Doublewrite 默认工作在 DETECT_AND_RECOVER 模式，也支持 OFF 与 DETECT_ONLY。Checkpoint 使用双 4 KiB slot 持久化标签；推进前先持久化事务恢复 sidecar， 再写 Redo checkpoint label，最后才允许回收旧 Redo，避免日志边界跑到事务恢复信息前面。

10崩溃后如何恢复

CrashRecoveryService 只恢复物理页与事务状态，不执行普通 SQL。恢复期间流量保持关闭；任何阶段发现不满足继续运行的不变量， 都抛出启动恢复异常并 fail-closed。

  1. TRAFFIC_CLOSED。保持存储入口关闭，防止前台访问观察到半恢复状态。
  2. DOUBLEWRITE_REPAIR。从两个通道扫描完整页副本，修复被 torn write 破坏且可判定目标身份的数据页。
  3. REDO_REPLAY。从 checkpoint 边界扫描完整 Redo batch；页类记录仅在目标 pageLSN 落后时幂等重放，事务状态 delta 则按日志顺序恢复。
  4. REDO_BOUNDARY_INSTALL。让新 LSN 从 recoveredToLsn 续写，并把 closed/written/flushed 运行时边界复位到该值；不在此改写 reclaim 边界。
  5. UNDO_TABLESPACE_RESUME。恢复被中断的 Undo tablespace truncate 状态机，使生命周期回到可解释状态。
  6. SPACE_FILE_RECONCILE。读取恢复后的 page0.currentSizeInPages，并用 ensureCapacity 幂等补齐可能缺失的文件尾容量。
  7. UNDO_ROLLBACK。恢复 committed history、裁决 PREPARED，并按 Undo 逆序撤销 crash 前仍为 ACTIVE 的事务。
  8. RESUME_PURGE。在恢复出的 committed history 上真实推进 Purge，继续清理由已提交旧版本留下的垃圾。
  9. 恢复写 force，然后 OPEN_TRAFFIC。先 flush rollback/purge 新产生的 Redo（若有），再 force 恢复直接写出的数据页，最后开放流量；force 不是独立的 RecoveryStageName。
只读诊断模式走另一条链：它扫描 Redo 与 Doublewrite slots，并只读检查其引用的候选页及启动所需元数据， 但不会全表空间逐页扫描，也不 apply、force 或写入恢复结果，最终进入 READ_ONLY_DIAGNOSTIC_OPEN。这不是“用只读方式执行一次普通恢复”。

公共 DatabaseEngine 先取得实例锁，打开 XA registry，再完成 DD catalog/manifest/control 准入并从 committed binding discovery 用户表空间；随后把 XA 决议提供器交给 StorageEngine 执行 recovery，完成 registry recovery decision 后再执行包含 SDI reconcile 的 DDL recovery， 最后才构造 Session 门面并发布完整数据库 OPEN。本文聚焦存储部分，因此没有把外层 DDL 阶段画成 CrashRecoveryService 的内部步骤。

关键结论速查

#结论为什么重要
1页大小可配置，Extent 内页数不是永远 64。分析页号、XDES 和容量时必须先看 PageSize。
2Disk/FSP 不理解 record，Record 也不直接碰文件。这是当前存储分层最核心的依赖边界。
3MTR 封闭短物理临界区与 Redo group，Transaction 负责跨 MTR 的业务原子性。一次表级 INSERT 会跨多个短 MTR；未提交 MTR 不提供页面 content undo。
4事务锁等待前必须释放 page latch/fix，成功后重新 B+Tree 定位。避免锁序反转，也防止使用已经移动的物理位置。
5Page Directory 是二分入口，不是记录本体。实际查找是“slot 二分 + 分组链扫”。
6二级键 UPDATE 采用 new-before-old。语句中途失败时仍能由 Undo 找回可达入口。
7INSERT Undo 与 UPDATE Undo 的提交后命运不同。只有 UPDATE Undo 必须为历史一致性读长期保留。
8数据页写盘前必须满足 Redo WAL。否则崩溃后可能出现无法由日志解释的新数据页。
9Doublewrite、Redo、Undo 解决三种不同问题。分别处理 torn page、物理状态重放、未提交事务与旧版本回溯。
10Checkpoint 是安全交集，不是“当前最大 LSN”。推进过快会回收仍被脏页或事务恢复状态需要的日志。
11恢复必须按阶段串行并保持流量关闭。Doublewrite 修复、Redo、Undo、Purge 的顺序具有恢复语义。
12当前实现借鉴 InnoDB 思想，但不承诺 MySQL 二进制兼容。阅读布局数字时应以本项目源码为准，不能套用 MySQL 常量。
