01

总览：锁与 MVCC 的分工

第四篇讲清了 MVCC：普通 SELECT（快照读）靠版本链 + ReadView 读历史快照，完全不加锁。但并发正确性不可能全靠 MVCC——一旦涉及写，或显式的当前读，多个事务对同一数据的操作就必须被"串行化"，这就是锁的职责。

先把分工钉死，这是理解全篇的地基：

三个问题

本篇所有的锁，作用对象都是当前读这条路径。把锁系统讲透，本质是回答三个问题：
① 有哪些锁？（类型与粒度，第 2–9 章）
② 谁和谁冲突？（兼容矩阵，第 5、8 章）
③ 一条 SQL 到底锁住了什么、什么时候锁？（加锁规则与时机，第 10–14 章），以及它出问题时怎么办（死锁与监控，第 17–18 章）。

02

两个维度：粒度 × 模式

InnoDB 的锁可以从两个正交维度理解，后面所有锁类型都是它们的组合：

  • 粒度（锁多大）：表级锁 锁整张表（意向锁、表 S/X、AUTO-INC、元数据锁 MDL）；行级锁 只锁某些索引记录或间隙（记录锁、间隙锁、临键锁、插入意向锁）。InnoDB 以行级锁为主——这正是它相比只有表锁的 MyISAM 能高并发的根本。
  • 模式（锁的强度/意图）：基本模式是共享锁 S（读，可共存）与 排他锁 X（写，独占）；表级还有意向锁 IS/IX 做协调。
03

共享锁 S 与排他锁 X

S/X 是锁的两种基本模式，在行级和表级都存在，对应"读"和"写"两种意图：

  • 共享锁 S（Shared）—— 读锁。多个事务可同时持有同一对象的 S 锁，彼此不冲突。手动加：SELECT ... FOR SHARE（旧语法 LOCK IN SHARE MODE）。
  • 排他锁 X（Exclusive）—— 写锁。一个对象同时只能有一个事务持 X 锁，与 S、X 都冲突。UPDATE / DELETE 以及 SELECT ... FOR UPDATE 都加 X。

行级 S/X 的兼容关系是一切的起点——"读读相容、读写互斥、写写互斥"：

请求 ↓ / 持有 →SX
S兼容 ✓冲突 ✗
X冲突 ✗冲突 ✗
易忽略

"行级 S/X"和"表级 S/X"是两套东西，作用对象不同但兼容逻辑一样。日常 DML 几乎只用行级 S/X；表级 S/X 出现在 LOCK TABLES ... READ/WRITE 或某些 DDL 场景。第 5 章那张 4×4 矩阵里的 S/X 指的是表级，别和这张 2×2 的行级表混淆。

04

意向锁 IS / IX

意向锁是表级锁，但它存在的唯一目的是为行级锁服务。先看它要解决的问题：

问题

事务 A 给大表里的某一行加了 X 锁。此时事务 B 想给整张表加表级 X 锁（如某些 DDL）。B 必须确认"表里有没有任何一行正被锁着"。若逐行扫描检查，百万行的表会慢到不可接受。

解决办法是意向锁：事务在给某行加行锁之前，先在表级打一个意向标记，声明"我打算在这张表里加某种行锁"。这样表锁请求只看意向锁即可 O(1) 判断冲突。

  • IS（意向共享）—— 打算给某些行加 S 锁前，先加表级 IS。
  • IX（意向排他）—— 打算给某些行加 X 锁前，先加表级 IX。所有 DML 写都会先拿 IX。
两条规则

① 意向锁之间永远相容——IS、IX 彼此不冲突，它们只是"意图声明"，真正的冲突发生在行锁层面。
② 意向锁只与表级 S/X 冲突——IX 与表级 S/X 冲突，IS 与表级 X 冲突。意向锁由 InnoDB 自动添加，无法手动控制。

05

锁兼容矩阵

把 IS、IX、S、X 放一起，得到表级锁的完整兼容矩阵。判断"一个锁请求能否立即获得"——只要请求的锁与已持有的任一锁冲突，就得等待：

请求 ↓ / 持有 →ISIXSX
IS✓✓✓✗
IX✓✓✗✗
S✓✗✓✗
X✗✗✗✗
怎么记

不用死背 16 格，抓三条：
① 左上 2×2（IS/IX 互相）全相容。
② X 一行一列全 ✗——排他与谁都冲突。
③ 只剩 IX↔S 冲突——"有人要改某行(IX)"与"有人要锁全表读(S)"无法共存。
重要：这张矩阵管的是表级对象。行级锁还有 Gap/Next-Key/插入意向，它们的兼容规则更特殊，见第 8、9 章——尤其"间隙锁之间永远兼容"这条，不在此表内。

06

锁的内存结构与存储

很多人以为"锁住一行"是在那行数据上写个标记。其实不是——InnoDB 的锁是独立的内存对象，集中存放在一个叫 lock_sys 的全局结构里，与数据页本身分离。理解这点，才能解释"为什么锁那么多行内存却没爆""一个 lock 对象怎么表示一批行"。

一个锁对象（lock_t）描述什么

InnoDB 不会为每一把行锁都建一个对象。一个 lock_t 对象描述的是：同一个事务、同一个页面、同一种锁模式的一组行锁。对象里挂一个位图（bitmap），页内每条记录对应一个 bit（按 heap_no 编号），bit 置 1 表示这条记录被这把锁锁住。

三个推论

① 锁很省内存：锁住同一页的 100 行同模式锁，可能只需一个对象 + 100 个 bit，而不是 100 个对象。
② 锁与数据解耦：数据页里没有"这行被锁了"的标记（隐式锁是例外，见第 10 章），锁信息全在 lock_sys。
③ heap_no=0 是 supremum 伪记录：每页有一条虚拟的"最大记录"哨兵，用来锁住"最后一条记录到 +∞"的间隙（第 7、8 章会用到它）。

07

行锁三形态与 supremum

InnoDB 的"行锁"不止锁行——它有三种形态，区别在锁住的"范围"不同。把索引想象成一条数轴：记录是点，记录之间是"间隙"，数轴两端有 -∞ 和 +∞ 哨兵。

① 记录锁 Record Lock —— 锁一个点

精确锁住索引上某条已存在的记录（不含其前后间隙）。监控里标记为 X,REC_NOT_GAP 或 S,REC_NOT_GAP。

② 间隙锁 Gap Lock —— 锁一段开区间

锁住两条记录之间的间隙（不含记录本身），目的只有一个：阻止别的事务往这个间隙 INSERT 新记录。它不锁已有记录、不阻止读和改，纯为防幻读而生。标记为 X,GAP。

③ 临键锁 Next-Key Lock —— 点 + 它左边的间隙

= 记录锁 + 间隙锁，锁住左开右闭区间 (前一记录, 当前记录]。这是 RR 隔离级别下行锁的默认形态。监控里就标记为 X（无后缀即代表 Next-Key）。

临键锁的两种退化（减小锁范围、提升并发）

退化为触发条件原因
记录锁唯一索引等值查询命中唯一键不可能再插重复值，间隙无需锁
间隙锁等值查询未命中任何记录没有记录可锁，只剩间隙锁防"插入这个不存在的值"
supremum 的作用

范围查询若扫到表的"末尾"，最后会对 (最后一条记录, +∞) 这个间隙加锁，物理上就是锁住 supremum 伪记录的临键锁。这保证了像 WHERE id > 30 这种开放区间也能防住"在末尾插入新行"的幻读。supremum 永远是这页"最大"的存在，锁它就等于锁住通向无穷大的那段间隙。

08

间隙锁的特殊性

间隙锁是 InnoDB 锁系统里最反直觉的一种，它不遵守普通 S/X 的兼容规则。单独成章讲清，否则后面的死锁分析会看不懂。

特性一：间隙锁之间永远兼容（哪怕一个 S 一个 X）

两个事务可以对同一个间隙同时持有间隙锁，无论是 S 型还是 X 型，都不冲突。原因很简单：间隙锁的目的只是"不让别人插入"，多个事务同时表达"这里别插入"毫不矛盾。

特性二：间隙锁只阻塞"插入"，不阻塞"读 / 改"

间隙是空的（没有记录），所以间隙锁不会挡住对已有记录的读或改，它唯一会挡住的是 INSERT（准确说是挡住别人申请插入意向锁，见第 9 章）。

特性三：间隙锁不分方向，"S/X"对它几乎无意义

由于特性一，间隙锁标记上虽然也带 S/X，但因为间隙锁彼此从不冲突，这个 S/X 标记实际上不影响它和其它间隙锁的关系——只在它"是否阻塞插入意向锁"上起作用。

请求 ↓ / 持有 →间隙锁(Gap)插入意向锁(II)记录锁(Record)
间隙锁兼容兼容兼容
插入意向锁阻塞 ✗兼容兼容
记录锁兼容兼容看 S/X
关键非对称

看表里这一格："已持有间隙锁"不阻塞"请求插入意向锁"，但"已持有间隙锁"会阻塞"后来的插入意向锁请求"——准确表述是：当你想插入（申请插入意向锁）时，如果目标间隙上已有别人的间隙锁，你会被阻塞。方向是单向的：间隙锁挡插入意向锁，反之不挡。这条单向阻塞，就是 RR 防幻读的物理落点，也是大量"插入被莫名卡住""间隙锁死锁"的根源。

提示

间隙锁只在 RR 及以上隔离级别启用。RC 隔离级别几乎不用间隙锁（仅外键检查、唯一键冲突等极少数场景），所以 RC 下并发插入冲突少、但有幻读。这也是不少高并发系统选择 RC 的原因之一。

09

插入意向锁 Insert Intention Lock

名字带"意向"，但它是行级锁，是间隙锁的一个特殊变种，专为 INSERT 设计，解决一个矛盾：间隙锁会禁止插入，但两个事务往同一间隙的不同位置插入，本不该互相阻塞。

当事务要在某间隙插入一行时，会先申请该位置的插入意向锁（II），表达"我打算插在这个间隙里的某个具体位置"。它的兼容规则（承接第 8 章）：

  • 插入意向锁之间互相兼容——多个事务往同一间隙不同位置插入，不阻塞（只要键值不同）。这是它存在的意义：放行并发插入。
  • 但它会被已存在的间隙锁 / 临键锁阻塞——若别的事务已用间隙锁锁住该间隙（为防幻读），插入意向锁就得等待。这正是 Next-Key Lock 防住幻读的"最后一公里"。
闭合第四篇伏笔

RR 当前读不幻读的物理机制是：范围查询先用临键锁/间隙锁锁住区间，后来者 INSERT 申请插入意向锁时被这个间隙锁挡住，于是插不进来。"间隙锁防插入"落到锁层面就是"间隙锁 ⊥ 插入意向锁"这一条单向冲突规则。

10

隐式锁与显式锁

前面讲的锁都是显式锁——在 lock_sys 里实实在在建了 lock_t 对象。但 InnoDB 还有一种隐式锁（Implicit Lock），它不在 lock_sys 里建对象，是一种"省钱"的延迟策略，新手几乎不知道它的存在，却是理解 INSERT 加锁的关键。

隐式锁是什么

当一个事务 INSERT 一条新记录时，InnoDB 默认不为它建立显式锁。但这条新记录的聚簇索引里写着 DB_TRX_ID（最近修改它的事务，正是插入者）。这个 DB_TRX_ID 本身就构成了一把"隐式的 X 锁"：

  • 别的事务想访问这条记录时，会去检查记录上的 DB_TRX_ID 是否属于一个仍然活跃（未提交）的事务。
  • 如果是 → 说明这条记录正"隐式地"被那个事务锁着。此时由访问者"帮"持有者把隐式锁转成显式锁（在 lock_sys 里补建一个 X 锁对象记到持有者名下），然后访问者再正常去排队等待这把显式锁。
为什么这样设计

绝大多数 INSERT 插入后，短时间内没有别的事务来争这条新行。为这种"大概率无竞争"的情况预先建锁对象是浪费。隐式锁把建锁的成本推迟到真正发生冲突的那一刻，由冲突方代为补建。这是典型的"乐观假设 + 惰性求值"，让高频 INSERT 的锁开销趋近于零。隐式锁只存在于聚簇索引（靠 DB_TRX_ID）；二级索引没有该字段，处理方式略有不同，必要时会去聚簇索引回查。

11

唯一键冲突与 INSERT 加锁

INSERT 看似简单，加锁却最容易出意外，尤其在有唯一索引时。它也是生产中死锁的头号来源，必须单独讲。

正常 INSERT 的加锁

一次普通插入大致做两件加锁的事：① 对要插入的间隙申请插入意向锁（若间隙被锁则等待）；② 插入成功后，对新记录持有隐式锁（见上一章），无人争则不显化。

遇到唯一键冲突时的加锁（重点）

当插入的值违反唯一约束（已存在相同的唯一键），InnoDB 不是简单报错了事——它需要先读到那条已存在的冲突记录来确认冲突，这个"读"会给冲突记录加锁：

  • 普通 INSERT 撞唯一键 → 对已存在的冲突记录加一个 S 型 next-key 锁（RR 下）。这是为了防止它在你这次插入失败的窗口里被删除/修改。
  • INSERT ... ON DUPLICATE KEY UPDATE 撞键 → 加 X 锁（因为接下来要改它）。
经典三事务死锁

这是面试与故障复盘的常客：三个事务并发插入同一个唯一键值。
① T1、T2、T3 都 INSERT 同一个唯一值 → T1 先成功插入（持隐式锁），T2、T3 都撞键，于是各自申请那条记录的 S 锁并等待 T1。
② T1 此时回滚 → 它的隐式锁释放，T2、T3 都拿到 S 锁。
③ T2、T3 现在都想真正插入，各自申请插入意向 X 锁，但都被对方持有的 S 锁挡住 → 互相等待，死锁。
根因：S 锁可共存（多人能同时持），但接下来要升级到的插入需要排他，于是形成"都持 S、都等 X"的环。

规避

① 业务层对"插入同一唯一键"的并发做去重/串行（如分布式锁、消息幂等）。② 用 INSERT ... ON DUPLICATE KEY UPDATE 或先 SELECT ... FOR UPDATE 探测，把竞争收敛到一把 X 锁上，避免"多人持 S 再抢 X"。③ 批量插入按唯一键排序后再插，减少交叉等待。

12

锁加在索引上

一个极重要、却常被忽略的原理：InnoDB 的行锁锁的是索引记录，不是"数据行"本身。这能解释许多反直觉的加锁现象。

最危险后果

若查询条件没用到任何索引，InnoDB 无法精确定位要锁的索引记录，只能在聚簇索引上全表扫描，并对扫到的每条记录加临键锁——实际效果近似锁全表，其它写操作几乎全被阻塞。所以 UPDATE t SET ... WHERE 无索引列 = x 在并发下是灾难。加锁查询务必走索引，是性能红线。

聚簇索引与二级索引都会被锁

通过二级索引定位加锁时，InnoDB 不仅锁命中的二级索引记录，还会回表锁住对应的聚簇索引（主键）记录。这是为防别的事务绕过二级索引、直接用主键改这行。

两个推论

① 同一行用不同条件可能锁不同范围。WHERE id=2 走主键，只锁聚簇索引那条；WHERE name='Bob' 走二级索引，锁二级索引 + 聚簇索引两处，还可能因二级索引值可重复而多加间隙锁。
② 覆盖索引不改变加锁对象。即便查询被二级索引"覆盖"无需回表读数据，加锁（当前读）时该回表锁聚簇索引记录还是会回表。覆盖索引省的是读的回表，不是锁的回表。

13

半一致性读 Semi-consistent Read

这是 RC 隔离级别下 UPDATE 的一项专门优化，名字冷僻但很实用，能显著减少 UPDATE 之间的锁等待。

问题：UPDATE 必须当前读，会被锁挡住

UPDATE t SET ... WHERE c > 100 是当前读，要扫描并锁住所有 c>100 的行。但若某行当前正被别的事务锁着、且改完后其实不满足 c>100，本不该锁它、更不该为它干等——可普通流程会先去申请锁、被阻塞。

半一致性读怎么优化

在 RC 下，InnoDB 对 UPDATE 扫描时使用半一致性读：遇到一条已被其它事务加锁的记录，不立刻等锁，而是先读它的最新已提交版本（走一次 MVCC 快照）来判断 WHERE 条件：

  • 若该版本不满足 WHERE 条件 → 这行本来就不用改，直接跳过，不申请锁、不等待。
  • 若该版本满足条件 → 才老老实实去申请 X 锁并等待。
效果与边界

效果：大幅减少 UPDATE 因"扫到无关的被锁行"而产生的无谓等待，提升并发更新吞吐。
边界：它只在 RC（及 READ UNCOMMITTED）下生效，RR 下不启用——因为 RR 要靠对扫描路径全程加锁来防幻读，跳过加锁会破坏可重复读语义。这又一次印证：隔离级别越高，能做的并发优化越少。"半一致"三个字，指的就是"判断条件时用快照（一致性读），真要改时再用当前读"这种半快照半锁定的混合姿态。

14

实战：RR 加锁逐条分析与 RC 对比

把前面的规则用到具体 SQL。设表 t(id PK, c KEY, d)，数据 (id,c) = (5,10)、(10,20)、(15,30)，隔离级别先看 RR、FOR UPDATE。

两点落地

① RC 释放不匹配行的锁。RR 扫描时把路径上的行都临键锁住且到事务结束才放（为防幻读）；RC 则对 WHERE 不匹配的行加锁后立刻释放，加上无间隙锁，并发明显更好——代价是放弃了防幻读。
② 加锁形态由"隔离级别 × 索引类型 × 是否命中"三者共同决定。分析任何一条加锁 SQL，都按这三个维度套：先看隔离级别决定要不要间隙锁，再看走哪个索引决定锁几处，最后看命中与否决定退化形态。

AUTO-INC 锁与自增

自增主键（AUTO_INCREMENT）背后有一套独立的锁机制，很多人不知道它的存在，但它直接影响批量插入的并发与主从复制的安全。

为什么自增需要锁

多个事务并发插入时，必须保证每个拿到的自增值唯一且不重复。最朴素的做法是用一把表级 AUTO-INC 锁：插入语句开始时拿锁、语句结束才释放，使同一时刻只有一条插入语句在分配自增值。安全，但并发插入被串行化了。

三种模式：innodb_autoinc_lock_mode

模式名称行为并发 / 连续性
0traditional 传统所有插入都持表级 AUTO-INC 锁到语句结束并发差，值严格连续
1consecutive 连续能预知行数的简单插入用轻量互斥（分配完即放）；行数不可知的（INSERT...SELECT）退回表锁折中（8.0 前默认）
2interleaved 交错一律用轻量互斥，多语句可交错分配自增值并发最好，值可能不连续
8.0 的默认变化与陷阱

MySQL 8.0 默认 innodb_autoinc_lock_mode = 2（交错模式），并发最好。但模式 2 有两个要注意的点：
① 自增值可能不连续。多事务交错分配 + 回滚会"用掉但不回收"中间的值，主键出现空洞是正常现象，不要依赖"自增值连续"。
② 与 binlog 格式相关。模式 2 下，基于语句的复制（SBR / statement 格式 binlog）可能导致主从自增值不一致，因此模式 2 要求配合行级 binlog（ROW 格式）。这也是 8.0 把默认 binlog 改成 ROW、把锁模式改成 2 的连带原因。

要点

AUTO-INC 锁是一种语句级而非事务级的锁——它在语句结束（而非事务提交）就释放（模式 1/2 的轻量分配更是分配完立即释放）。所以它造成的阻塞通常很短，但在大批量 INSERT...SELECT 且模式为 0/1 时，可能整条语句期间锁住自增、拖慢其它插入，是排查"批量导入卡住并发写"的一个方向。

16

元数据锁 MDL（Metadata Lock）

MDL 是 MySQL Server 层（不是 InnoDB 引擎层）的表级锁，保护的是表的结构（元数据）。它是生产环境"线上改表把整张表搞挂"事故的头号元凶，务必理解。

MDL 解决什么

设想事务 A 正在查询表 t，同时事务 B 执行 ALTER TABLE t DROP COLUMN x。如果不加保护，A 读到一半表结构变了，会直接出错。MDL 的规则是：

  • 任何 DML（增删改查）都会自动给表加一把 MDL 读锁（共享），从语句开始一直持有到整个事务提交。多个 DML 的 MDL 读锁互相兼容。
  • 任何 DDL（改表结构）要加 MDL 写锁（排他），与所有 MDL 读锁、写锁都冲突。
最致命的连锁

图里的链条是真实事故模型：① 一个忘记提交的长事务持着 MDL 读锁；② 你执行 ALTER，DDL 申请 MDL 写锁被它挡住，开始排队；③ 一旦写锁在排队，所有后续新查询的 MDL 读锁都要排在它后面——于是这张表对所有人"假死"，连最简单的 SELECT 都被阻塞。很多"改个表全站雪崩"的事故就是这么来的。

规避

① 改表前先查 SELECT * FROM performance_schema.metadata_locks 或 information_schema.innodb_trx，确认没有长事务占着这张表。② 给 DDL 设置 lock_wait_timeout，让它在拿不到 MDL 时快速失败而不是无限等待、连累后续查询。③ 用支持 ALGORITHM=INSTANT/INPLACE 的在线 DDL，或 gh-ost / pt-osc 这类工具做无锁变更。④ 应用层杜绝"开了事务做完查询不及时提交"。

17

死锁：成因、案例与检测

有锁就有死锁：两个或多个事务各持对方想要的锁，互相等待，谁也无法前进。死锁的四个必要条件（互斥、持有并等待、不可剥夺、循环等待）缺一不可，破坏任意一条即可避免——InnoDB 主要从"循环等待"下手检测。

等待图与检测原理

InnoDB 内部维护一张等待图（wait-for graph）：节点是事务，有向边 A→B 表示"A 在等 B 持有的锁"。图中只要出现环，就是死锁。每次有锁等待发生，检测器就增量检查是否成环，发现即处理。

三个真实死锁案例

案例一 · 加锁顺序相反（最经典）

-- A: 锁 id=1 再锁 id=2 ；B: 锁 id=2 再锁 id=1
A: UPDATE t SET d=d+1 WHERE id=1;   B: UPDATE t SET d=d+1 WHERE id=2;
A: UPDATE t SET d=d+1 WHERE id=2;   B: UPDATE t SET d=d+1 WHERE id=1;  -- 死锁

根因：两个事务访问顺序相反。修复：所有事务统一按主键升序访问行。

案例二 · 唯一键并发插入（见第 11 章）

多个事务插同一唯一值，都持冲突记录的 S 锁，再都申请插入意向 X 锁 → "都持 S、都等 X"成环。修复：业务去重 / 串行化该键的写。

案例三 · 间隙锁交叉（RR 特有）

-- 表中无 id=5 的行，A、B 都先探测再插入
A: SELECT * FROM t WHERE id=5 FOR UPDATE;  -- 拿到间隙锁(范围)
B: SELECT * FROM t WHERE id=5 FOR UPDATE;  -- 间隙锁互相兼容，也拿到
A: INSERT INTO t VALUES(5,...);   -- 申请插入意向锁，被 B 的间隙锁挡住
B: INSERT INTO t VALUES(5,...);   -- 同样被 A 挡住 → 死锁

根因：第 8 章说过"间隙锁互相兼容、但都挡插入意向锁"。两个事务都先拿到兼容的间隙锁，再都要插入 → 互挡。修复：RC 隔离级别（无间隙锁），或用 INSERT ... ON DUPLICATE KEY UPDATE 收敛到一把锁。

处理：回滚牺牲者

检测到死锁后，InnoDB 主动选一个事务回滚（牺牲者），让另一个继续。选择倾向"改动数据少（undo/持锁少）就回滚谁"，回滚代价低。被回滚方收到 ERROR 1213 (40001): Deadlock found，应用层应捕获并重试整个事务。

减少死锁的五条

① 固定加锁顺序（按主键升序）——最有效，从源头不让环形成。
② 缩短事务——持锁时间越短，相遇概率越低。
③ 加锁走索引、缩小范围——避免无索引扫全表的大范围锁。
④ 一次性锁定所需全部资源，减少"持有再申请"的链条。
⑤ 必要时降级到 RC，消除间隙锁这一大类死锁。
超大并发下检测本身有开销，可用 innodb_deadlock_detect=OFF 关检测、靠 innodb_lock_wait_timeout 超时兜底，但需权衡响应延迟。

锁监控实操与系列总结

理论之外，得会用工具看"现在谁锁了什么、谁在等谁"。MySQL 8.0 的 performance_schema 提供了直观的锁视图。

三张关键视图

  • performance_schema.data_locks —— 当前所有持有/等待中的锁，一行一把锁。
  • performance_schema.data_lock_waits —— 谁在等谁的配对（阻塞锁 ↔ 等待锁），排查锁等待的首选。
  • information_schema.innodb_trx —— 活跃事务列表，含开始时间（找长事务）、状态、正在等的锁。

data_locks 关键字段怎么读

字段含义典型值 / 解读
LOCK_TYPE粒度TABLE（表锁/意向锁）/ RECORD（行锁）
LOCK_MODE模式 + 形态X=X临键；X,REC_NOT_GAP=记录锁；X,GAP=间隙锁；X,INSERT_INTENTION=插入意向；IX=意向排他

实操套路

排查"SQL 卡住"三步：① SELECT * FROM data_lock_waits 找到"等待者 ↔ 阻塞者"两个 trx_id；② 拿阻塞者的 trx_id 去 innodb_trx 看它在干什么、跑了多久（多半是长事务）；③ 必要时 KILL 阻塞者的连接。复盘死锁则看 SHOW ENGINE INNODB STATUS 的 LATEST DETECTED DEADLOCK 段，里面有双方的 SQL 与持锁/等锁详情。

InnoDB 系列 · 五篇成体系

① 存储结构→② Buffer Pool→③ 故障恢复→④ 事务与 MVCC→⑤ 锁系统

数据在哪 → 怎么用 → 出事怎么救 → 并发读怎么控（MVCC）→ 并发写怎么控（锁）。第四、五篇合起来才是 InnoDB 隔离性的完整答案：读靠 MVCC 不阻塞，写靠锁保正确。
