01

总览：两份日志与一致性难题

一次 UPDATE 提交时，MySQL 其实要写两份完全独立的日志，它们由系统里两个不同的"层"分别管理：

  • redo log —— 由 InnoDB 存储引擎层负责。物理日志，保证崩溃后数据不丢（持久性 D），是第三篇《故障恢复》的主角。
  • binlog —— 由 MySQL Server 层负责。逻辑日志，记录"做了什么操作"，用于主从复制和按时间点恢复。所有存储引擎共用它。

问题来了：这两份日志各写各的，但一个事务的"提交"必须让两者要么都生效、要么都不生效。否则一旦在"只写了一份"的瞬间崩溃，主库和从库（或本地与备份）就会数据不一致。

本篇要回答的

① redo 和 binlog 各是什么、为什么需要两份？（第 2–4 章，详解与对比）
② 不做特殊处理会出什么乱子？（第 5 章，两种崩溃不一致——这是 2PC 的动机）
③ 两阶段提交怎么解决、崩溃后怎么判断？（第 6–9 章，核心机制）
④ 这么做会不会很慢、参数怎么配？（第 10–11 章，组提交与双 1）。
本篇是第三篇（恢复，redo/undo）与第四篇（事务，提交语义）之间缺失的那块拼图。

02

redo log 详解

第三篇从"恢复"的角度讲过 redo，这里从"它本身是什么"的角度讲全。

本质：物理日志 + WAL

redo log 记录的是"对某个数据页的某个位置做了什么物理修改"（如"第 5 号表空间第 12 页偏移 100 处写入值 X"）。它遵循 WAL（Write-Ahead Logging，预写日志）原则：修改数据页之前，先把 redo 写下来。这样即使脏页还没刷盘就崩溃，重启后用 redo 把修改"重做"一遍即可，数据不丢。

循环写与两个指针

redo log 文件大小固定（由多个文件组成一个环），循环写入——写到末尾就回到开头覆盖。靠两个指针管理这个环：

  • write pos —— 当前写到的位置，随写入向前推进。
  • checkpoint —— 已经安全刷盘、可被覆盖的位置。它和 write pos 之间是"已写但对应脏页还没落盘"的 redo，不能覆盖。

两指针之间的空白就是可写空间。一旦 write pos 追上 checkpoint（环写满），就必须强制推进 checkpoint（刷脏页），此时写入会短暂停顿——这是 redo 写满导致抖动的根源。

写入链路：redo log buffer → 文件

写 redo 不是直接落盘。先进内存的 redo log buffer，再在某些时机刷到 OS、再 fsync 到磁盘。刷盘时机由 innodb_flush_log_at_trx_commit 控制，这是最重要的持久性参数：

取值提交时行为崩溃后果
0只写 redo buffer，由后台线程每秒刷盘一次mysqld 崩溃即可能丢最近 1 秒事务
1（默认）每次提交都写文件并 fsync最安全，提交即落盘，崩溃不丢
2每次提交写到 OS 缓存，但 fsync 每秒一次mysqld 崩溃不丢（OS 还在），但主机/OS 崩溃可能丢 1 秒
LSN：贯穿一切的序号

承接第三篇：每条 redo 都有一个全局递增的 LSN（Log Sequence Number），相当于"日志的字节偏移时间轴"。数据页头也记着 page_lsn（这页最后被改到哪条 redo）。恢复时比较"redo 的 LSN"和"页的 page_lsn"，只重做 page_lsn 更小的部分——LSN 让 redo 重放天然幂等，重做多少次结果都一样。这个性质在第 8 章崩溃恢复里至关重要。

03

binlog 详解

binlog（二进制日志）是 MySQL Server 层的日志，与具体存储引擎无关。它记录的不是"物理页改动"，而是"逻辑上发生了什么变更"，主要服务于两件大事：主从复制和按时间点恢复（PITR）。

本质：逻辑日志 + 追加写

与 redo 的"循环覆盖"不同，binlog 是追加写：写满一个文件就新建下一个（mysql-bin.000001、000002…），历史永不覆盖。正因如此它能用来"回放从某个时间点起的所有变更"，做数据恢复和搭建从库。这也意味着 binlog 会持续占用磁盘，需要靠 expire_logs_days / binlog_expire_logs_seconds 定期清理。

三种格式（binlog_format）

格式记录内容优点 / 缺点
STATEMENT记录原始 SQL 语句日志小；但 NOW()、UUID()、含 LIMIT 无序等不确定语句主从执行结果可能不一致
ROW（8.0 默认）记录每一行改动前后的值精确、复制安全；但批量更新时日志量大
MIXED一般用 STATEMENT，遇不确定语句自动切 ROW折中

写入链路：binlog cache → 文件

事务执行过程中，binlog 事件先写进每个线程私有的 binlog cache（因为一个事务的 binlog 必须连续、完整，不能和别的事务交错）。事务提交时，再把 cache 里的内容一次性 write 到 binlog 文件（OS 缓存），并按 sync_binlog 决定是否 fsync 落盘：

sync_binlog行为后果
0只 write 到 OS，由 OS 决定何时刷盘最快，但崩溃可能丢多个事务的 binlog
1（推荐）每次提交都 fsync最安全，binlog 提交即落盘
N（>1）每 N 次提交 fsync 一次性能与安全折中，崩溃可能丢最近 N 个
为什么有了 redo 还要 binlog

常见疑问：redo 也能恢复，为什么还要 binlog？因为两者用途根本不同：
redo 是"循环覆盖"的，只保留"最近未落盘"的那一小段，只能把崩溃前一刻的数据补齐，无法回放历史，更不是为复制设计的。
binlog 是"全量追加"的逻辑日志，能把"从上次全量备份到此刻的所有变更"完整回放，所以它才是主从复制和误删数据后恢复到某时刻的依据。
一句话：redo 保证"崩溃不丢"，binlog 保证"可复制、可追溯"。两者缺一不可，这也正是要让它们保持一致的原因。

04

redo vs binlog 全面对比

把两份日志并排对照，差异一目了然——正是这些差异决定了它们各自的用途，也决定了为什么提交时要协调两者：

维度redo logbinlog
所属层InnoDB 引擎层（引擎专属）Server 层（所有引擎共用）
日志类型物理日志：页的物理改动逻辑日志：行变更 / SQL
写入方式循环写，覆盖旧内容追加写，历史永久保留
核心用途崩溃恢复，保证持久性主从复制、时间点恢复、归档
记录范围只记"已写未落盘"的近期改动记录全部已提交事务的变更
幂等性靠 LSN 天然幂等靠事务/GTID 保证不重复执行
关键参数innodb_flush_log_at_trx_commitsync_binlog
一个易错认知

不要以为"binlog 记了，数据就一定改了"，也不要以为"redo 记了，复制就一定能跟上"。binlog 管的是"对外可见的逻辑变更（给从库和恢复用）"，redo 管的是"对内的物理持久化（给本机崩溃恢复用）"。它们是两条独立的链路，各自有各自的"提交"。如果不强制协调，完全可能出现"一份记了、另一份没记"的撕裂——下一章就专门看这种撕裂有多致命。

05

核心动机：不用 2PC 的两种崩溃不一致

这是理解两阶段提交的关键。假设提交时不做任何协调，只是简单地"先写一份、再写另一份"。无论哪个先写，都存在一个"写完第一份、还没写第二份"的瞬间窗口——若恰好在这个窗口崩溃，两份日志就会撕裂。我们分两种顺序看后果。

背景：执行 UPDATE t SET c=2 WHERE id=1（原值 c=1）。崩溃恢复时，本机用 redo 恢复数据，从库用 binlog 同步——这两个动作的依据不同，撕裂就会让它们对不上。

顺序一：先写 redo，后写 binlog

redo 写完并落盘 → 崩溃 → binlog 没来得及写。

顺序二：先写 binlog，后写 redo

binlog 写完并落盘 → 崩溃 → redo 没来得及写。

结论：单纯定顺序救不了

两种顺序都会在崩溃窗口里产生不一致，只是方向相反。问题的根本在于："写 redo"和"写 binlog"是两个独立动作，中间存在一个无法消除的时间窗口，崩溃可能落在其中。无论谁先谁后，都没法保证"要么都成、要么都不成"。
解决思路只能是引入一个能在崩溃后判断"这个事务到底算不算提交成功"的协调协议——让恢复时有据可依地把撕裂的事务统一裁决为"提交"或"回滚"。这就是两阶段提交。

06

两阶段提交的流程

两阶段提交（Two-Phase Commit, 2PC）把"提交"这一个动作，拆成 prepare（准备） 和 commit（提交） 两个阶段，中间夹着 binlog 的写入。核心思想：用 binlog 是否写成功，作为整个事务"最终是否提交"的唯一裁判。

三步走

  1. 1Prepare 阶段（写 redo，标记 prepare）：InnoDB 把事务的 redo log 写入，并将其状态标记为 PREPARE，同时记下事务的 XID，然后把这部分 redo 刷盘。此刻 redo 已持久，但事务尚未"正式提交"。
  2. 2写 binlog 并刷盘：把事务的 binlog 事件（同样带上那个 XID）写入 binlog 文件并 fsync。这一步成功，是整个提交的"分水岭"——一旦 binlog 落盘，事务就被认定为"该提交"。
  3. 3Commit 阶段（redo 写 commit 标记）：InnoDB 在 redo log 里把该事务标记为 COMMIT。这一步即使不立刻 fsync 也不影响正确性——因为 binlog 已落盘，崩溃恢复时能据此把事务补提交。
为什么这样拆就对了

关键在于把"事务是否提交"的判定权，唯一地交给 binlog。redo 的 prepare 状态相当于"我准备好了、随时能提交，但还没拍板"。真正拍板的信号是 binlog 落盘。这样无论崩溃发生在哪一步，恢复时都只需问一句："这个 prepared 的事务，它的 XID 在 binlog 里吗？"——在，就提交；不在，就回滚。一个简单的问题，消除了第 5 章那两种撕裂。下一章先讲清这根把两者绑起来的"线"——XID。

07

XID：连接两份日志的纽带

2PC 能成立，靠的是一个把 redo 和 binlog "绑在一起"的共同标识——XID（事务标识）。它同时被写进 redo 的 prepare 记录和 binlog 的事件里，使恢复时能把"同一个事务在两份日志里的痕迹"对应起来。

XID 出现在两个地方

  • redo log 的 prepare 记录里：标记"这个 prepared 事务的身份是 XID"。
  • binlog 里的 XID_EVENT：每个事务在 binlog 中以一个特殊的 XID 事件收尾，记录同一个 XID，表示"这个事务在 binlog 中已完整落地"。
有了 XID 才能裁决

恢复时，InnoDB 从 redo 里找出所有处于 PREPARE 状态的事务，得到它们的 XID 列表；再去 binlog 里看这些 XID 在不在。"在不在 binlog"这个判断，全靠 XID 作为对应键。没有这根线，redo 里一个孤零零的 prepare 事务，根本没法知道它在 binlog 那边是否也写成了。下一章就是这套裁决逻辑的完整展开。

08

崩溃恢复：用 2PC 状态做决断

这是 2PC 的"兑现"环节，也是它全部意义所在。重启后，InnoDB 扫描 redo log，对每个事务根据它在两份日志里的状态统一裁决。一共三种情况：

逐种情况对照第 5 章的两个撕裂

崩溃时机redo 状态binlog 有无 XID裁决结果
prepare 之前无 / 不完整无回滚事务等于没发生，一致
prepare 后、写 binlog 前PREPARE无回滚对应第 5 章顺序一：本机也不提交 → 和从库都没有，一致
写 binlog 后、commit 前PREPARE有提交对应第 5 章顺序二：本机补提交 → 和从库都有，一致
commit 已写COMMIT有已提交正常完成
撕裂被消除的根本原因

对比第 5 章：那时"先 redo 后 binlog 崩溃"会让本机有、从库无。现在同样的崩溃时机（prepare 后、binlog 前），裁决是"回滚"——本机主动把这个 prepared 事务撤掉，于是本机和从库都没有它，一致。反过来，"binlog 已落盘后崩溃"则裁决"提交"，两边都有它，也一致。
binlog 成了那把唯一的尺子：以 binlog 是否记录为准，本机向 binlog 看齐。这正是把"判定权唯一交给 binlog"的威力——它让两份日志在任何崩溃点都能收敛到同一个结论。

09

一个完整的提交时序

把前面所有环节连成一条完整时间线，看一次 COMMIT 从头到尾发生了什么（默认双 1 配置）：

两个值得注意的细节

① 三次"刷盘"里，前两次是关键。prepare 的 redo 刷盘 + binlog 刷盘，是双 1 配置下每次提交的两次 fsync，也是性能开销的大头——这正是下一章组提交要优化的对象。第三步 commit 标记可以不立即 fsync，因为它不影响崩溃裁决。
② 提交返回成功 ≠ 数据页已落盘。返回成功只代表 redo + binlog 已持久（崩溃能恢复），数据页本身可能还是脏页，留待后台慢慢刷——这就是第三篇"WAL：日志先行，数据页后行"的体现。崩了也不怕，redo 能重做。

10

组提交 Group Commit

两阶段提交带来了正确性，但也带来了开销：双 1 配置下，每个事务提交都要做两次 fsync（prepare 的 redo 一次、binlog 一次）。fsync 是昂贵的磁盘同步操作，高并发下成千上万个事务各自 fsync，磁盘根本扛不住。组提交（Group Commit）就是把多个事务的 fsync"凑成一次"，找回性能。

核心思想：攒一批，刷一次

既然 fsync 一次可以把"当前文件里所有待落盘的内容"一起刷下去，那就不急着提交，稍微等一等，把同一时刻涌进来的多个事务凑成一组，一次 fsync 全部落盘。这样 N 个事务只付 1 次 fsync 的代价。

binlog 组提交的三个阶段

MySQL 把提交流程组织成三个串行的阶段队列，每个阶段同一时刻只有一个"组长（leader）"在干活，它替整组完成操作：

两个调优旋钮

组提交的"组"越大，fsync 摊销越划算。MySQL 提供两个参数主动制造更大的组（用一点延迟换更高吞吐）：
① binlog_group_commit_sync_delay —— Sync 阶段故意等待 N 微秒，让更多事务挤进这一组再一起刷。
② binlog_group_commit_sync_no_delay_count —— 但如果在等待期间已攒够这么多事务，就提前触发 fsync，不再傻等。
redo log 侧也有类似的合并刷盘机制。组提交是 MySQL 5.6 起的重要演进，正是它让"双 1 的强一致"在高并发下依然有可接受的性能——正确性靠 2PC，性能靠组提交，两者配套出现。

11

关键参数：双 1 配置

把第 2、3 章的两个刷盘参数放一起，就得到决定"崩溃丢不丢数据"的组合。所谓"双 1"，即 innodb_flush_log_at_trx_commit = 1 且 sync_binlog = 1，是最安全的生产标配。

flush_log / sync_binlog含义崩溃数据安全性性能
1 / 1（双 1）redo 与 binlog 每次提交都 fsync最强，提交即不丢、主从一致最低（靠组提交挽回）
2 / 1redo 写 OS 缓存(每秒 fsync)，binlog 每次 fsyncmysqld 崩溃不丢；主机崩溃可能丢约 1 秒 redo较高
1 / Nredo 每次 fsync，binlog 每 N 次 fsync崩溃可能丢最近 N 个事务的 binlog（影响复制）较高
0 / 0两者都靠后台/OS 定期刷最弱，崩溃可能丢约 1 秒数据最高
怎么选

金融、订单等不容丢数据 → 必须双 1。配合组提交，双 1 的性能损失在多数场景可以接受。
日志、埋点等可容忍少量丢失、追求吞吐 → 可放宽到 2/N 甚至 0/0。
要特别注意：sync_binlog 放宽不仅影响本机，还直接影响主从——丢了的 binlog 永远到不了从库，会造成主从数据差异，比丢本机数据更难补救。所以即便放宽 redo（设 2），通常也建议 sync_binlog=1 守住复制底线。这就是"2/1"组合常见的原因。

12

与 XA 分布式事务的关系

"两阶段提交"这个词有两层含义，容易混。本篇前面讲的是 MySQL 内部的 2PC；它和你可能听过的 XA 分布式事务，原理同源、场景不同。

内部 XA（本篇）外部 XA（分布式）
协调谁同一个 MySQL 内的 redo log 与 binlog跨多个数据库 / 多个 MySQL 实例
协调者MySQL Server（binlog 作裁判）外部事务管理器（如应用 / 中间件）
参与者InnoDB 引擎 + binlog各个资源（多个 RM）
触发方式自动，每次提交内部进行显式 XA START / PREPARE / COMMIT 语句
目的让两份日志原子一致让多个独立系统的事务原子一致
共同的内核

两者都遵循 2PC 的同一套骨架：先让所有参与者 prepare（进入"可提交但未提交"的状态并持久化），再由协调者根据"是否全部 prepare 成功"统一下达 commit 或 rollback。区别只在于"参与者是谁、协调者是谁"。
本篇的内部 XA 里，InnoDB 是参与者，它 prepare（redo 写 PREPARE）；binlog 的落盘是协调者的"决定"，决定一下，InnoDB 就 commit。理解了内部 2PC，再看分布式 XA 就只是把"两份日志"换成"多个系统"而已。
需要提醒：外部 XA 在 MySQL 中存在已知的复杂性与坑（如 prepare 后连接断开的悬挂事务、与 binlog 配合的历史问题），生产中使用需谨慎评估。

13

常见疑问与边界

Q1 · 只有 redo 而没开 binlog，还有 2PC 吗？

如果没开 binlog（或用不写 binlog 的引擎），InnoDB 内部就不需要和 binlog 做 2PC——一次普通的 redo 提交即可。2PC 是"因为有两份要协调的日志"才产生的。但生产环境几乎都开 binlog（复制、恢复刚需），所以 2PC 是常态。

Q2 · 为什么不干脆只用一份日志？

有人设想"让 binlog 兼任崩溃恢复，去掉 redo"或反之。但前面讲过：redo 是物理日志、循环写、为本机快速恢复优化；binlog 是逻辑日志、追加写、为复制与追溯优化。两者的写入方式、内容粒度、用途都不同，强行合并会顾此失彼。InnoDB 选择保留两份、用 2PC 协调，是分层架构（引擎层 vs Server 层）下的必然。

Q3 · prepare 阶段的 redo 一定要 fsync 吗？

是。prepare 的语义是"我已准备好，崩溃后能凭这份 redo 把事务重建到可提交状态"。若 prepare 的 redo 没落盘，崩溃后这个 prepared 事务就丢了，可它的 binlog 可能已经写了 → 又回到撕裂。所以 prepare 阶段必须确保 redo 持久（双 1 下即如此）。

Q4 · commit 阶段的 redo 为什么可以不 fsync？

因为裁决权在 binlog。只要 binlog 落盘了，崩溃恢复时即使看到 redo 还停在 prepare（commit 标记没刷下去），也会因"binlog 有该 XID"而补做提交（第 8 章 B 情况）。所以 commit 标记丢不丢都不影响最终结果——这正是把判定权交给 binlog 换来的简化。

一句话收束

2PC 的全部精妙，就在于"prepare 让 redo 先持久但不拍板，binlog 落盘才拍板，崩溃后以 binlog 为准统一裁决"。三句话背后，是对"两份独立日志在任意崩溃点都要一致"这一难题的完整解法。

14

系列总结：连接第三、四篇

本篇是为第三篇（故障恢复）和第四篇（事务与 MVCC）补上的桥。回看这三篇如何咬合：

篇讲了什么留下的接口
③ 故障恢复redo/undo/双写 怎么让崩溃后数据不丢、可回滚redo 能恢复——但恢复到哪个"提交"为准？
④ 事务与 MVCC事务的提交/回滚语义、版本可见性"提交"在引擎里发生——但它怎么和 binlog 对齐？
⑦ 两阶段提交用 prepare-commit + XID + binlog 裁决，把两份日志的"提交"绑成一个原子动作补全了"提交"的完整定义
三篇合一的"提交"全貌

现在能完整说清一次提交了：事务修改产生 redo/undo（③④）；提交时 InnoDB 先 prepare——redo 标 PREPARE 并刷盘（③ 的 redo + 本篇的状态机）；再写 binlog 并刷盘，这一刻确定事务"成立"；最后 redo 写 commit 标记，事务对 MVCC 可见（④ 的可见性）。崩溃后，恢复程序用 redo 重做、用 binlog 裁决 prepared 事务（③ 的恢复 + 本篇的裁决），并撤销未提交者（③ 的 undo）。三篇的机制在这里汇成一条逻辑闭环——这就是 InnoDB"提交"二字的完整含义。

③ 故障恢复＋④ 事务与 MVCC＝⑦ 两阶段提交（桥）

redo 保证"崩溃不丢"，binlog 保证"可复制可追溯"，两阶段提交保证"两者在任何崩溃点都一致"。正确性靠 2PC，性能靠组提交——这一对设计，撑起了 MySQL 既强一致又高并发的提交语义。
